# md-reader 完整方案

## 方案概述

### 1. 总体目标和范围

`md-reader` 的目标是构建一个面向常规项目的 Markdown 阅读器与编辑器，在尽量保持原生 `.md` 内容不变的前提下，提供接近 Notion 的视觉质感和编辑体验。

本方案聚焦以下范围：

- 以原生 Markdown 作为正文真相，保证文本可读、可版本管理、可迁移。
- 通过渲染层和编辑器交互层提升阅读与编辑体验，而不是将正文切换为专有 block JSON。
- 对超出原生 Markdown 表达能力、但确有价值的展示状态，使用独立 `sidecar metadata` 承载。
- 第一阶段聚焦文档类场景，不覆盖 Notion 数据库、多列自由布局、复杂页面组件等非 Markdown 核心能力。

不在本轮范围内的能力：

- 持久化多彩字体、任意字号、任意字体族。
- 任意块背景色、边框色、复杂卡片布局。
- Notion 数据库、看板、日历、多列自由排版。
- 为兼容富样式而向 Markdown 正文引入大量专有语法。

### 2. 各阶段任务概要

第一阶段：阅读器 MVP

- 主要工作：搭建 Markdown 渲染链、统一页面样式、完成 Notion 风格排版和基础增强。
- 预期成果：同一份原生 Markdown 在阅读态具备更现代的视觉质量。
- 执行顺序：先完成渲染与样式基线，再补充表格、引用块、代码块等重点节点的增强。

第二阶段：编辑器 MVP

- 主要工作：引入 Markdown-first 编辑器核心，补齐基础编辑、快捷插入、outline、folding、block hover 等交互。
- 预期成果：在不污染 Markdown 正文的前提下，提供更接近 Notion 的编辑体验。
- 执行顺序：先打通标准 Markdown 的稳定输入输出，再增加交互壳层能力。

第三阶段：sidecar metadata

- 主要工作：引入独立元数据文件，支持表格列宽、callout 类型、页面宽度等增强状态。
- 预期成果：在保持 Markdown 正文纯净的同时，实现有限但高价值的持久化展示增强。
- 执行顺序：先支持表格与页面级 metadata，再扩到块级 metadata。

第四阶段：一致性与工程化

- 主要工作：统一阅读态与编辑态表现，补充测试、示例文档、导入导出校验和回归验证。
- 预期成果：形成可在常规项目中复用的稳定 Markdown 阅读/编辑基础设施。
- 执行顺序：在 MVP 可用后集中清理一致性和质量保障问题。

### 3. 整体结构框架

系统分为四层：

- Markdown 正文层：承载文档语义，是唯一内容真相。
- Sidecar 元数据层：承载不适合写入 Markdown 正文的展示与交互状态。
- Renderer / Editor 体验层：负责 Notion-like 的视觉呈现和编辑交互。
- Browser workspace / project 层：负责项目接入、项目注册表、profile/local 持久化与三栏工作区壳层。

首发建议的实现模块：

- `src/app`：应用壳层、顶栏、三栏布局、模式切换。
- `src/workspace`：项目注册表、项目配置、profile/local、文件树。
- `src/renderer`：Markdown 渲染链、阅读态组件、样式主题。
- `src/editor`：编辑器封装、工具栏、slash menu、heading 提取、表格编辑辅助。
- `src/outline`：标题树生成、active 跟随、折叠状态与导航交互。
- `src/metadata`：sidecar schema、读写与绑定。
- `src/shared`：公共类型、存储键、路径与基础工具。

## 背景与核心约束

### 项目定位

本项目不是要复刻 Notion 的全部内容模型，而是要在常规项目中提供一个更好用的 Markdown 文档体验层。核心约束是：

- 文档内容尽可能保留原生 Markdown 形态。
- 保存时优先输出标准 Markdown，而不是编辑器内部专有数据结构。
- 只有在原生 Markdown 无法表达且需求价值足够高时，才引入 sidecar metadata。

### 关键矛盾

Markdown 与 Notion 的本质差异在于：

- Markdown 擅长文本语义和跨工具兼容。
- Notion 擅长结构化 block 交互和富样式页面体验。

因此，本方案不试图让 Markdown 承担不适合它的富样式状态，而是采取以下边界：

- 内容语义交给 Markdown。
- 展示和交互体验交给 renderer/editor。
- 少量高价值持久化展示状态交给 sidecar metadata。

## 技术选型

### 选型结论

推荐路线：

- 编辑器核心：`Milkdown`
- 阅读态渲染：`unified` 生态中的 `remark` / `rehype`
- 样式系统：自定义 Notion-like 主题层
- 元数据承载：与正文同名的 `*.md.meta.json`
- 首发运行形态：纯浏览器

### 为什么优先选择 Milkdown

选择 `Milkdown` 的原因：

- 它的定位本身就是 Markdown-first 的所见即所得编辑器，更符合“Markdown 正文是真相”的前提。
- 相比以富文本模型为中心的路线，更容易把保存结果稳定收敛在原生 Markdown。
- 它可以承载足够好的编辑体验，但不会天然把数据模型推向专有 block JSON。

### 为什么不选 BlockNote 作为主路线

`BlockNote` 更适合“像 Notion 的产品”，但不适合“常规项目中的 Markdown 主存储”：

- 它的核心优势在 block-based 交互，而不是 Markdown 保真。
- Markdown 导入导出更适合作为互操作能力，而不是唯一真相层。
- 如果未来围绕原生 `.md` 做工程化管理，路线会逐渐拧巴。

### 为什么不选 Tiptap 作为首选

`Tiptap` 是强备选，但不是当前约束下的首选：

- 它更容易演进到“富文本体验优先，Markdown 变导出格式”。
- 如果严格克制，只用可逆扩展，它也能完成任务，但团队执行上更容易边界漂移。
- 对当前项目而言，先从 Markdown-first 路线起步，更稳。

### 为什么仍然保留 Tiptap 作为二阶段备选

如果未来需求发生变化，例如：

- 明确接受更多专有扩展。
- 需要更复杂的 inline formatting。
- 需要更接近真正 Notion 的 block 能力。

则可以评估局部切换或在特定场景下引入 `Tiptap` 作为富交互编辑面。

## 总体架构

### 首发运行形态

本项目首发运行形态明确为：**纯浏览器 Web 应用**。

这意味着：

- 编辑器核心和渲染链都运行在浏览器环境内。
- `Project Registry`、项目切换、profile、local fallback 都需要围绕浏览器可访问的持久化能力设计。
- 第一版不假设 Electron、Tauri、本地 Node 服务或桌面原生宿主。

由此带来的约束：

- 文件访问与项目接入必须采用浏览器友好的方式设计。
- 本机 `Project Registry` 在实现层面应理解为“当前用户环境下的浏览器侧项目注册表”，而不是依赖桌面宿主私有 API 的前提能力。
- 项目根目录和内容根目录的接入需要基于浏览器授予的文件系统访问能力，例如 File System Access API。
- 后续如果增加桌面壳，应视为宿主扩展，而不是首发前提。

### 浏览器能力边界与降级策略

首发方案需要把浏览器能力边界写成显式产品约束，而不是实现细节：

- 支持 `File System Access API` 且可恢复目录句柄时，提供完整“项目接入 -> 文件树 -> 文档读写 -> 项目级配置落盘”能力。
- 不支持目录授权，或用户未授权目录时，只允许进入“浏览器内受限工作区”：
  - 可恢复 registry、profile、local 等浏览器侧状态。
  - 可提示重新授权项目目录。
  - 不伪装成已经具备真实项目文件直连能力。
- 若项目级配置无法写回 `<project>/.md-reader/project.json`，UI 需要明确提示当前配置只保存在浏览器侧，避免用户误判为团队共享配置已经落盘。

首发推荐支持矩阵：

- Chromium 系浏览器：正式支持。
- 不具备稳定目录授权能力的浏览器：只提供受限模式，不承诺完整项目接入体验。

### 架构原则

1. Markdown 正文是内容真相。
2. 编辑态和阅读态共享一套语义解释边界。
3. 视觉增强优先在渲染层完成。
4. 超出 Markdown 原生语义的持久化状态，进入 sidecar。
5. 第一版拒绝为了样式而污染正文。

### 分层架构

#### 1. Markdown 正文层

职责：

- 存放原始文档内容。
- 保持标准 Markdown 语义。
- 支持与 Git、其他编辑器、CI 文档流程直接协作。

承载内容：

- 标题、段落、列表、任务列表、引用、代码块、表格、图片、链接、分隔线。

不承载内容：

- 颜色、列宽、块主题、折叠 UI 状态、布局模式。

#### 2. Sidecar 元数据层

职责：

- 承载 Markdown 无法自然表达、但需要持久化的展示和交互状态。
- 与 Markdown 正文解耦，避免污染正文内容。

建议文件形式：

- `article.md`
- `article.md.meta.json`

适合放入 sidecar 的内容：

- 页面宽度模式。
- 表格列宽。
- callout 类型。
- 图片显示模式，例如 `full-width`。

#### 3. Renderer / Editor 体验层

职责：

- 将 Markdown 语义映射为更高质量的阅读与编辑体验。
- 提供接近 Notion 的页面质感与块级交互外壳。

阅读态能力：

- 页面宽度与留白控制。
- 标题、段落、列表、链接、图片、代码块统一主题化。
- 引用块 callout 化展示。
- 表格增强显示。

编辑态能力：

- slash menu
- block hover 操作
- 快捷键插入
- floating toolbar
- outline / heading folding
- 表格编辑辅助

## 数据模型设计

### 文档文件

建议采用以下存储约定：

```text
docs/
  guide/
    intro.md
    intro.md.meta.json
```

规则：

- 没有增强状态的文档可只存在 `.md`。
- 只有需要页面模式、列宽、块状态时，才生成 `.md.meta.json`。
- 不要求所有文档都强制携带 sidecar。

### Sidecar 结构

示例：

```json
{
  "version": 1,
  "page": {
    "width": "narrow"
  },
  "tables": {
    "table-1": {
      "columns": ["240px", "1fr", "160px"]
    }
  },
  "blocks": {
    "blockquote-3": {
      "callout": "warning"
    },
    "heading-7": {
      "collapsed": true
    },
    "image-2": {
      "display": "full-width"
    }
  }
}
```

字段说明：

- `version`：sidecar 版本号，便于后续演进。
- `page`：页面级样式或布局状态。
- `tables`：表格级元数据，主要用于列宽等增强配置。
- `blocks`：块级增强状态，用于 callout、折叠、图片显示模式等。

### 块标识策略

这是 sidecar 能否稳定工作的关键点。

推荐采用“逐级增强”的策略：

第一阶段：

- 基于 AST 遍历顺序生成运行时块路径。
- 先保证同一文件在轻量编辑后的稳定性，不追求极端复杂变更下的完美跟踪。

第二阶段：

- 对需要持久化绑定的对象引入更稳定的标识策略，例如内容哈希或编辑器辅助 ID。

原则：

- 不要为了追求绝对稳定块 ID，提前向 Markdown 正文注入专有标记。
- 在没有真实痛点前，避免把块标识系统做得过重。

## 能力边界

### 100% 不改 Markdown 正文即可实现的能力

#### 页面与排版

- Notion 风格版心宽度
- 页面留白和阅读节奏
- 标题层级视觉
- 段落、列表、引用、分隔线优化
- 代码块、链接、图片统一主题

#### 阅读器级语义增强

- 将普通 `blockquote` 渲染为 callout 风格外观
- 将 task list 渲染为更现代的 checklist
- 为表格增加 sticky header、hover、zebra stripe、横向滚动
- 为图片增加 caption 与居中显示策略

#### 编辑器交互增强

- slash menu
- 悬浮工具栏
- block hover
- outline
- heading folding
- 快捷键插入标准 Markdown 结构

### 建议使用 sidecar metadata 的能力

#### 表格列宽

原因：

- 标准 Markdown 表格没有列宽语义。
- 这类需求对阅读体验价值很高，但不值得污染正文。

#### Callout 类型

如果只是把所有引用块都渲染成统一 callout 风格，可不需要 sidecar。  
如果需要区分 `note`、`tip`、`warning`、`danger`，建议用 sidecar 表示。

#### 折叠状态

- 某些 section 默认展开或收起

这属于文档交互状态，而不是正文语义。

#### 页面显示模式

- `narrow`
- `wide`

这属于页面容器表现层，不应写入正文。

#### 图片展示模式

- `inline`
- `centered`
- `full-width`

### 第一版应放弃或明显降级的能力

- 持久化多彩字体
- 持久化任意字号与字体族
- 任意块背景色与边框色
- 多列自由布局
- Notion 数据库块
- 需要复杂专有语法支撑的组件化页面能力

原因：

- 它们超出 Markdown 原生语义。
- 一旦做深，系统会快速滑向富文本专有模型。
- 其收益低于构建成本和兼容性代价。

## 阅读器设计

### 渲染链

建议结构：

- `remark-parse`
- `remark-gfm`
- 自定义语义增强插件
- `remark-rehype`
- `rehype` 侧样式与节点转换
- React/Vue/Solid 组件映射层

核心目标：

- 让原生 Markdown 渲染结果稳定。
- 让增强效果集中在组件和样式层，而不是正文语法层。

### 页面视觉系统

建议统一一套设计 token：

- 页面宽度
- 字体层级
- 段落间距
- 标题锚点与 hover 行为
- 代码块背景和边框
- 表格边框、斑马纹、hover
- callout 背景、图标、边线

风格目标：

- 不是机械模仿 Notion 的像素细节。
- 而是复用它在信息层级、留白、交互反馈上的体验原则。

### Callout 的实现方式

第一版建议：

- 仍以原生 `blockquote` 作为正文输入。
- 渲染时统一套用更强的视觉样式。
- 如果存在 sidecar 中的 `callout` 类型，再根据类型切换颜色与图标。

这样可以得到：

- 没有 sidecar 时也能美化引用块。
- 有 sidecar 时可以稳定增强为 `note` / `warning` 等类型。

## 编辑器设计

### 基础目标

编辑器负责提升输入体验，但不改变内容真相。

需要保证：

- 标准 Markdown 输入输出稳定。
- 操作过程可以更像块编辑器，但保存结果仍是 Markdown。
- 所见即所得的增强尽量建立在现有 Markdown 语义上。

### 编辑器能力拆分

#### 核心编辑

- 标题、段落、列表、任务列表、引用、代码块、表格、图片
- 粘贴文本的 Markdown 规范化
- 常见快捷键

#### 壳层交互

- slash menu
- 插入菜单
- block hover 控件
- outline 导航
- heading folding
- 当前块聚焦

#### 增强编辑能力

- 表格编辑辅助
- 粘贴 CSV / Excel 自动转 Markdown 表格
- 图片插入与展示模式切换

### 编辑态与阅读态一致性

必须避免出现：

- 阅读态是 A 套语义解释，编辑态是 B 套。
- 编辑器生成的 Markdown 在阅读器里呈现明显退化。

约束方案：

- 阅读态与编辑态共享同一套 Markdown 语义边界。
- 共用样式 token 和块级组件规则。
- 所有增强能力都要先判断是“渲染态增强”还是“正文语义增强”。

## 标题侧边导航设计

### 目标

在编辑器侧边根据当前 Markdown 标题节点自动生成一个快捷导航面板，视觉和交互上接近常见文档大纲侧栏。

该能力的边界如下：

- 只基于 Markdown 标题节点 `h1-h6` 生成。
- 不把普通段落、列表项或视觉上像小节的文本混入导航。
- 支持点击跳转、当前标题高亮、滚动跟随、父子级折叠和平滑滚动定位。

### 数据来源策略

推荐采用混合路线：

- 编辑态：从 `Milkdown / ProseMirror` 当前文档树提取标题节点。
- 阅读态：从 Markdown AST 提取标题节点。
- 两侧共用同一套标题树数据结构和导航组件约定。

这样做的原因是：

- 编辑态需要更准确的实时同步和定位能力，直接读取编辑器文档树更稳。
- 阅读态不应依赖编辑器内部模型，仍应保持在 Markdown 渲染链内完成。
- 项目整体仍然保持 Markdown-first，而不是被编辑器内部状态绑死。

### 标题树数据结构

建议统一一个 `heading outline model`：

```ts
type HeadingOutlineItem = {
  id: string
  text: string
  level: number
  children: HeadingOutlineItem[]
  position: {
    start: number
    end?: number
  }
  collapsed?: boolean
  isActive?: boolean
}
```

字段职责：

- `id`：目录项与正文标题的绑定键，用于点击跳转和 active 关联。
- `text`：标题文本。
- `level`：标题层级，映射 `h1-h6`。
- `children`：子标题树。
- `position`：用于滚动定位和 active 判断。
- `collapsed`：目录树 UI 状态，不属于 Markdown 正文语义。
- `isActive`：运行时状态，只在界面层维护。

### 交互规则

#### 1. 点击跳转

- 点击目录项后，正文区域平滑滚动到对应标题。
- 若目标标题位于已折叠的父级目录之下，先自动展开父级，再执行定位。
- 跳转完成后，目标目录项进入 active 状态。

#### 2. 当前标题高亮

- active 项不以“最后点击项”为准，而以当前滚动位置实时计算。
- 推荐规则是：选取当前视窗内最靠上、且仍属于主阅读区域的标题作为 active。
- 这样滚动时目录高亮能真实反映当前阅读位置，不会与正文脱节。

#### 3. 滚动跟随

- 正文滚动时，目录同步更新 active 项。
- 当 active 项滚出目录可视区域时，目录容器只做轻量跟随滚动，保证当前项可见。
- 不建议每次都强制滚到目录中线，否则会造成明显抖动。

#### 4. 折叠规则

- 折叠只作用于目录树，不直接改变正文标题结构。
- 父级标题可以折叠其所有子级。
- 默认建议展开到 `h1-h3`，深层标题跟随父级状态展示。
- 用户手动折叠后的状态属于个人导航偏好，应优先持久化到 profile 或 local fallback，不写回 Markdown 正文，也不进入 sidecar。

#### 5. 程序滚动与手动滚动区分

- 点击目录会触发程序性滚动。
- 如果不区分程序滚动和手动滚动，active 状态会在动画过程中频繁跳动。
- 需要一个短暂的 `programmatic scrolling` 运行时状态，在平滑滚动期间抑制无意义的 active 抖动，滚动结束后再恢复正常计算。

### 与 sidecar 的关系

标题侧边导航本身不需要写回正文。

其中可进入 sidecar 的只有少量高价值状态：

- 页面是否展示导航侧栏
- 侧栏宽窄等页面级显示偏好

不建议进入 sidecar 的内容：

- 当前 active 项
- 临时滚动位置
- hover 状态
- 程序滚动中的中间态
- 标题导航折叠状态

### 模块落点

建议模块职责如下：

- `src/outline`
  - 定义 `HeadingOutlineItem`
  - 统一标题树构造结果和基础类型
  - 管理 active、折叠、点击跳转和滚动跟随
- `src/editor`
  - 从 `Milkdown / ProseMirror` 提取标题树
  - 渲染编辑态侧边导航组件
- `src/renderer`
  - 从 Markdown AST 构造标题树
  - 在阅读态复用相同导航组件和样式规则
- `src/app`
  - 管理导航侧栏显隐、布局挂载和模式切换
- `src/metadata`
  - 管理页面级显示偏好等允许进入 sidecar 的增强状态

### MVP 范围内的落地建议

标题导航建议直接进入 MVP 必做，而不是拖到后续阶段。

MVP 第一版应至少支持：

- 自动从 `h1-h6` 生成目录树
- 点击目录项平滑定位到标题
- 当前标题高亮
- 正文滚动时目录自动跟随
- 父子级折叠

MVP 第一版可以暂缓的部分：

- 复杂动画和高级过渡效果
- 多文档共享导航偏好
- 移动端的复杂抽屉式目录策略

### 验证重点

这一能力需要单独验证以下场景：

- 空文档、单标题文档、深层嵌套标题文档
- 标题频繁增删后的目录同步
- 点击目录跳转与手动滚动之间的 active 一致性
- 长文档下目录自动跟随是否平稳
- sidecar 缺失或损坏时，导航是否仍能按标题正常生成

## 工作区壳层与配置框架

### 目标

`md-reader` 不只是一篇文档的阅读器或编辑器，还需要具备类似 `data-editor` 的工作区壳层能力：

- 左侧文件树作为主导航入口
- 顶部项目切换器支持多项目切换
- 视觉风格和布局组织接近已有的 `data-editor`
- 配置采用“本机 registry + 项目级配置 + 个人 profile + local fallback”的分层模型

这部分能力属于工作区框架，而不是单篇 Markdown 文档能力。

### 参考来源与继承原则

`C:\Code\data-editor` 已经具备可复用的壳层组织方式：

- `Sidebar` 负责项目切换、文件树和导航入口
- 项目切换基于项目注册表
- 个人导航偏好和布局状态分层保存

`md-reader` 应继承它的结构思想，但不照搬其“文件 -> collection -> 主表 -> 详情”的数据编辑模型。

本项目只继承以下原则：

- 左侧工作区导航壳层
- 浏览器侧多项目 registry
- 项目级配置目录
- 个人 profile / 浏览器本地 fallback 分层
- 侧栏宽度、树展开状态、页面偏好等 UI 配置分离保存

### 四层配置模型

#### 1. 浏览器侧 Project Registry

作用：

- 保存当前机器上可切换的 Markdown 项目列表
- 提供项目切换器的数据源
- 保存当前 active project

建议位置：

- 浏览器侧持久化存储
- 优先使用 `IndexedDB`
- 可选提供导入/导出 JSON 的能力，便于迁移或备份

建议结构：

```json
{
  "version": 1,
  "activeProjectId": "notes",
  "projects": [
    {
      "id": "notes",
      "name": "Notes",
      "rootHandleKey": "project-handle-notes",
      "contentRoots": ["docs", "notes"]
    }
  ]
}
```

字段职责：

- `id`：稳定项目标识
- `name`：项目显示名称
- `rootHandleKey`：项目根目录句柄在浏览器侧持久化存储中的关联键
- `contentRoots`：该项目的 Markdown 内容根

边界：

- registry 只负责项目发现与切换
- 不存复杂 UI 状态
- 不承载单文档或个人阅读偏好

实现说明：

- 在纯浏览器首发形态下，registry 不应实现为真实文件路径协议的强依赖。
- 推荐将项目定义持久化到 `IndexedDB`，并把目录访问授权与项目定义关联保存。
- 如需可移植配置，可额外提供 JSON 导出，但它不是首发运行前提。

#### 2. 项目级配置

作用：

- 定义某个项目如何被 `md-reader` 理解和组织
- 保存团队共享的工作区结构规则

建议位置：

```text
<project>/.md-reader/project.json
```

建议结构：

```json
{
  "version": 1,
  "contentRoots": [
    {
      "id": "docs",
      "label": "Docs",
      "path": "docs"
    },
    {
      "id": "notes",
      "label": "Notes",
      "path": "notes"
    }
  ],
  "defaultDocument": "docs/README.md",
  "fileFilter": {
    "include": ["**/*.md", "**/*.mdx"],
    "exclude": ["**/node_modules/**", "**/.git/**"]
  },
  "workspace": {
    "showOutlineByDefault": true,
    "showFileTreeByDefault": true
  }
}
```

适合放入项目级配置的内容：

- 内容根目录
- 文件包含/排除规则
- 默认打开文档
- 项目级默认工作区显示策略

不适合放入项目级配置的内容：

- 个人侧栏宽度
- 个人主题和字号
- 个人最近打开项
- 当前滚动位置

纯浏览器接入说明：

- 项目级配置的概念可以继续保留为 `project.json` 逻辑模型。
- 但首发实现需要先明确：浏览器是否直接读写项目目录中的 `.md-reader/project.json`，取决于用户是否已授权对应目录。
- 如果授权不可用，系统应允许退回到浏览器侧持久化，并在 UI 中清晰提示“当前项目配置尚未落盘到项目目录”。

#### 3. 个人 Profile 配置

作用：

- 保存某个用户在某个项目内的长期工作区偏好
- 作为显式可切换的用户概念存在，而不是隐式内部状态

建议位置：

```text
<project>/.md-reader/profiles/<profile>.json
```

建议结构：

```json
{
  "version": 1,
  "appearance": {
    "theme": "light",
    "fontSize": 16,
    "pageWidth": "narrow"
  },
  "layout": {
    "sidebarWidth": 280,
    "outlineWidth": 320,
    "sidebarCollapsed": false,
    "outlineCollapsed": false
  },
  "navigation": {
    "expandedFileNodes": ["docs", "docs/guides"],
    "expandedHeadingNodes": {
      "docs/README.md": ["heading-1", "heading-3"]
    },
    "lastOpenedDocument": "docs/README.md"
  },
  "documents": {
    "docs/README.md": {
      "mode": "split"
    }
  }
}
```

适合放入 profile 的内容：

- 主题、字号、页面宽度
- 左右侧栏宽度与显隐状态
- 文件树展开状态
- 标题导航折叠状态
- 最近打开文档
- 文档级默认工作模式，例如 `read / edit / split`

profile 交互规则：

- profile 是显式用户可切换概念。
- 顶栏需要提供 profile 切换入口。
- 第一版至少支持 `default` profile，并允许后续扩展到多 profile。

原则：

- profile 保存长期偏好
- 不承担短时瞬态交互状态

#### 4. 浏览器 Local Fallback

作用：

- 在未选择 profile 或不值得长期持久化的场景下，保存临时工作现场

适合放入 local fallback 的内容：

- 当前会话打开的文档
- 当前滚动位置
- 当前 active heading
- 临时展开/折叠的界面状态

原则：

- local fallback 只保存短期会话上下文
- 不作为项目长期真相

### 配置层次边界

建议严格保持四层职责分离：

- registry：项目发现与切换
- project：项目共享结构规则
- profile：个人长期偏好
- local：本地短期上下文

此外还要与文档层区分：

- `*.md`：正文内容
- `*.md.meta.json`：单文档增强状态

不要把工作区壳层状态混进 `*.md.meta.json`，也不要把单文档增强状态混进 profile。

## 三栏工作区 UI 结构

### 总体布局

`md-reader` 的 Web 工作区建议采用三栏主结构：

- 左侧：`Project Sidebar`
- 中间：`Document Workspace`
- 右侧：`Outline Sidebar`

外层再加一个轻量 `Top Bar`，整体结构如下：

```text
App Shell
  Top Bar
  Main Workspace
    Left Sidebar
    Center Document Area
    Right Outline Sidebar
```

### Top Bar

职责：

- 项目切换入口
- profile 切换
- 阅读 / 编辑 / 分栏模式切换
- 主题和页面宽度切换
- 文档搜索入口
- 当前文档标题或路径显示

原则：

- 顶栏只承载工作区级动作
- 不承载正文局部编辑逻辑
- 不演化成重型 IDE toolbar

### Left Sidebar

左侧栏是主导航入口，建议拆为三个区域：

#### 1. Project Switcher

- 显示当前项目名
- 下拉切换已注册项目
- 提供项目设置入口

#### 2. Markdown File Tree

- 基于 `contentRoots` 构建文件树
- 支持 folder 展开/折叠
- 支持当前文档高亮
- 支持记住展开状态
- 支持按项目配置过滤文件

#### 3. Optional Quick Access

这一段不再沿用 `data-editor` 的 `collection` 概念，而更适合文档语义：

- 最近打开文档
- 收藏文档
- pinned docs

第一版可以不做，预留结构即可。

### Center Document Workspace

中间区域负责正文阅读与编辑。

建议支持三种模式：

- `read`
- `edit`
- `split`

内部可拆分为：

- 文档头区
- 主正文区
- 可选底部状态区

文档头区适合承载：

- 文档标题
- 文件路径
- 保存状态
- 文档级模式切换

### Right Outline Sidebar

右侧栏负责标题导航与文档辅助信息。

它与左侧栏的职责必须明确分开：

- 左侧栏解决“我当前看哪篇文档”
- 右侧栏解决“我在当前文档的哪一段”

当前阶段，右侧栏至少承载：

- `h1-h6` 标题导航
- 当前标题高亮
- 点击跳转
- 滚动跟随
- 目录折叠

### 与 data-editor 一致和不一致的部分

建议保持一致的部分：

- 项目切换器形式
- 左侧工作区壳层组织方式
- 侧栏宽度可调
- 展开状态持久化
- registry / project / profile / local 的配置分层

建议不要照搬的部分：

- `collection` 概念
- 数据主表 / 详情面板双结构
- 面向数据编辑的高密度工具栏布局

## 模块落点补充

在首发单包结构下，建议补充工作区职责：

- `src/shared`
  - registry 类型
  - project config 类型
  - profile 类型
  - outline model 类型
- `src/app`
  - App Shell
  - Top Bar
  - Left Sidebar / Right Sidebar 布局挂载
  - 项目切换、profile 切换、工作区显隐
- `src/editor`
  - Markdown 编辑器
  - 编辑态标题导航提取
- `src/renderer`
  - 阅读态渲染
  - 阅读态标题导航提取
- `src/workspace`
  - project registry 读取
  - project config / profile / local 协调
  - 文件树状态和工作区偏好状态管理

当后续实现规模明显增长时，再把 `src/workspace`、`src/renderer`、`src/editor`、`src/metadata` 等职责抽离为 `packages/`，而不是首发就按多包仓库组织。

## MVP 范围

### MVP 必做

- 纯浏览器工作区壳层
- 浏览器侧 Project Registry 与显式 profile 切换
- 左侧 Markdown 文件树
- 右侧标题导航侧栏
- 三栏工作区布局与侧栏宽度状态
- 原生 Markdown 文档读取与渲染
- Notion-like 页面排版主题
- 引用块、代码块、表格、图片增强显示
- 基础 Markdown 编辑
- slash menu / outline / heading folding
- sidecar 读写能力
- 表格列宽、页面宽度、callout 类型的最小增强实现

### MVP 可选

- 图片展示模式
- CSV/Excel 粘贴转表格
- 文档级偏好设置
- 多 profile 管理 UI 的增强能力

### MVP 不做

- 多彩字体持久化
- 任意块色彩系统
- 多列自由布局
- 数据库块
- 深度协同编辑

## 分阶段实施计划

### 阶段一：渲染基线

目标：

- 先把纯浏览器工作区壳层和阅读器基线做对、做稳、做美观。

任务：

- 明确纯浏览器运行边界与浏览器侧项目注册表方案。
- 搭建三栏工作区壳层。
- 打通项目切换、显式 profile 切换和左侧文件树基础流程。
- 打通 Markdown 渲染链。
- 建立主题 token 与基础样式。
- 完成标题、段落、列表、代码块、引用块、表格、图片的阅读态组件。
- 完成阅读态标题导航基础生成与高亮。

完成标准：

- 对标准 Markdown 文档能稳定渲染。
- 能在纯浏览器环境下切换项目、切换 profile、切换文档。
- 页面质感达到“明显优于默认 Markdown renderer”的水平。

### 阶段二：编辑器基线

目标：

- 保证 Markdown-first 编辑稳定可用。

任务：

- 接入 Milkdown。
- 打通基础 Markdown 输入输出。
- 加入基础工具栏、slash menu、outline、heading folding。
- 接入编辑态标题导航、点击跳转、滚动跟随与 active 高亮。
- 确保编辑态和阅读态的节点风格一致。

完成标准：

- 从编辑到保存再到阅读显示闭环稳定。
- 常用文档编写操作流畅。
- 左右侧栏与正文模式切换协同稳定。

### 阶段三：sidecar 增强

目标：

- 在不污染正文的前提下引入少量高价值持久化能力。

任务：

- 定义 sidecar schema。
- 实现 sidecar 读写与容错。
- 支持页面宽度、表格列宽、callout 类型。
- 将标题导航折叠状态和工作区偏好明确保留在 profile / local，而不是 sidecar。

完成标准：

- 无 sidecar 文档可正常工作。
- 有 sidecar 文档可稳定恢复增强状态。

### 阶段四：质量与验证

目标：

- 建立长期可维护的工程质量基线。

任务：

- 建立 Markdown fixture 文档集。
- 增加 renderer/editor round-trip 测试。
- 补充 sidecar schema 校验与兼容策略。
- 验证空 sidecar、损坏 sidecar、缺失块 ID 等降级路径。

完成标准：

- 主要能力具备自动化回归保障。
- sidecar 缺失或异常时，文档仍可降级读取。

## 验证策略

### 1. 渲染验证

- 使用固定 Markdown fixture 对比渲染快照。
- 重点覆盖标题、列表、引用、代码块、表格、图片、长文档。

### 2. 编辑验证

- 验证 Markdown 输入输出稳定性。
- 覆盖插入、删除、粘贴、撤销、保存、重新加载。

### 3. Round-trip 验证

验证链路：

- `Markdown -> 编辑器加载 -> 用户编辑 -> 保存 Markdown -> 阅读器渲染`

关注点：

- 是否引入非预期语法变化。
- 是否打乱列表、表格、引用结构。

### 4. Sidecar 验证

- sidecar 存在时增强能恢复。
- sidecar 缺失时文档仍能正常渲染。
- sidecar 损坏时有安全降级策略。

## 风险与控制

### 风险 1：需求滑向富文本系统

表现：

- 新需求不断要求颜色、布局、块样式持久化。

控制：

- 明确坚持“先问是否属于 Markdown 语义，再决定是否进入 sidecar”。
- 第一版严格限制 sidecar 类型。

### 风险 2：块标识不稳定导致 metadata 漂移

表现：

- 编辑后表格列宽或 callout 类型错绑到其他块。

控制：

- MVP 只为少量高价值块引入 metadata。
- 块标识策略分阶段增强，不在第一版过度设计。

### 风险 3：阅读态与编辑态风格断裂

表现：

- 同一文档在两个模式下视觉和结构感差异过大。

控制：

- 统一样式 token 和组件映射规则。
- 优先让阅读器和编辑器共用节点渲染约定。

### 风险 4：为模仿 Notion 牺牲 Markdown 兼容性

表现：

- 为了复刻视觉细节，引入大量专有语法或隐藏数据。

控制：

- 以“常规项目中的 Markdown 文档系统”为主目标，而不是 Notion 克隆。

## 关键决策记录

1. 内容真相选择 Markdown，而不是专有 block JSON。
2. 用 `Milkdown` 作为编辑器核心，避免系统天然偏向富文本模型。
3. 用 `remark` / `rehype` 构建阅读态渲染链。
4. 用 `*.md.meta.json` 承载少量高价值展示状态。
5. 第一版放弃持久化多彩字体等高破坏性能力。

## 后续落地建议

建议下一步继续产出两份配套文档：

1. `MVP 实施计划`

- 明确第一轮开发拆分、文件清单、技术栈、里程碑、验证命令。

2. `sidecar schema 详细设计`

- 明确字段定义、块标识策略、兼容与迁移规则。

如果要马上进入实现阶段，推荐优先顺序：

1. 先做阅读器渲染链和主题。
2. 再接入编辑器并保证 Markdown round-trip 稳定。
3. 最后再做 sidecar metadata。
