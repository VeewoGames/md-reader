# md-reader 完整方案

## 方案概述

### 1. 总体目标和范围

`md-reader` 的目标是构建一个面向常规项目的 Markdown 阅读器与编辑器，在尽量保持原生 `.md` 内容不变的前提下，提供接近 Notion 的视觉质感和编辑体验。

本方案聚焦以下范围：

- 以原生 Markdown 作为正文真相，保证文本可读、可版本管理、可迁移。
- 通过渲染层和编辑器交互层提升阅读与编辑体验，而不是将正文切换为专有 block JSON。
- 对超出原生 Markdown 表达能力、但确有价值的展示状态，使用独立 `sidecar metadata` 承载。
- 首阶段聚焦大文档库下的读取、导航、搜索和状态恢复，不覆盖 Markdown 写回与 sidecar 写回。

不在本轮范围内的能力：

- 首阶段 Markdown 写回。
- 首阶段 sidecar 写回。
- 持久化多彩字体、任意字号、任意字体族。
- 任意块背景色、边框色、复杂卡片布局。
- Notion 数据库、看板、日历、多列自由排版。
- 为兼容富样式而向 Markdown 正文引入大量专有语法。

### 2. 各阶段任务概要

第一阶段：只读工作区 MVP

- 主要工作：搭建本地服务优先工作区、Markdown 渲染链、标题导航、搜索与状态恢复。
- 预期成果：同一份原生 Markdown 在阅读态具备更现代的视觉质量，并能在大项目中稳定打开和导航。
- 执行顺序：先完成运行器和服务基座，再补渲染、导航和搜索。

第二阶段：编辑器 MVP

- 主要工作：引入 Markdown-first 编辑器核心，补齐基础编辑、快捷插入、outline、folding、block hover 等交互。
- 预期成果：在不污染 Markdown 正文的前提下，提供更接近 Notion 的编辑体验。
- 执行顺序：先打通标准 Markdown 的稳定输入输出，再增加交互壳层能力。

第三阶段：sidecar metadata

- 主要工作：引入独立元数据文件，支持表格列宽、callout 类型、页面宽度等增强状态。
- 预期成果：在保持 Markdown 正文纯净的同时，实现有限但高价值的持久化展示增强。
- 执行顺序：先支持表格与页面级 metadata，再扩到块级 metadata。

第四阶段：一致性与工程化

- 主要工作：统一阅读态与编辑态表现，补充测试、示例文档、导入导出校验和回归验证。
- 预期成果：形成可在常规项目中复用的稳定 Markdown 阅读/编辑基础设施。
- 执行顺序：在 MVP 可用后集中清理一致性和质量保障问题。

### 3. 整体结构框架

系统分为四层：

- Markdown 正文层：承载文档语义，是唯一内容真相。
- Sidecar 元数据层：承载不适合写入 Markdown 正文的展示与交互状态。
- Renderer / Editor 体验层：负责 Notion-like 的视觉呈现和编辑交互。
- Local-service workspace / project 层：负责项目接入、项目注册表、profile/local 持久化与三栏工作区壳层。

首发建议的实现模块：

- `src/app`：应用壳层、顶栏、三栏布局、模式切换。
- `src/workspace`：项目注册表、项目配置、profile/local、文件树。
- `src/renderer`：Markdown 渲染链、阅读态组件、样式主题。
- `src/editor`：编辑器封装、工具栏、slash menu、heading 提取、表格编辑辅助。
- `src/outline`：标题树生成、active 跟随、折叠状态与导航交互。
- `src/metadata`：sidecar schema、读写与绑定。
- `src/shared`：公共类型、存储键、路径与基础工具。

## 背景与核心约束

### 项目定位

本项目不是要复刻 Notion 的全部内容模型，而是要在常规项目中提供一个更好用的 Markdown 文档体验层。核心约束是：

- 文档内容尽可能保留原生 Markdown 形态。
- 保存时优先输出标准 Markdown，而不是编辑器内部专有数据结构。
- 只有在原生 Markdown 无法表达且需求价值足够高时，才引入 sidecar metadata。

### 关键矛盾

Markdown 与 Notion 的本质差异在于：

- Markdown 擅长文本语义和跨工具兼容。
- Notion 擅长结构化 block 交互和富样式页面体验。

因此，本方案不试图让 Markdown 承担不适合它的富样式状态，而是采取以下边界：

- 内容语义交给 Markdown。
- 展示和交互体验交给 renderer/editor。
- 少量高价值持久化展示状态交给 sidecar metadata。

## 技术选型

### 选型结论

推荐路线：

- 编辑器核心：`Milkdown`
- 阅读态渲染：`unified` 生态中的 `remark` / `rehype`
- 样式系统：自定义 Notion-like 主题层
- 元数据承载：与正文同名的 `*.md.meta.json`
- 首发运行形态：本地服务优先 + Web 前端
- 首发最低 Node 版本：`22.13+`

### 为什么优先选择 Milkdown

选择 `Milkdown` 的原因：

- 它的定位本身就是 Markdown-first 的所见即所得编辑器，更符合“Markdown 正文是真相”的前提。
- 相比以富文本模型为中心的路线，更容易把保存结果稳定收敛在原生 Markdown。
- 它可以承载足够好的编辑体验，但不会天然把数据模型推向专有 block JSON。

### 为什么不选 BlockNote 作为主路线

`BlockNote` 更适合“像 Notion 的产品”，但不适合“常规项目中的 Markdown 主存储”：

- 它的核心优势在 block-based 交互，而不是 Markdown 保真。
- Markdown 导入导出更适合作为互操作能力，而不是唯一真相层。
- 如果未来围绕原生 `.md` 做工程化管理，路线会逐渐拧巴。

### 为什么不选 Tiptap 作为首选

`Tiptap` 是强备选，但不是当前约束下的首选：

- 它更容易演进到“富文本体验优先，Markdown 变导出格式”。
- 如果严格克制，只用可逆扩展，它也能完成任务，但团队执行上更容易边界漂移。
- 对当前项目而言，先从 Markdown-first 路线起步，更稳。

## 总体架构

### 首发运行形态

本项目修订后的首发运行形态明确为：**本地服务优先 + Web 前端**。

这意味着：

- 编辑器核心和渲染链仍运行在浏览器环境内。
- 项目扫描、文件树、文档读取、搜索与状态恢复通过本机 Node 服务提供。
- 第一版不假设 Electron、Tauri 等桌面壳，但明确依赖本地 Node 服务。

由此带来的约束：

- 文件访问与项目接入统一通过本机服务设计。
- 本机 `Project Registry` 应理解为当前用户机器上的本地配置，而不是浏览器私有存储。
- 后续如果增加桌面壳，应视为宿主扩展，而不是首发前提。

### 运行环境边界与降级策略

首发方案需要把运行环境边界写成显式产品约束，而不是实现细节：

- 本地服务健康时，提供完整“项目接入 -> 文件树 -> 文档读取 -> 搜索 -> 状态恢复”能力。
- 本地服务不可达时，只允许进入“前端受限工作区”：
  - 可显示已有 UI 壳层。
  - 可提示启动或恢复本地服务。
  - 不伪装成已经具备真实项目文件直连能力。
- 首阶段不写回 Markdown 或 sidecar，因此不存在“浏览器授权后直接写项目目录”的前提。

首发推荐支持矩阵：

- Chromium 系浏览器 + 本机 Node `22.13+`：正式支持。
- Codex 内嵌浏览器 + 本机 Node `22.13+`：正式支持。
- 无法连到本机服务的环境：只提供受限模式，不承诺完整项目接入体验。

### 架构原则

1. Markdown 正文是内容真相。
2. 编辑态和阅读态共享一套语义解释边界。
3. 视觉增强优先在渲染层完成。
4. 超出 Markdown 原生语义的持久化状态，进入 sidecar。
5. 首阶段拒绝为了样式而污染正文。

## 标题侧边导航设计

### 目标

在工作区右侧根据当前 Markdown 标题节点自动生成一个快捷导航面板，视觉和交互上接近常见文档大纲侧栏。

该能力的边界如下：

- 只基于 Markdown 标题节点 `h1-h6` 生成。
- 不把普通段落、列表项或视觉上像小节的文本混入导航。
- 支持点击跳转、当前标题高亮、滚动跟随、父子级折叠和平滑滚动定位。

### 数据来源策略

首阶段采用只读路线：

- 阅读态：从 Markdown AST 提取标题节点。
- 搜索结果跳转后：仍复用同一套标题树与滚动定位逻辑。
- 编辑态：放到后续阶段再从 `Milkdown / ProseMirror` 文档树提取。

### 标题树数据结构

建议统一一个 `heading outline model`：

```ts
type HeadingOutlineItem = {
  id: string
  text: string
  level: number
  children: HeadingOutlineItem[]
  position: {
    start: number
    end?: number
  }
  collapsed?: boolean
  isActive?: boolean
}
```

### 交互规则

- 点击目录项后，正文区域平滑滚动到对应标题。
- active 项不以“最后点击项”为准，而以当前滚动位置实时计算。
- 正文滚动时，目录同步更新 active 项。
- 折叠只作用于目录树，不直接改变正文标题结构。
- 标题导航折叠状态属于个人导航偏好，应优先持久化到 profile 或 local fallback，不写回 Markdown 正文，也不进入 sidecar。

## 工作区壳层与配置框架

### 目标

`md-reader` 不只是一篇文档的阅读器，还需要具备类似 `data-editor` 的工作区壳层能力：

- 左侧文件树作为主导航入口
- 顶部项目切换器支持多项目切换
- 视觉风格和布局组织接近已有的 `data-editor`
- 配置采用“本机 registry + 项目级配置 + 个人 profile + local fallback”的分层模型

### 参考来源与继承原则

`C:\Code\data-editor` 已经具备可复用的壳层组织方式：

- `Sidebar` 负责项目切换、文件树和导航入口
- 项目切换基于项目注册表
- 个人导航偏好和布局状态分层保存
- 运行器约定以 `open/stop/health` 为核心

`md-reader` 应继承它的结构思想，但不照搬其“文件 -> collection -> 主表 -> 详情”的数据编辑模型。

本项目只继承以下原则：

- 左侧工作区导航壳层
- 本机多项目 registry
- 项目级配置目录
- 个人 profile / 本机 local fallback 分层
- 侧栏宽度、树展开状态、页面偏好等 UI 配置分离保存
- `open/stop/health` 运行器约定

## 四层配置模型

### 1. 本机 Project Registry

作用：

- 保存当前机器上可切换的 Markdown 项目列表
- 提供项目切换器的数据源
- 保存当前 active project

建议位置：

- 本机配置目录
- 可选提供导入/导出 JSON 的能力，便于迁移或备份

建议结构：

```json
{
  "version": 1,
  "activeProjectId": "notes-a13f20d1",
  "projects": [
    {
      "id": "notes-a13f20d1",
      "name": "Notes",
      "root": "C:\\Code\\Notes",
      "contentRoots": ["docs", "notes"]
    }
  ]
}
```

### 2. 项目级配置

作用：

- 定义某个项目如何被 `md-reader` 理解和组织
- 保存团队共享的工作区结构规则

建议位置：

```text
<project>/.md-reader/project.json
```

说明：

- 项目级配置的概念继续保留为 `project.json` 逻辑模型。
- 但首阶段先不要求写回项目目录中的 `.md-reader/project.json`。
- 若项目目录中已有该文件，可由本地服务读取并参与内容根和过滤规则解析。

### 3. 个人 Profile 配置

作用：

- 保存某个用户在某个项目内的长期工作区偏好
- 作为显式可切换的用户概念存在，而不是隐式内部状态

建议位置：

```text
%APPDATA%/md-reader/profiles/<profile>.json
```

### 4. 本机 Local Fallback

作用：

- 保存临时工作现场

适合放入 local fallback 的内容：

- 当前会话打开的文档
- 当前滚动位置
- 当前 active heading
- 临时展开/折叠的界面状态
- 阅读进度

## 三栏工作区 UI 结构

### 总体布局

`md-reader` 的 Web 工作区建议采用三栏主结构：

- 左侧：`Project Sidebar`
- 中间：`Document Workspace`
- 右侧：`Outline Sidebar`

### Top Bar

职责：

- 项目切换入口
- profile 切换
- 搜索入口
- 当前文档标题或路径显示
- 服务健康状态

### Left Sidebar

左侧栏是主导航入口，建议拆为三个区域：

- `Project Switcher`
- `Markdown File Tree`
- `Quick Access`

### Center Document Workspace

中间区域负责正文阅读。

首阶段只要求：

- 文档头区
- 主正文区
- 可选底部状态区

### Right Outline Sidebar

右侧栏负责标题导航与文档辅助信息。

当前阶段，右侧栏至少承载：

- `h1-h6` 标题导航
- 当前标题高亮
- 点击跳转
- 滚动跟随
- 目录折叠

- 左侧 Markdown 文件树

## 分阶段实施计划

### 阶段一：只读工作区基线

目标：

- 先把本地服务优先工作区壳层、阅读器基线、标题导航、搜索和状态恢复做对、做稳、做美观。

任务：

- 明确本地服务运行边界与本机项目注册表方案。
- 搭建三栏工作区壳层。
- 打通项目切换、显式 profile 切换和左侧文件树基础流程。
- 打通 Markdown 渲染链。
- 建立主题 token 与基础样式。
- 完成标题、段落、列表、代码块、引用块、表格、图片的阅读态组件。
- 完成阅读态标题导航基础生成与高亮。
- 完成基础搜索入口、结果展示与状态恢复。

完成标准：

- 对标准 Markdown 文档能稳定渲染。
- 能在本地服务环境下切换项目、切换 profile、切换文档。
- 能恢复当前文档、阅读进度和标题导航上下文。
- 页面质感达到“明显优于默认 Markdown renderer”的水平。

### 阶段二：编辑器基线

目标：

- 保证 Markdown-first 编辑稳定可用。

### 阶段三：sidecar 增强

目标：

- 在不污染正文的前提下引入少量高价值持久化能力。

### 阶段四：质量与验证

目标：

- 建立长期可维护的工程质量基线。

## 风险与控制

### 风险 1：需求滑向富文本系统

控制：

- 明确坚持“先问是否属于 Markdown 语义，再决定是否进入 sidecar”。
- 首阶段严格限制在只读链路。

### 风险 2：搜索索引实现复杂度扩散

控制：

- 搜索必须经由独立模块访问，不把具体引擎细节扩散到前端和其他服务模块。

### 风险 3：阅读态与后续编辑态风格断裂

控制：

- 统一样式 token 和组件映射规则。
- 优先让阅读器和编辑器共用节点渲染约定。

## 关键决策记录

1. 内容真相选择 Markdown，而不是专有 block JSON。
2. 首发运行形态改为“本地服务优先 + Web 前端”。
3. 最低 Node 版本提高到 `22.13+`。
4. 用 `Milkdown` 作为编辑器核心，避免系统天然偏向富文本模型。
5. 用 `remark` / `rehype` 构建阅读态渲染链。
6. 用 `*.md.meta.json` 承载少量高价值展示状态。
7. 首阶段只做读取、导航、搜索和状态恢复，不写回 Markdown。
8. 第一版放弃持久化多彩字体等高破坏性能力。
