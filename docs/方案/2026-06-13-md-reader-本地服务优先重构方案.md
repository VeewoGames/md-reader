# md-reader 本地服务优先重构方案

## 方案概述

### 1. 总体目标和范围

本方案将 `md-reader` 的首发运行形态从“纯浏览器目录授权优先”修订为“本地服务优先 + Web 前端”，目标是在上千 Markdown 文档的项目场景下，把项目接入、文件树加载、文档读取、标题提取、搜索和状态恢复的性能作为一等约束。

本轮范围：

- 统一运行形态为本地 Node 服务 + 浏览器前端。
- 最低 Node 版本显式提高到 `22.13+`。
- 首阶段只做读取、导航、搜索和状态恢复，不进入写回 Markdown 或 sidecar 保存。
- 本地服务负责项目扫描、文件树构建、文档读取、搜索索引与缓存。
- 前端负责阅读、标题导航、profile/local 状态恢复与工作区交互。
- 保留 `Markdown-first + sidecar metadata + Notion-like` 的长期路线，不改正文真相边界。

本轮不包含：

- 桌面宿主打包。
- sidecar 完整写回链路。
- Markdown 写回保存。
- 多人协同、远程同步、云端项目中心。
- 非本机环境下的远程文件访问协议。

### 2. 各阶段任务概要

第一阶段：运行形态重构与运行器复用

- 主要工作：新增本地 bridge 服务，统一前端项目接入入口，移除对 `showDirectoryPicker()` 作为主链路的依赖，并复用 `data-editor` 的 `open/stop/health` 运行器约定。
- 预期成果：Chrome、Codex 内嵌浏览器、本机其他浏览器都通过同一条本地服务链路读取项目。
- 执行顺序：先完成本地服务与前端 provider 抽象，再替换当前浏览器目录授权入口。

第二阶段：文件树与文档读取性能基座

- 主要工作：实现项目扫描缓存、文件树快照、按需文档读取、标题抽取缓存与恢复链路。
- 预期成果：上千文档项目首屏加载和文档切换明显快于纯浏览器递归扫描。
- 执行顺序：先做只读文件树和文档内容 API，再补缓存和增量刷新。

第三阶段：索引与搜索能力

- 主要工作：建立全文搜索能力、最近文档、标题索引和增量重建策略。
- 预期成果：全文搜索、标题导航、最近打开文档和预热加载可在大文档库下稳定运行。
- 执行顺序：先做标题与路径索引，再接全文索引和增量更新。

第四阶段：阅读器/编辑器与保存扩展

- 主要工作：让阅读器、标题导航和后续编辑器都消费统一的本地服务接口，并在首阶段完成后再扩写保存链路。
- 预期成果：工作区、阅读态、标题导航、后续编辑态共享一套项目和文档来源，不再分裂。
- 执行顺序：先接阅读器与标题导航，再为编辑器预留保存接口。

### 3. 整体结构框架

系统改为五层：

- Markdown 正文层：磁盘中的原生 `.md` / `.mdx` 文件，是唯一内容真相。
- Sidecar 元数据层：承载不适合写入正文的展示状态，后续继续按独立文件管理。
- Local Service 层：负责项目发现、目录扫描、缓存、索引、文档读取和后续保存。
- Web Workspace 层：负责 profile/local 状态、模式切换、工作区 UI 和接口编排。
- Renderer / Editor 体验层：负责 Notion-like 阅读与编辑体验。

建议的实现模块：

- `server/`
  - `bridge-server.mjs`
  - `project-registry.mjs`
  - `project-scanner.mjs`
  - `document-reader.mjs`
  - `heading-indexer.mjs`
  - `search-indexer.mjs`
  - `cache-store.mjs`
  - `file-watcher.mjs`
- `src/workspace`
  - `workspace-provider.ts`
  - `local-bridge-access.ts`
  - `registry.ts`
  - `profile-store.ts`
  - `local-state.ts`
- `src/renderer`
- `src/editor`
- `src/outline`
- `src/metadata`
- `src/shared`

## 一、为什么要从纯浏览器主路径切到本地服务优先

当前纯浏览器主路径存在三个根问题：

1. 运行环境不一致

- Chrome 支持 `showDirectoryPicker()` 和浏览器本地持久化。
- Codex 内嵌浏览器当前不支持 `showDirectoryPicker()`，也缺少可用的 `indexedDB` 运行能力。
- 结果是不同浏览器下项目接入链路分裂，无法形成统一产品行为。

2. 大文档库性能上限偏低

- 目录递归扫描、文档批量读取、标题提取和全文搜索都放在前端，会和渲染线程竞争资源。
- 上千文档场景下，首屏文件树构建、切换项目、刷新恢复、索引预热都会更慢。

3. 后续功能扩展受限

- 最近文档、全文搜索、增量扫描、项目级缓存、文件变更监听都更适合放在本地服务侧。
- 若继续坚持纯浏览器主路径，后面迟早还要补本地桥接，届时成本更高。

因此，本轮不建议再维持“纯浏览器为主，Codex fallback 为辅”的双通道长期架构，而是直接把首发运行形态统一到本地服务优先。

## 二、推荐运行形态

### 2.1 统一形态

推荐形态：

- 本地启动一个只监听 `127.0.0.1` 的 Node 服务。
- Web 前端统一访问本地服务 API。
- 浏览器不再直接扫描项目目录。
- `showDirectoryPicker()` 可保留为未来可选导入助手，但不再作为主链路。

### 2.2 和 data-editor 的关系

本方案和 `data-editor` 在运行形态上保持一致：

- 都是本地服务 + Web 前端。
- 都通过本机服务读取项目目录。
- 都不依赖浏览器本身支持目录授权才能工作。

但 `md-reader` 仍与 `data-editor` 保持职责差异：

- `data-editor` 偏结构化 JSON / CSV 数据编辑。
- `md-reader` 偏 Markdown 阅读、标题导航、搜索和后续编辑增强。

本轮不建议直接复用 `data-editor` 的完整服务本体，而是采用“运行器约定复用 + Markdown 独立服务实现”的方式，避免把数据编辑器的领域模型强行引入 Markdown 系统。

明确区分如下：

- 直接复用：`open`、`stop`、`health`、本地 `127.0.0.1` 服务、项目参数传入、运行状态探测、浏览器打开方式。
- 选择性复用：项目 registry 文件位置、日志与 PID 管理、健康检查格式。
- 不直接复用：`data-editor` 的 adapter、collection、结构化数据模型、详情面板和主表概念。

## 三、服务端总体设计

### 3.1 本地服务边界

建议新增独立服务目录：

```text
server/
  bridge-server.mjs
  project-registry.mjs
  project-scanner.mjs
  document-reader.mjs
  heading-indexer.mjs
  search-indexer.mjs
  cache-store.mjs
  file-watcher.mjs
```

服务职责：

- 维护项目 registry。
- 扫描项目中的 Markdown 文档。
- 构建文件树。
- 读取文档原文。
- 提取标题结构。
- 构建搜索索引。
- 维护缓存和增量刷新。

### 3.2 服务监听方式

建议默认端口：

- `http://127.0.0.1:8797`

边界要求：

- 仅监听 `127.0.0.1`。
- 首发仅暴露本机只读能力，后续再扩写回。
- 所有文件访问必须落在已注册项目根目录下，禁止任意路径穿透。

### 3.3 服务启动方式

建议新增脚本：

- `npm run open`
- `npm run stop`
- `npm run health`
- `npm run service:dev`

其中：

- `open`：首发主入口。负责检查或启动服务、写入当前 project context、打开浏览器。
- `stop`：停止 `md-reader` 本地服务。
- `health`：输出服务健康状态、当前项目、索引状态和端口信息。
- `service:dev`：前端开发 + 本地 bridge 联调。

这里有意不照搬 `data-editor` 的整套内部脚本命名，而是保留用户真正需要的最小运行器表面：

- `open/stop/health` 是用户可见约定。
- 进程拉起、PID 文件、日志目录、端口占用处理可以在 `scripts/runner/` 内部实现。
- 如果后续确实需要更细粒度调试入口，再补 `service:start` 等内部脚本。

### 3.4 运行时 source of truth

运行时状态需要有明确归属，避免前端和服务各记一份：

- 服务存活与端口占用：以本地 runner 状态文件和健康检查为准。
- 项目 registry：以本机 profile registry 文件为准。
- 当前 active project：以 profile registry 为准。
- 当前 active document、scroll progress、最近阅读位置：以本机 local state 为准。
- 文件树、标题缓存、搜索索引：以服务缓存目录为准。

### 3.5 dev / prod 形态

开发态：

- `npm run open -- --project <root>` 启动本地服务。
- 前端继续通过 Vite dev server 提供页面。
- 本地服务只提供 `/api/*` 与运行状态。

首发交付态：

- 仍然是 Web 核心。
- 可由一个轻量静态前端 + 本地 Node 服务共同运行。
- 不要求先做 Electron/Tauri 壳。

## 四、前端总体设计

### 4.1 workspace provider 抽象

前端不应再直接调用浏览器目录 API，而应只依赖统一 provider：

- `workspace-provider.ts`
  - `listProjects(profileId)`
  - `registerProject(profileId, rootPath)`
  - `setActiveProject(profileId, projectId)`
  - `getFileTree(projectId)`
  - `readDocument(projectId, relativePath)`
  - `getHeadings(projectId, relativePath)`
  - `searchDocuments(projectId, query)`

这样后续阅读器、编辑器、标题导航和搜索都消费统一接口。

### 4.2 profile / local 分层继续保留

继续沿用当前已经收敛过的边界：

- `profile`
  - 项目 registry
  - active project
  - 长期工作区偏好
- `local`
  - active document path
  - reading progress
  - 当前模式
  - 临时工作现场

但数据来源变为本地服务与本机配置，而不是浏览器目录句柄。

### 4.3 UI 交互变化

顶部栏建议调整为：

- 项目切换：调用本地服务返回的项目列表。
- 接入项目：在 Codex / 本地服务模式下，改为“输入或选择本机项目根目录”。
- 状态区：显示服务状态、当前项目、当前文档、索引状态。

## 五、项目 registry 设计

### 5.1 source of truth

项目 registry 的 source of truth 不再是浏览器的 `IndexedDB`，而改为本机项目注册表文件。

建议路径：

```text
%APPDATA%/md-reader/projects.json
```

内容结构建议：

```json
{
  "version": 1,
  "profiles": {
    "default": {
      "activeProjectId": "nocturnel-7f24c3a1",
      "projects": [
        {
          "id": "nocturnel-7f24c3a1",
          "name": "Nocturnel",
          "root": "C:\\Code\\Nocturnel",
          "contentRoots": ["."],
          "adapter": "markdown"
        }
      ]
    }
  }
}
```

### 5.2 projectId 设计

`projectId` 不能直接拿展示名或临时序号，否则跨重启和重命名会漂。

建议规则：

- 默认 `projectId = slug(name) + '-' + shortHash(normalizedRootPath)`。
- `normalizedRootPath` 统一大小写、分隔符与尾部斜杠后再计算。
- 一旦创建写入 registry，后续不因项目显示名变化而重算。
- 若 root path 变化，视为新项目接入，而不是静默覆盖旧项目。

### 5.3 设计理由

- 项目记录属于个人配置层，不属于团队共享文档内容。
- 用本机 registry 能保证 Chrome、Codex、未来本机其他浏览器看到同一套项目列表。
- 前端无需再依赖浏览器是否支持目录句柄持久化。

## 六、文件树与文档读取性能方案

### 6.1 文件树加载策略

不建议每次前端刷新都重新全量递归扫盘。推荐：

- 服务启动时扫描项目并生成文件树快照。
- 对项目维护内存缓存和磁盘缓存。
- 前端只请求已构建好的树。

缓存结构建议：

```text
%APPDATA%/md-reader/cache/<projectId>/tree.json
```

### 6.2 文档读取策略

文档内容采用按需读取：

- 文件树阶段只返回路径、标题、mtime、size 等轻量信息。
- 点击文档时再读取原文。
- 最近打开文档可以做小规模预热缓存。

### 6.3 标题提取策略

建议服务端对每个文档缓存：

- 标题层级树
- 文档摘要
- mtime

这样右侧标题导航不需要每次在前端重新全量解析 Markdown。

## 七、全文索引与搜索方案

### 7.1 推荐索引技术

推荐把“搜索能力接口”作为首阶段目标，把 `SQLite FTS5` 作为优先实现候选，但隔离在搜索索引模块内部，不让前端和其他服务模块感知具体引擎。

理由：

- `SQLite FTS5` 适合本地单机全文检索场景。
- 可以把路径、标题、正文、tags、摘要一起建索引。
- 数据库文件单体可维护，便于本机缓存和增量更新。

同时要明确两个边界：

- 由于当前已接受最低 Node 版本提升到 `22.13+`，首阶段可以合法依赖较新的 Node 能力。
- 但搜索模块仍需做实现隔离，避免后续如需切换到其他索引引擎时牵动整个系统。

建议索引文件：

```text
%APPDATA%/md-reader/cache/<projectId>/index.sqlite
```

### 7.2 索引分层

建议分三类索引：

1. 文件元信息索引

- 相对路径
- 文档名
- 最近修改时间
- 文件大小

2. 标题索引

- 一级到六级标题
- 标题层级路径
- 标题文本

3. 正文全文索引

- 原始 Markdown 文本
- 可选标准化纯文本版本

### 7.3 增量更新

不建议每次启动都全量重建所有文档全文索引。推荐：

- 启动时读取现有缓存和索引版本。
- 对比文件 `mtime` / `size`。
- 只对新增、修改、删除文档做增量更新。

## 八、文件监听与增量刷新

### 8.1 监听策略

建议服务端增加文件系统监听：

- 监听项目根目录及内容根目录。
- 识别 `.md` / `.mdx` 的新增、修改、删除。

监听目标：

- 更新文件树快照
- 更新标题缓存
- 更新全文索引
- 向前端推送“项目已更新”

### 8.2 推送方式

首发建议简单实现：

- 前端轮询 `project version`

后续可升级为：

- `SSE` 或 `WebSocket`

原因：

- 首发先保证链路稳定和简单。
- 文件更新通知不是当前最大风险点，优先级低于扫描、读取和索引性能。

## 九、读取与缓存路径

### 9.1 首屏启动

推荐链路：

1. 前端请求 `/api/health`
2. 前端请求 `profile` 下项目 registry
3. 读取 active project
4. 服务返回缓存好的文件树
5. 前端根据 local state 恢复 active document
6. 前端按需请求文档内容和标题树
7. 前端按需请求搜索结果

### 9.2 最近文档预热

对大文档库，推荐首发就做一个很轻的预热：

- 当前文档
- 最近 3 到 5 个文档

不要在首屏预热全项目文档内容。

### 9.3 缓存层次

建议分层：

- 内存缓存：当前会话热数据
- SQLite / JSON 磁盘缓存：跨重启恢复
- 前端 local state：仅保存 UI 工作现场

## 十、API 设计

建议最小 API：

### 10.1 服务状态

- `GET /api/health`

返回：

```json
{
  "ok": true,
  "mode": "local-service",
  "projectsLoaded": 1
}
```

### 10.2 项目 registry

- `GET /api/profiles/:profileId/projects`
- `POST /api/profiles/:profileId/projects/register`
- `POST /api/profiles/:profileId/projects/active`

### 10.3 文件树

- `GET /api/projects/:projectId/tree`

### 10.4 文档内容

- `GET /api/projects/:projectId/document?path=<relativePath>`

### 10.5 标题树

- `GET /api/projects/:projectId/headings?path=<relativePath>`

### 10.6 搜索

- `GET /api/projects/:projectId/search?q=<query>`

### 10.7 本地状态恢复

- `GET /api/profiles/:profileId/state`
- `POST /api/profiles/:profileId/state/local`

## 十一、和现有代码的改造边界

### 11.1 保留的部分

这些可以继续保留：

- `src/app`
- `src/shared`
- `src/workspace/profile-store.ts`
- `src/workspace/local-state.ts`
- `TopBar` / `AppShell` / `WorkspaceLayout`

### 11.2 需要重构的部分

这些需要从浏览器目录模式迁出：

- `src/workspace/browser-project-access.ts`
- `src/workspace/registry.ts` 中以浏览器存储为 source of truth 的假设
- `App.tsx` 中直接围绕目录句柄恢复的链路

### 11.3 新增的关键入口

- `src/workspace/workspace-provider.ts`
- `src/workspace/local-bridge-access.ts`
- `server/bridge-server.mjs`

### 11.4 当前浏览器态数据迁移

已有浏览器态数据不能直接丢掉，需要一轮显式迁移：

- 首次启动本地服务版本时，前端读取旧的浏览器 `profile/local` 数据。
- 若旧数据存在且本机服务中尚无对应 profile，则导入 profile 偏好。
- 若旧数据包含最近文档、滚动位置、阅读进度，则写入新的 local state 文件。
- 旧的目录句柄不再作为长期 source of truth；只能用于一次性辅助识别“此前接入过哪些项目”。
- 迁移完成后，前端进入新的本机 registry + local state 模型。

## 十二、实施顺序

### 阶段 A：服务基座与运行器对齐

- 建 `server/bridge-server.mjs`
- 建 `project-registry.mjs`
- 建 `project-scanner.mjs`
- 打通 `health / projects / tree / document`
- 接入 `open/stop/health` 运行器脚本

完成标准：

- 可在本机读取 `C:\Code\Nocturnel`
- 可返回 Markdown 文件树

### 阶段 B：前端 provider 切换

- 引入 `workspace-provider.ts`
- 让 `App.tsx` 改为只调 provider
- 项目切换和文档切换改走本地 API
- 导入旧浏览器态 profile/local 数据

完成标准：

- Codex 内嵌浏览器和 Chrome 都能通过本地服务看到同一项目

### 阶段 C：缓存与标题索引

- 文件树缓存
- 标题抽取缓存
- local state 恢复当前文档与阅读进度
- 接入基础搜索入口与结果展示

完成标准：

- 刷新恢复项目和当前文档
- 标题导航可直接用缓存结果

### 阶段 D：全文索引

- `SQLite FTS5` 全文索引
- 增量重建
- 搜索 API

完成标准：

- 上千文档项目中搜索可用且响应稳定

## 十三、验收标准

完成后应满足：

- 在 Codex 内嵌浏览器中，不依赖 `showDirectoryPicker()` 也能打开 `C:\Code\Nocturnel`。
- 在 Chrome 中也通过同一本地服务链路访问项目，不再依赖浏览器目录授权作为主路径。
- 上千 Markdown 文档的项目可以稳定生成文件树。
- 项目切换、文档切换、标题导航恢复、最近阅读位置恢复和搜索都能工作。
- 项目记录保存在 profile 作用域。
- 文件位置和阅读进度保存在 local 作用域。
- 首阶段不写回 Markdown，未来接入保存时也不需要再次推翻运行形态。

---

## 十四、关键决策

本轮建议明确固化以下决策：

1. `md-reader` 的首发运行形态修订为“本地服务优先 + Web 前端”，不再以纯浏览器目录授权作为长期主链路。
2. 最低 Node 版本提高到 `22.13+`。
3. 对上千文档项目，性能优先于“尽量纯浏览器”的原则。
4. 首阶段范围严格收敛为读取、导航、搜索和状态恢复，不进入写回 Markdown。
5. 不做长期双通道架构；Chrome 和 Codex 都统一走本地服务主链路。
6. `profile` / `local` 分层保持不变，只把 source of truth 从浏览器能力迁到本地服务和本机 registry。
7. 搜索首选可用 `SQLite FTS5`，但必须通过独立搜索模块隔离实现细节。

## 十五、技术依据

本方案的技术依据主要来自：

- Node.js 官方文件系统能力，覆盖目录读取、文件读取和监听能力。
- SQLite FTS5 官方全文检索能力，适合本地单机索引场景。
- Vite 官方后端集成方式，适合本地服务 + 前端联调。
