# md-reader 本地服务模式编辑保存方案

## 方案概述

### 1. 总体目标和范围

本方案用于收敛 `md-reader` 下一阶段的“编辑与保存”能力，明确后续**只考虑本地服务模式**，不再以浏览器目录授权模式作为正式目标运行形态。

本轮方案聚焦以下范围：

- 所有 Markdown 文档编辑与保存都通过本地服务链路完成。
- 保持 `Markdown-first`，原始 `.md` 正文仍是唯一内容真相。
- 先补最小可用的编辑模式与分栏模式，再进入更强的 Markdown-first 编辑器核心。
- 保存目标是“尽量不改变原文结构语义”，而不是引入富文本专有数据模型。
- `profile` / `local` 分层继续保留，编辑工作现场与保存状态按职责分别落位。
- 后续正式实现中直接移除浏览器授权模式，不再保留双模式产品结构。

本方案不包含以下内容：

- 协同编辑、云端同步、多人冲突合并。
- 首轮就接入完整 `Milkdown` 富交互编辑器。
- 首轮就写回 `sidecar metadata`。
- 为拟态 Notion 而把正文切换为 block JSON。

### 2. 各阶段任务概要

第一阶段：本地服务可编辑最小链路

- 主要工作：补文档写回 API、前端原始 Markdown 编辑区、保存按钮、脏状态、基础错误提示。
- 预期成果：在本地服务模式下，用户可以打开 Markdown、修改正文并稳定保存回原始 `.md` 文件。
- 执行顺序：先补服务端写回接口，再接前端编辑 UI 和保存链路。

第二阶段：分栏编辑与预览一致性

- 主要工作：补 `split` 模式，在中间文档工作区内部做左右分栏；统一阅读态与编辑态的标题导航、滚动和保存后刷新行为。
- 预期成果：用户可以一边编辑原始 Markdown，一边看到接近最终阅读态的实时效果。
- 执行顺序：先保证编辑保存稳定，再接入分栏联动。

第三阶段：编辑体验增强

- 主要工作：在最小可用保存链路稳定后，再引入 `Milkdown` 或等价 Markdown-first 编辑器核心，增强块级交互、快捷插入和结构编辑。
- 预期成果：在不破坏 Markdown 真相边界的前提下，逐步接近更流畅的 Notion-like 编辑体验。
- 执行顺序：先验证 raw Markdown 编辑链路的边界，再替换或包裹编辑器内核。

第四阶段：sidecar 与一致性扩展

- 主要工作：把高价值但不适合写入正文的状态接到 `sidecar metadata`，并统一阅读、编辑、分栏三种模式下的表现。
- 预期成果：形成“正文真相 + 本地服务保存 + 有限增强状态”的稳定体系。
- 执行顺序：只有在正文写回链路稳定后才进入。

### 3. 整体结构框架

后续编辑保存能力建议固定为五层：

- Markdown 正文层：磁盘中的 `.md` / `.mdx` 文件，是唯一内容真相。
- Local Service 文档层：负责安全读取、写回、mtime 校验、冲突判定与文件监听。
- Workspace 状态层：负责当前项目、当前文档、模式、脏状态、保存状态、恢复状态。
- Renderer / Editor 体验层：负责阅读态渲染、raw 编辑区、分栏视图，以及后续 `Milkdown` 增强。
- Sidecar 元数据层：后续只承载不适合写入正文的展示状态，不与正文写回混用。

## 一、当前现状与问题

当前仓库已经有 `read / edit / split` 三种模式状态，但编辑能力尚未真正实现。

现有事实：

- [src/app/TopBar.tsx](../../src/app/TopBar.tsx) 已提供模式切换按钮。
- [src/App.tsx](../../src/App.tsx) 已保存 `activeMode` 到 local state。
- [src/app/AppShell.tsx](../../src/app/AppShell.tsx) 会把 `mode` 继续往下传。
- [src/app/WorkspaceLayout.tsx](../../src/app/WorkspaceLayout.tsx) 当前未消费 `mode`，中心区始终渲染阅读态 Markdown。

这意味着：

- “编辑”按钮目前只是在切换 UI 状态。
- 当前没有可编辑输入控件。
- 当前没有保存动作。
- 当前没有写回服务接口。
- 当前也没有脏状态、冲突检测和保存反馈。
- 当前代码中仍保留浏览器授权模式分支，但这条分支不再进入后续正式方案，应在编辑保存实施前清理。

所以这轮的核心不是修一个小 bug，而是正式补齐“编辑模式的最小产品闭环”。

## 二、目标运行形态

本方案明确规定，后续编辑保存只按以下运行形态设计：

- 前端运行在浏览器环境。
- 项目接入、文档读取、文档保存都通过本机 Node 服务。
- 服务只监听 `127.0.0.1:8797`。
- 浏览器本身不再承担目录句柄、文件句柄或直接写磁盘的职责。
- 浏览器授权模式从正式实现、测试和回归目标中直接移除。

由此带来的直接好处：

- Chrome、Codex 内嵌浏览器、未来本机其他浏览器都走同一条保存链路。
- 前端不需要处理浏览器写文件权限差异。
- 后续 dirty state、冲突检测、文件监听、自动刷新都可以统一在本地服务侧收敛。

## 三、产品边界与关键原则

### 3.1 正文真相边界

必须继续坚持：

- Markdown 正文是唯一内容真相。
- 保存回磁盘时优先保持原生 Markdown 结构。
- 不因为追求视觉编辑体验，把系统切换成富文本专有数据模型。

### 3.2 编辑器演进边界

本轮不要求一步到位直接上完整富交互编辑器。建议演进顺序是：

1. 原始 Markdown 文本编辑可用。
2. 分栏预览可用。
3. 保存链路稳定。
4. 再接 `Milkdown` 或其他 Markdown-first 编辑器增强体验。

这样做的原因是：

- 先证明“读 -> 改 -> 存 -> 刷 -> 恢复”闭环稳定。
- 先建立保存边界，再引入更复杂的编辑器状态同步。
- 避免在还没搞清写回语义时，被编辑器框架反向决定架构。

### 3.3 sidecar 边界

本轮仍然不把以下内容写入正文：

- 页面宽度
- 表格列宽
- callout 类型
- 自定义显示模式

这些后续仍应进入 `*.md.meta.json`，但不与第一版正文编辑保存链路耦合。

## 四、推荐产品形态

### 4.1 阅读模式 `read`

职责：

- 继续显示当前阅读态 Markdown。
- 支持标题导航、滚动同步、搜索结果跳转。
- 不提供输入焦点和保存动作。

### 4.2 编辑模式 `edit`

第一版建议形态：

- 中间区替换为单栏原始 Markdown 编辑区。
- 使用 `textarea` 或等价轻量代码编辑组件作为第一版实现。
- 右侧标题导航仍保留，但基于当前编辑内容重新计算标题树。

用户在 `edit` 模式下应看到：

- 原始 Markdown 文本
- 文档路径与保存状态
- `保存` / `放弃修改` / `重新载入` 等操作
- 当前是否有未保存修改

### 4.3 分栏模式 `split`

第一版建议形态：

- 只在**中间文档工作区内部**做左右分栏。
- 左半区是原始 Markdown 编辑区。
- 右半区是阅读态预览。
- 工作区最右侧现有标题导航栏继续独立保留，不被 `split` 模式挤占或替换。
- 标题导航继续以中间工作区右半区的预览内容为准。

推荐顺序：

- `edit` 先可用。
- `split` 在此基础上复用相同的编辑状态与保存状态。

理由：

- `split` 本质上是 `edit + preview` 的组合，不应另造一套状态模型。

## 五、服务端设计

### 5.1 新增文档写回接口

建议在现有 bridge 服务基础上新增：

- `PUT /api/projects/:projectId/document`

请求体建议：

```json
{
  "profileId": "default",
  "path": "docs/guide.md",
  "content": "# Title\n\nUpdated",
  "expectedMtimeMs": 1718265600000
}
```

返回建议：

```json
{
  "ok": true,
  "path": "docs/guide.md",
  "mtimeMs": 1718265601234,
  "size": 2048
}
```

同时，读取接口也必须同步升级，不能再只返回正文内容。建议把文档读取统一为返回：

```json
{
  "path": "docs/guide.md",
  "content": "# Title\n\nBody",
  "mtimeMs": 1718265601234,
  "size": 2048
}
```

没有这项 `mtimeMs`，前端就无法为第一版冲突保护提供可靠的 `expectedMtimeMs`。

### 5.2 保存前校验

服务端保存前必须做以下检查：

1. `projectId` 必须属于当前已注册项目。
2. `path` 必须落在允许内容根下，不能穿透到项目外。
3. 文件扩展名必须是 `.md` 或 `.mdx`。
4. 若请求带 `expectedMtimeMs`，则与当前磁盘状态做比对。

### 5.3 冲突处理

第一版推荐最小策略：

- 如果当前磁盘文件 `mtime` 与前端读取时记录的 `expectedMtimeMs` 不一致，则拒绝保存并返回 `409 conflict`。
- 前端收到冲突后，提示：
  - 文件已被外部修改
  - 可选择重新载入磁盘版本
  - 或后续再补“对比合并”能力

第一版不建议直接做自动 merge。

### 5.4 写回实现边界

服务端写回职责只包括：

- 读取当前磁盘版本
- 做冲突校验
- 写入新内容
- 返回新 `mtime`
- 触发缓存失效

不在第一版服务端职责内的内容：

- Markdown lint 自动修复
- frontmatter 排版重写
- prettier 格式化
- sidecar 联动写回

### 5.5 缓存失效

保存成功后，服务端应立即失效以下缓存：

- 当前文档正文缓存
- 当前文档标题缓存
- 当前文档搜索索引条目

如果已接文件监听，可让保存写回和 watcher 汇合到同一刷新机制；否则保存接口内部直接刷新。

## 六、前端状态模型

### 6.1 当前缺口

当前前端只有：

- `currentDocumentPath`
- `currentDocumentContent`
- `mode`
- `isDocumentLoading`

编辑保存至少还需要：

- `editingDocumentContent`
- `savedDocumentContent`
- `isDocumentDirty`
- `isDocumentSaving`
- `documentMtimeMs`
- `saveErrorMessage`

### 6.2 推荐状态定义

建议在 [src/App.tsx](../../src/App.tsx) 主链路里新增：

```ts
type DocumentSessionState = {
  path: string | null
  savedContent: string | null
  editingContent: string | null
  mtimeMs: number | null
  isDirty: boolean
  isSaving: boolean
  lastSaveError: string | null
}
```

### 6.3 状态职责划分

- `savedContent`：最近一次成功从磁盘读取或写回成功后的内容。
- `editingContent`：用户当前正在编辑的内容。
- `isDirty`：`editingContent !== savedContent`。
- `mtimeMs`：磁盘版本锚点，用于冲突校验。

这样区分的价值是：

- 阅读态和编辑态都能引用 `savedContent` 作为已提交版本。
- 编辑器可以自由修改 `editingContent`，不污染阅读态真相。
- 保存成功后，再把 `savedContent` 与 `editingContent` 对齐。

## 七、前端交互设计

### 7.1 进入编辑模式

用户点击 `编辑`：

- 若当前没有打开文档，则只切模式，不显示输入区。
- 若当前有文档，则将 `editingContent` 初始化为 `savedContent`。
- 中间区渲染编辑器，而不是只读 Markdown 视图。

### 7.2 编辑中的标题导航

标题导航需要继续可用，但来源要改成：

- `read`：使用 `savedContent`
- `edit`：使用 `editingContent`
- `split`：右侧预览使用 `editingContent` 渲染，标题导航也跟着 `editingContent`

这样用户改标题后，右侧导航会立刻更新，而不是等保存后才更新。

### 7.3 保存交互

推荐第一版提供三种动作：

- `Ctrl+S` / `Cmd+S`
- 顶部或文档头部 `保存`
- `重新载入`

行为规则：

- 无修改时保存按钮禁用。
- 保存中按钮显示忙碌状态。
- 保存成功后刷新 `savedContent`、`mtimeMs` 和阅读态/导航态。
- 保存失败后保留 `editingContent`，不丢用户输入。

### 7.4 放弃修改

第一版建议提供：

- `放弃修改`

行为：

- 将 `editingContent` 重置为 `savedContent`
- 清空错误状态
- `isDirty = false`

### 7.5 模式切换规则

建议规则：

- `read -> edit`：进入编辑区
- `edit -> split`：复用当前编辑内容
- `split -> read`：若有未保存修改，保留内存状态，但阅读态仍显示 `savedContent`

注意：

- 不建议在用户离开 `edit` 模式时自动保存。
- 也不建议在用户离开 `edit` 模式时自动丢弃修改。

### 7.6 未保存修改的统一拦截规则

只要当前文档存在未保存修改，以下动作都必须走统一确认流程：

- 切换文档
- 切换项目
- 切换 profile
- 重启服务
- 关闭服务
- 刷新页面
- 离开页面

统一弹出三选项确认：

- `保存`
- `不保存`
- `取消`

行为定义：

- `保存`：先执行保存；保存成功后继续原始动作；保存失败则停留在当前文档。
- `不保存`：放弃当前未保存修改，再继续原始动作。
- `取消`：中断原始动作，继续留在当前文档。

为了避免分散判断，这套逻辑应抽成统一的文档会话守卫，而不是在每个入口各写一份。

## 八、最小实现方案

### 8.1 第一版技术路线

第一版不直接上 `Milkdown`，而是：

- `edit`：原始 Markdown 文本编辑
- `split`：原始 Markdown 文本编辑 + `react-markdown` 预览
- 保存：通过 bridge API 写回原始文件

推荐原因：

- 改动小，能快速证明链路可用。
- 保存结果完全可控，不会被复杂编辑器中间态污染。
- 便于后续把编辑区替换成更强的 Markdown-first 编辑器。

### 8.2 具体落点

建议新增模块：

- `src/editor/raw-markdown-editor.tsx`
- `src/editor/document-save-bar.tsx`
- `src/workspace/document-session.ts`

并改造：

- `server/bridge-server.mjs`
- `server/project-reader.mjs`
- 新增 `server/project-writer.mjs`
- `src/workspace/local-bridge-access.ts`
- `src/App.tsx`
- `src/app/AppShell.tsx`
- `src/app/WorkspaceLayout.tsx`

并移除：

- `src/workspace/browser-project-access.ts`
- `src/App.tsx` 中面向浏览器授权模式的正式运行分支
- `showDirectoryPicker()`、目录句柄恢复和浏览器权限状态相关的正式产品逻辑

这里不是“继续兼容但不推荐”，而是后续正式实现中**直接移除浏览器授权模式**，避免编辑保存阶段继续被双模式复杂度拖住。

实施顺序上，这项移除不应拖到编辑保存完成后再处理，而应作为第一阶段正式编码前置步骤先完成。否则后续每个入口都要继续背负双模式分支，和本方案目标直接冲突。

### 8.3 为什么不是直接 Milkdown

当前最需要先稳定的是：

- 保存边界
- 冲突处理
- dirty state
- 文档切换行为
- 阅读/编辑/分栏三种模式的状态一致性

这些问题不先收敛，直接接 `Milkdown` 只会让调试面更大。

## 九、第二阶段增强路线

当第一版 raw 编辑保存稳定后，再进入第二阶段增强：

### 9.1 引入 Markdown-first 编辑器

优先候选仍是：

- `Milkdown`

原因不变：

- 更容易维持 Markdown 输出边界。
- 更适合在“原始 Markdown 是真相”的体系中演进。

### 9.2 增强目标

第二阶段可以逐步补：

- 标题快捷插入
- callout 快捷包裹
- 表格编辑辅助
- block hover 工具栏
- slash menu
- 折叠与结构跳转

但这些都必须建立在：

- 仍可稳定回落到标准 Markdown
- 保存结果可预测
- 不把系统推向 block JSON

## 十、状态恢复与本地持久化

### 10.1 继续保留的 local 状态

适合继续放在 `local state`：

- 当前模式
- 当前文档路径
- 阅读进度
- 侧栏宽度
- 右侧导航宽度

### 10.2 不建议持久化的编辑中状态

第一版不建议持久化：

- 未保存的 `editingContent`

原因：

- 这会引入“磁盘版本”和“本地草稿版本”两套真相。
- 与外部编辑器改动后的冲突处理会立刻复杂化。

第一版建议：

- 未保存修改只保留在当前前端会话内存中。
- 刷新页面或重启服务后，默认以磁盘版本重新打开。
- 但若刷新页面或离开页面前存在未保存修改，仍必须先经过 `保存 / 不保存 / 取消` 三选项确认。

如果后续要做草稿恢复，应另开一轮方案，明确草稿文件位置、冲突合并和清理策略。

## 十一、错误与冲突处理

### 11.1 可预期错误

第一版必须覆盖：

- 服务不可用
- 路径无效
- 文件已删除
- 文件已被外部修改
- 写入失败

### 11.2 前端反馈建议

建议统一成文档级状态条：

- `未保存修改`
- `正在保存…`
- `已保存`
- `保存失败：<message>`
- `文件已被外部修改，请重新载入`

### 11.3 用户动作

冲突后建议只提供两个明确动作：

- `重新载入磁盘版本`
- `保留当前修改`

第一版不做三方合并 UI。

## 十二、验收标准

完成后至少应满足：

- 在本地服务模式下，打开任意 Markdown 文档后，切到 `编辑` 模式能看到可编辑输入区。
- 用户修改 Markdown 后，`isDirty` 能正确变化。
- 点击保存或按 `Ctrl+S` 能通过本地服务把内容写回原始 `.md`。
- 保存成功后，阅读态、分栏预览和标题导航都刷新到新内容。
- 磁盘文件被外部修改时，保存会被冲突保护拦截，不会静默覆盖。
- 无未保存修改时，切模式与切文档不会报错。
- 现有本地服务工作区、标题导航、文件树和状态恢复链路不被破坏。

---

## 十三、非目标

本轮明确不做：

- 自动格式化保存
- 自动 merge 冲突
- 本地草稿恢复
- sidecar 写回
- 富文本颜色、字体、任意块样式
- 协同编辑
- 保留浏览器授权模式作为正式可用分支

---

## 十四、关键决策

1. 后续编辑与保存能力只按本地服务模式设计，不再维持正式的浏览器授权模式兼容路线。
2. 第一版先做 raw Markdown 编辑和分栏预览，不直接上完整富交互编辑器。
3. 保存链路统一通过本地 bridge API 写回原始 `.md`，前端不直接操作磁盘。
4. 第一版冲突处理采用 `mtime` 乐观并发保护，拒绝静默覆盖。
5. `split` 模式固定为中间工作区内部左右分栏，右侧标题导航栏继续独立保留。
6. 未保存修改统一通过 `保存 / 不保存 / 取消` 三选项确认守卫处理。
7. 第一版不持久化未保存草稿，仍以磁盘正文作为唯一真相。
8. `Milkdown` 等编辑器增强只在最小可用保存链路稳定后接入。
