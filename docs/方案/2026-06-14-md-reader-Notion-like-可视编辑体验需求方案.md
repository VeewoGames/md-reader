### 1. 总体目标和范围

本方案用于收敛 `md-reader` 下一阶段的 **Notion-like 可视编辑体验** 需求。核心目标不是复刻 Notion 的完整内容模型，而是在继续坚持 `Markdown-first` 的前提下，让编辑态尽量接近阅读态，让用户在编辑时看到的就是接近最终效果的页面，而不是长期停留在 `raw textarea + preview` 的双模式心智里。

本方案聚焦以下范围：

- 明确 `md-reader` 的主编辑模式应从原始 Markdown 文本编辑，演进为 **Markdown-first 可视编辑器**。
- 明确编辑态与阅读态的一致性目标，包括标题、列表、引用、代码块、任务列表等基础结构的实时可视化。
- 明确 `read / edit / source` 三种模式的新职责边界。
- 明确未被可视编辑器稳定结构化的内容，必须优先保真，不允许保存时自动重写。
- 明确哪些需求属于交互体验层，哪些属于持久化表达层，避免错误地把所有体验诉求都写入 Markdown 正文或 sidecar。

本方案不包含以下内容：

- 直接切换为专有 block JSON 内容模型。
- 直接追求 Notion 数据库、看板、日历、多列自由排版。
- 在本轮需求阶段承诺具体实现时序细节，例如精确到某个插件或某个命令的接入顺序。

### 2. 各阶段任务概要

第一阶段：重新定义主编辑模式

- 主要工作：把“编辑模式”从 raw Markdown 文本编辑重新定义为 Markdown-first 可视编辑。
- 预期成果：明确 `edit` 不再只是源码输入区，而是接近阅读态的结构化编辑页。
- 执行顺序：先明确体验边界，再决定实现路线。

第二阶段：拆分体验需求与持久化需求

- 主要工作：区分哪些能力只需要前端交互层支持，哪些能力必须写入 Markdown 或 sidecar。
- 预期成果：避免为了体验提升而污染正文真相边界。
- 执行顺序：先分层，再定义功能优先级。

第三阶段：收敛模式职责与验收标准

- 主要工作：明确 `read / edit / source` 新职责、编辑核心能力、非目标能力与阶段性验收口径。
- 预期成果：为后续 `Milkdown` 接入和工作区模式重排提供统一需求依据。
- 执行顺序：先明确主模式，再定义源码兜底模式。

### 3. 整体结构框架

本方案下，编辑体验相关需求分成四层：

- Markdown 正文层：仍是唯一内容真相，保存结果必须可稳定回落到原生 Markdown。
- Renderer / Editor 体验层：负责编辑态和阅读态的一致性、块级交互和可视化呈现。
- Sidecar 元数据层：只承载高价值、非正文语义的持久化展示状态。
- Workspace 模式层：负责 `read / edit / source` 的工作区组织与行为切换。

## 一、需求重定义

### 1.1 当前用户诉求的真实含义

用户当前最核心的诉求不是“支持编辑”，而是：

- 切换到 `edit` 模式后，页面仍然保持接近阅读态的视觉结构。
- 用户输入 Markdown 触发语法时，当前块会实时变成对应语义结构。
- 用户不需要切到 `source` 才知道最终长什么样。

典型例子：

- 输入 `# ` 后立即变成 `H1`
- 输入 `## ` 后立即变成 `H2`
- 输入 `### ` 后立即变成 `H3`
- 输入 `- ` 或 `1. ` 后立即进入列表
- 输入 `> ` 后立即进入引用
- 输入 ````` ``` ````` 后进入代码块

这说明目标不是：

- `textarea` 输入 Markdown 源码
- 再去别处看预览

而是：

- **在编辑态本身就直接编辑结构化内容**
- **Markdown 输入只是触发结构语义的方式**

### 1.2 新的产品定义

`md-reader` 的主编辑体验应定义为：

- 一个 **Markdown-first 的可视编辑器**
- 编辑态和阅读态尽量共享页面结构、留白、层级和视觉语言
- 用户输入 Markdown 触发语法时，编辑器立即把它解释为结构化块
- 保存时仍然回落为标准 Markdown，而不是专有内部格式

因此，项目目标应表述为：

- 像 Notion 一样顺手地编辑
- 但不像 Notion 那样改写内容真相层

## 二、核心需求分层

### 2.1 交互体验需求

这类需求主要属于 `Renderer / Editor` 层，不一定需要持久化：

#### A. 基础输入顺滑

- 标题、列表、引用、代码块、任务列表的回车行为正确
- 退格行为符合块语义，而不是纯文本删除
- `Tab / Shift+Tab` 可以调整列表层级
- 中文输入法组合输入稳定
- 粘贴普通文本和 Markdown 时结果可预期
- 光标、选区、滚动跟随稳定

#### B. 编辑态的结构可视化

- 当前块有明确的编辑焦点
- 标题在编辑态直接显示为标题样式
- 列表、任务列表、引用、代码块在编辑态直接显示为对应结构
- 页面留白、段落节奏、层级关系尽量接近阅读态

#### C. 块级交互感

- block hover
- slash menu
- 块间插入
- 轻量 block toolbar 或快捷操作入口
- 后续可演进到 folding / drag handle，但不作为首轮刚性需求

#### D. 结构导航同步

- 标题导航在编辑态继续可用
- 当前编辑内容变化时，标题树实时刷新
- 点击目录项跳转到对应块
- 滚动正文时目录 active 跟随

### 2.2 持久化表达需求

这类需求不能只谈交互，必须回答“存到哪里”：

#### A. 仍然属于 Markdown 正文的

- 标题层级
- 列表结构
- 任务列表
- 引用
- 代码块
- 表格正文

这些都应继续保存回原生 Markdown。

#### B. 适合进入 sidecar 的

- 页面宽度
- 表格列宽
- callout 类型
- 图片展示模式
- 少量高价值、非正文语义的展示增强

#### C. 不应进入 sidecar 的

- 当前滚动位置
- 当前高亮标题
- 标题导航折叠状态
- 文件树展开状态
- 当前模式 `read / edit / source`

这些属于 `profile / local` 或工作区现场状态，不属于文档内容状态。

## 三、主编辑模式的重新定义

### 3.1 当前模式问题

当前仓库里的模式语义仍偏向：

- `read`：阅读态
- `edit`：原始 Markdown 文本编辑
- `split`：源码 + 预览

这套模式适合做最小可用编辑闭环，但不符合当前用户最核心的体验诉求。

因为对用户来说：

- `edit` 不应该只是源码模式
- `source` 不应该是“唯一能看到效果的模式”

### 3.2 新的模式职责

建议把工作区模式重排为：

#### `read`

职责：

- 纯阅读
- 保留标题导航、搜索跳转、滚动同步
- 不承担编辑输入

#### `edit`

职责：

- **主编辑模式**
- 使用 Markdown-first 可视编辑器
- 编辑态本身尽量接近阅读态
- 输入 Markdown 触发语法时实时结构化
- 标题导航、块结构、排版层级都继续可见

这应是未来最主要、最常用的文档工作模式。

#### `source`

职责：

- 精确源码模式，不承担主视觉编辑职责
- 用于查看、检索、精确修改原始 Markdown 文本
- 面向复杂结构排查、保真修订、以及处理可视编辑器暂不稳定支持的片段

结论：

- **主模式是 `edit`**
- **`source` 是精确源码模式**
- **不是反过来**

### 3.3 为什么移除 `split`

本轮不再保留 `split`，原因不是它完全没有价值，而是它会持续制造错误心智：

- 容易让用户以为“想看最终效果就得开双栏”
- 容易让实现层把 `edit` 继续做成半成品输入框
- 容易把真正需要的“源码兜底能力”与“工作区并排布局”混为一谈

本轮需求收敛后，正式模式只有三种：

- `read`：阅读
- `edit`：主可视编辑
- `source`：精确源码

如果后续需要临时对照能力，可以作为调试能力或内部开关重新评估，但不再作为正式工作区模式写入需求主线。

## 四、技术路线需求

### 4.1 为什么不能把 `textarea` 当正式目标

`textarea` 只能完成：

- 原始文本输入
- 最小可用保存闭环
- 临时过渡方案

它不能承载以下关键目标：

- 编辑态与阅读态一致
- Markdown 输入实时结构化
- 块级编辑感
- 像 Notion 一样的页面内编辑体验

所以 `textarea` 只能是阶段性过渡，而不应继续作为正式主编辑器目标。

### 4.2 为什么主路线应是 Markdown-first 可视编辑器

基于当前项目约束，主路线应继续是：

- `Markdown-first`
- `Milkdown` 优先
- 保存结果稳定回落到 Markdown

原因：

- 目标是“像 Notion 的体验”，不是“像 Notion 的数据模型”
- 必须同时满足：
  - 编辑态可视化
  - Markdown 正文是真相
- 这比 raw textarea 更符合用户诉求，也比 block-based 专有模型更符合项目边界

### 4.3 对编辑器核心的需求表达

后续编辑器核心至少要满足：

- 支持 Markdown 语义块的实时结构化呈现
- 支持 Markdown 输入触发语法即时转块
- 支持可稳定导出原生 Markdown
- 支持对不可稳定结构化内容做保真降级，而不是强行改写
- 支持后续接入标题提取、slash menu、block hover、folding
- 不把系统天然推向专有 block JSON

### 4.4 未被稳定结构化内容的处理规则

这里的“未被可视编辑器稳定结构化的内容，统一保留为源码块/源码片段，只允许最小编辑，不做自动重写”，意思是：

- 如果某段内容可视编辑器不能稳定理解、稳定呈现、稳定 round-trip，就不要假装已经支持它
- 在 `edit` 模式里，这类内容应以“源码块”或“源码片段”的形式直接保留在文档流中
- 用户可以做最小必要修改，但系统不要把它自动转换成别的结构，也不要在保存时顺手重排、补格式或改写语义
- 如果用户要做精确修订，就切到 `source` 模式直接编辑原文

典型内容包括：

- frontmatter
- HTML comment，例如 `<!-- page_id: ... -->`
- 原始 HTML
- 当前 schema 未正式支持的自定义扩展语法
- 无法稳定 round-trip 的复杂表格、复杂嵌套列表、混合块边界

这一条规则的核心不是“显示成代码好看”，而是：

- **优先保真**
- **优先避免误改原文**
- **先承认边界，再逐步扩大稳定支持范围**

## 五、功能优先级

### 5.1 必须做

- 编辑态接近阅读态
- Markdown 触发语法即时结构化
- 标题、列表、引用、代码块、任务列表的基础输入行为稳定
- 标题导航在编辑态继续工作
- 保存结果仍然是标准 Markdown
- 未稳定结构化内容在编辑态可保真降级，不会因为一次保存被重写

### 5.2 应该做

- slash menu
- block hover / 当前块弱高亮
- 更自然的块间插入体验
- 编辑态与阅读态的页面宽度、排版节奏更一致
- `source` 模式提供精确源码查看与修订入口

### 5.3 可以延后

- 表格列宽持久化
- callout 类型持久化
- 页面宽度 sidecar
- 图片展示模式
- 更强的 folding / drag handle / 块拖拽

### 5.4 绝对不要做

- 为了拟态 Notion，把正文改成 block JSON
- 把所有体验需求都塞进 Markdown 扩展语法
- 把标题导航、文件树展开态、当前高亮标题写进 sidecar
- 把 `source` 继续当成唯一“看效果”的主编辑模式
- 对未稳定结构化内容做自动规范化、自动重排或静默改写

## 六、验收标准

需求方案落地后，后续实现至少应满足以下验收点：

1. 用户进入 `edit` 模式后，不再面对长期意义上的 raw Markdown 文本编辑器，而是面对 Markdown-first 的可视编辑页面。
2. 用户输入 `# `、`## `、`### `、`- `、`> `、代码块等 Markdown 触发语法时，当前块会即时转成对应结构。
3. 编辑态与阅读态在标题层级、段落节奏、留白和主要块结构上保持高一致性。
4. 标题导航在 `read` 和 `edit` 模式下都稳定工作，且继续只来源于 Markdown 语义结构。
5. 保存后磁盘内容仍然是原生 Markdown，不依赖专有中间格式。
6. `source` 模式只承担精确源码查看与修改，不替代 `edit` 的主可视编辑职责。
7. frontmatter、HTML comment、原始 HTML、未正式支持的扩展语法等内容，在 `edit` 模式下可以降级为源码块或源码片段，但保存后必须保持原文稳定，不做自动重写。
8. 可视编辑器的支持范围必须有清晰边界；未被声明为稳定结构化支持的内容，不得在保存时被偷偷“规范化”。

---

## 七、结论

基于当前用户确认，`md-reader` 的编辑体验需求已经发生了一个关键收敛：

- 目标不再是“把 `edit` 做成更好用的源码输入框”
- 而是“把 `edit` 做成 Markdown-first 的可视编辑主模式”

因此，后续产品和技术路线应同步调整为：

- 先把保存与状态链路做稳
- 再尽快把主编辑模式从 raw textarea 升级为 `Milkdown` 类 Markdown-first 可视编辑器
- 正式工作区模式收敛为 `read / edit / source`
- 对未稳定结构化的内容采用“源码块 / 源码片段 + 最小编辑 + 禁止自动重写”的保真策略
- sidecar 只承载少量高价值的非正文语义状态

这条路线既保留 `Markdown-first` 的内容真相，也更准确地回应了“像 Notion 那样编辑”的真实诉求。
