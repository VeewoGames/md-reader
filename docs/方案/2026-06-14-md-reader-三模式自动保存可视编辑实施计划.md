## 方案概述

> 当前有效实现主线：本执行计划。
> [2026-06-14-md-reader-Notion-like-可视编辑体验需求方案.md](../../docs/方案/2026-06-14-md-reader-Notion-like-可视编辑体验需求方案.md) 中关于 `read / edit / source` 的模式定义已被本执行计划覆盖，不再作为正式实现边界。

### 1. 总体目标和范围

本执行文档用于指导 `md-reader` 从当前的 `read / edit / split + textarea` 临时编辑形态，收敛到正式的 `read / edit / split` 三模式工作区，并接入 **data-editor 风格自动保存** 与第一版 **Markdown-first 可视编辑器**。

本轮总体目标有四个：

- 把正式工作区模式统一为 `read / edit / split`，保留当前分栏形态，不再新增独立 `source` 源码视图。
- 把保存交互改成自动保存：输入后 debounce 保存，切文档/切项目/重启服务/停服务前自动 flush，不再弹三选项确认框。
- 把 `edit` 从 `textarea` 升级为第一版可视编辑器，并保证和 `read` 的阅读体验尽量一致。
- 对当前可视编辑器尚不能稳定结构化的内容，采用 **局部源码片段降级**；需要精确修订时，走 `split` 左侧源码栏，而不是独立源码模式。

本执行文档只覆盖本地服务正式形态，不再把浏览器授权模式纳入产品主线。它也不覆盖 Notion 数据库、多列自由布局、任意富文本样式等超出 `Markdown-first` 边界的能力。

### 2. 各阶段任务概要

阶段 A：模式与状态骨架重排

- 主要工作：固定 `read / edit / split` 为正式模式，纠正 `split` 的职责定义。
- 预期成果：工作区三模式语义稳定，`split` 不再被视为待移除旧模式。
- 执行顺序：先做类型和状态，再做工作区布局。

阶段 B：自动保存闭环落地

- 主要工作：建立 `dirty / saving / saveError / flush` 状态机，接通本地服务保存写回。
- 预期成果：编辑过程自动保存，切文档/切项目/重启服务/关闭服务前可自动 flush，失败时阻断切换并提示错误。
- 执行顺序：先测试保存调度，再补 App 状态和 bridge 写回。

阶段 C：第一版 `split` 分栏模式成型

- 主要工作：保留当前分栏，但把它升级成正式模式：左侧源码编辑，右侧预览。
- 预期成果：`split` 成为稳定的 Markdown 精确修订与预览对照模式。
- 执行顺序：先接 draft 共享，再补左右栏职责。

阶段 D：第一版可视编辑器接入

- 主要工作：在 `edit` 模式接入 Markdown-first 可视编辑器，支持第一版中档语法范围。
- 预期成果：`edit` 不再是原始源码输入框，而是接近阅读态的结构化编辑页。
- 执行顺序：先最小文档模型，再接 UI，再接 Markdown round-trip。

阶段 E：局部源码片段降级与一致性收尾

- 主要工作：把 unsupported content 识别为源码片段，并保证 `edit <-> split` 共用同一份 draft。
- 预期成果：复杂内容不会拖垮整篇文档的可视编辑，也不会在保存时被静默改写。
- 执行顺序：先识别与标注，再接序列化回写，再补往返测试。

阶段 F：验证与收尾

- 主要工作：补行为测试、集成验证和文档收口。
- 预期成果：三模式、自动保存、局部降级和标题导航联动具备可回归证据。
- 执行顺序：单元测试优先，集成验证随后，最后更新 truth / ADR。

### 3. 整体结构框架

本轮实施完成后，系统结构收敛为五层：

- Markdown 正文层：磁盘上的 `.md`，仍是唯一内容真相。
- 文档 draft 层：当前打开文档的内存草稿，`edit` 与 `split` 共用。
- Editor / Renderer 层：可视编辑器、阅读态渲染器、分栏源码编辑与源码片段降级展示。
- Workspace 状态层：模式、当前文档、阅读进度、自动保存状态、恢复状态。
- Local-service 持久化层：文档读取、写回、冲突保护、项目与 profile 切换。

## 二、目标状态定义

### 2.1 正式三模式

#### `read`

- 纯阅读模式。
- 使用稳定的 `remark / rehype` 渲染链路。
- 标题导航、搜索跳转、滚动同步继续保留。

#### `edit`

- 主编辑模式。
- 使用 Markdown-first 可视编辑器。
- 所见即所得程度尽量接近 `read`。
- 只在局部命中 unsupported content 时降级为源码片段。

#### `split`

- 正式分栏模式。
- 保留当前“源码编辑 + 预览”双栏工作区，用于复杂结构排查、精确 Markdown 修订和可视效果对照。
- 与 `edit` 共用同一份 draft，不是另一套独立内容状态。

### 2.2 自动保存行为

自动保存按 `data-editor` 风格收敛为以下规则：

- 用户修改后进入 `dirty`。
- 连续输入期间通过 `600-1000ms debounce` 自动保存，只保留最后一次待保存请求。
- 切文档、切项目、重启服务、关闭服务前先做 `flush save`。
- 保存成功则直接切换；保存失败则阻断切换并给出明确错误提示。
- 模式切换不弹确认框，也不单独保存；因为 `edit` 与 `split` 共用同一份 draft。
- 所有会中断 bridge 或离开当前文档上下文的动作都必须先 flush，包括切文档、切项目、重启服务、关闭服务。
- `edit / split -> read` 也必须先 `flush`；本轮不允许 dirty 状态下直接切回只读视图，否则用户会看到旧的 `savedDocumentContent`，误判为改动丢失。

### 2.3 第一版可视编辑范围

第一版 `edit` 支持以下 Markdown 结构：

- 段落
- `H1-H6`
- 无序列表
- 有序列表
- 任务列表
- 引用
- 代码块
- 分隔线
- 行内代码
- 粗体
- 斜体

本轮明确不纳入第一版稳定结构化范围：

- 表格
- callout 类型持久化
- frontmatter 可视结构化编辑
- 原始 HTML 可视结构化编辑
- 任意自定义扩展语法

### 2.4 Unsupported Content 降级规则

第一版采用 **局部降级**，不是文档级降级：

- 能稳定结构化的块继续保持可视编辑。
- 当前不能稳定识别、不能稳定 round-trip 的片段，以“源码片段”方式嵌在文档流里。
- 用户可在源码片段中做最小编辑，但系统不做自动重排、自动规范化和静默改写。
- 如果用户需要精确修订这些内容，可切到 `split` 模式左侧源码栏。

典型的源码片段候选：

- frontmatter
- HTML comment，例如 `<!-- page_id: ... -->`
- 原始 HTML
- 当前不支持的扩展语法
- 无法稳定 round-trip 的复杂表格、复杂嵌套结构

### 2.5 三个内容入口的统一规则

第一版需要把以下三个入口收敛到同一套解析与降级链路：

- `edit` 中键入 Markdown 触发语法
- 粘贴 Markdown 文本到 `edit`
- 从 `split` 切回 `edit`

统一规则是：

- 先走同一套 Markdown 解析与 editor document 构建流程
- 再对 unsupported content 做同一套源码片段标记
- 最后把可稳定结构化的块交给可视编辑器渲染

这意味着第一版不能为“键入”“粘贴”“split -> edit”分别维护三套分叉规则，否则 unsupported fragment 的边界会漂移，round-trip 也难以验证。

## 三、状态模型与数据流

### 3.1 文档状态拆分

建议在 `App` 层把当前文档状态显式拆成以下字段：

- `savedDocumentContent: string | null`
- `draftDocumentContent: string | null`
- `documentMtimeMs: number | null`
- `isDirty: boolean`
- `isSaving: boolean`
- `saveErrorMessage: string | null`
- `pendingAutosaveReason: 'debounce' | 'flush' | null`
- `activeSaveToken: string | null`
- `activeSaveTarget: { projectId: string; documentPath: string; expectedMtimeMs: number | null } | null`

其职责边界是：

- `savedDocumentContent`：最近一次成功读取或成功写回后的磁盘版本。
- `draftDocumentContent`：当前编辑中的内存版本。
- `isDirty`：`draftDocumentContent !== savedDocumentContent`。
- `isSaving`：当前是否有保存请求飞行中。
- `saveErrorMessage`：最近一次保存失败信息。
- `documentMtimeMs`：保存时用于 optimistic concurrency 的 `expectedMtimeMs`。
- `activeSaveToken`：当前保存请求的唯一标识，用于丢弃过期回包。
- `activeSaveTarget`：当前保存请求绑定的项目、文档和 `mtime` 快照，用于确保保存结果只回写到发起它的文档会话。

### 3.2 `edit` 的内存真相层定义

第一版必须明确：**跨模式共享的唯一工作真相是 Markdown 字符串 draft，而不是编辑器内部 state。**

具体约束如下：

- `draftDocumentContent` 是 `edit` 与 `split` 共用的唯一跨模式真相。
- 可视编辑器内部的 editor state 只是一层运行时投影，不是长期共享状态。
- `edit` 中每次稳定 transaction 后，都要把当前 editor document 序列化回 `draftDocumentContent`。
- 标题导航、自动保存、unsupported fragment 检测、`split` 左侧源码栏和右侧预览的内容，都统一基于 `draftDocumentContent`。
- `split -> edit` 时，再从 `draftDocumentContent` 重新解析 editor document。

这样做的原因是：

- `split` 左侧天然以 Markdown 文本为核心，跨模式共享时不需要双向维护两套长期真相。
- 当前标题导航与自动保存链路都更容易围绕 Markdown 字符串建立，而不是强耦合编辑器内部数据结构。
- unsupported fragment 的保真规则也更容易落在“字符串级 round-trip”上验证。

### 3.3 视图与状态关系

- `read`：读取 `savedDocumentContent`
- `edit`：读取并修改 `draftDocumentContent`
- `split`：左侧源码编辑与右侧预览都读取并修改 `draftDocumentContent`
- 标题导航：
  - `read` 依赖 `savedDocumentContent`
  - `edit` / `split` 依赖 `draftDocumentContent`

### 3.4 三模式读取与切换矩阵

| 模式 | 读取内容 | 是否可编辑 | 离开前是否需要 flush | 是否显示预览 |
| --- | --- | --- | --- | --- |
| `read` | `savedDocumentContent` | 否 | 不适用 | 是 |
| `edit` | `draftDocumentContent` | 是，可视编辑 | 是，切到 `read` / 切文档 / 切项目 / 重启服务 / 停服务前都要 flush | 否，直接在可视编辑面中呈现 |
| `split` | 左右两栏都读 `draftDocumentContent` | 是，左侧源码栏可编辑 | 是，切到 `read` / 切文档 / 切项目 / 重启服务 / 停服务前都要 flush | 是，右侧预览常驻 |

### 3.5 自动保存状态机

状态流转建议如下：

1. 文档打开：`saved = draft = loadedContent`
2. 用户编辑：`draft` 变化，`isDirty = true`
3. debounce 到期：触发自动保存
4. 保存中：`isSaving = true`
5. 保存成功：
   - `saved = serverReturnedContent`
   - `draft = serverReturnedContent`
   - `documentMtimeMs = serverReturnedMtimeMs`
   - `isDirty = false`
   - `saveErrorMessage = null`
6. 保存失败：
   - `isSaving = false`
   - `isDirty = true`
   - `saveErrorMessage = error.message`
   - 保留当前 `draftDocumentContent`
   - 不清除后续自动保存能力；用户继续输入时仍可触发新的自动保存
   - 顶栏提供显式重试入口
7. 切换动作触发 `flush`：
   - 若 `isDirty = false`，直接切换
   - 若 `isSaving = true`，等待当前保存完成
   - 若 `isDirty = true`，立即执行一次非 debounce 保存
   - 若 `flush` 失败，阻断切换

### 3.6 保存请求归属与过期结果丢弃

自动保存必须把“保存结果属于哪个文档会话”当成硬约束，而不是隐含假设。

每次保存请求都必须绑定：

- `projectId`
- `documentPath`
- `expectedMtimeMs`
- `saveToken`

保存结果回到前端时，必须按以下顺序校验：

1. `saveToken` 是否仍等于当前活动保存请求
2. `projectId` 是否仍等于当前活动项目
3. `documentPath` 是否仍等于当前打开文档
4. 返回结果是否仍对应同一轮 `expectedMtimeMs` 语义

只要其中任一条件不成立，就把这次返回视为**过期结果**，不允许覆盖当前文档状态槽。

### 3.7 保存失败后的恢复语义

保存失败后的默认语义必须明确：

- 保留当前 draft，不回退用户输入
- 保持 `isDirty = true`
- 顶栏状态显示 `保存失败`
- 用户继续输入时，仍允许新的 debounce 自动保存继续尝试
- 顶栏额外提供显式 `重试保存` 入口
- 所有需要离开当前文档上下文的动作在失败未恢复前继续被阻断

### 3.8 预览与标题导航刷新节奏

第一版需要把 `split` 右侧预览和标题导航的刷新节奏写死，避免后续实现时一边走“即时刷新”、一边走“保存后刷新”：

- `draftDocumentContent` 每次发生变更后，`split` 右侧预览与标题导航都应立即基于最新 draft 重新计算。
- 自动保存只负责落盘与状态切换，不负责驱动预览或目录刷新。
- 如果后续发现大文档即时刷新性能不足，优化方式应是对渲染链做节流或分层缓存，而不是把刷新时机退回到“保存成功后”。

### 3.9 已知限制

本轮不做本地 draft 崩溃恢复，因此已知存在一个可接受边界：

- 如果浏览器窗口或线程在 debounce 窗口内被意外关闭，最近一小段尚未 flush 的输入可能丢失

这个边界不应被误判为本轮 bug；后续若要支持崩溃恢复，应单独开 `draft-recovery` 任务。

## 七、风险与控制

### 7.1 编辑器接入风险

风险：

- `Milkdown` 接入后可能打破当前标题导航和样式链路。

控制：

- 先把三模式与自动保存骨架独立完成，再接编辑器。
- 标题导航继续依赖 Markdown 文本提取，不依赖编辑器内部 DOM 反扫。

### 7.2 Round-trip 风险

风险：

- 可视编辑器对复杂 Markdown 的解析与序列化可能漂移。

控制：

- 第一版只稳定支持中档范围。
- unsupported content 局部降级为源码片段，不强行结构化。
- 每新增一种结构，都补最小 round-trip 测试。

### 7.3 自动保存竞争风险

风险：

- 连续输入、切文档、切项目和服务重启/停机动作可能与保存请求并发冲突。

控制：

- App 层只保留一个有效保存请求。
- 所有导航动作与中断 bridge 的动作统一走 `flushPendingSave()`。
- 保存结果必须校验 `saveToken + projectId + documentPath + expectedMtimeMs`，过期结果直接丢弃。
- 保存失败阻断切换，不做乐观放行。

### 7.4 范围膨胀风险

风险：

- 在第一版可视编辑器里顺手把表格、callout、frontmatter 都做进去。

控制：

- 第一版严格卡在中档语法范围。
- 表格、callout、frontmatter 可视编辑单独后续开任务，不混入本轮。

## 十、建议的执行顺序

建议按以下顺序落地，不要跳步：

1. 先固定 `read / edit / split` 的正式模式职责与测试修正。
2. 再把 `split` 模式做成稳定分栏模式，并固定 `draftDocumentContent` 为跨模式唯一真相。
3. 再做自动保存状态机、保存 token 和 flush 闸门。
4. 再接第一版 `edit` 可视编辑器。
5. 最后补 unsupported fragment 局部降级与 round-trip 验证。

这样做的原因是：

- 先稳住模式和保存链路，才能让后续编辑器接入问题边界清晰。
- `split` 先成型后，`edit` 接入失败时仍有稳定兜底，也能先固定 Markdown draft 作为跨模式真相。
- 局部源码片段降级放在后段做，更容易围绕真实编辑器行为来定界。
