# 方案概述

## 1. 总体目标和范围

本方案的目标是为 `md-reader` 制定一条“阅读态与编辑态尽量共享同一渲染内核”的收敛路线，降低当前 `react-markdown` 与 `Milkdown/Crepe` 双链路长期并存带来的排版漂移、列表语义分叉和细节回归成本。

本方案覆盖：

- 是否应让 `read` 与 `edit` 使用同一种渲染方式
- 推荐的统一架构
- 对 `split`、标题导航、自动保存、unsupported content 的影响
- 分阶段迁移路径
- 风险、非目标和验收标准

本方案不覆盖：

- 直接实施代码改动
- 重新选择编辑器框架
- 扩展新的 Markdown 功能范围
- 改写 Markdown 真相层或 sidecar 边界

## 2. 各阶段任务概要

### 阶段一：只读 Milkdown 技术探针

- 主要工作：先做一个不替换默认阅读态的只读 `Milkdown` 原型，验证当前项目是否真的具备“共享单内核”的实现基础。
- 预期成果：确认只读态能否稳定覆盖首批节点、能否注入统一 heading id、能否支持目录点击与滚动高亮。
- 执行顺序：必须先于 shared core 抽象和正式替换。

### 阶段二：抽 Shared Document Core

- 主要工作：在技术探针通过后，再抽出共享 schema、parser、heading id、unsupported content 边界和样式 token。
- 预期成果：把“共享同一内核”从口头目标变成真实的模块边界，而不是继续把逻辑散在 `WorkspaceLayout` 和 `Crepe` 初始化里。
- 执行顺序：探针通过后再进入抽核。

### 阶段三：阅读态只读内核接入

- 主要工作：引入只读版 `Milkdown` 文档渲染器，让 `read` 不再走 `react-markdown`，而是复用与 `edit` 同源的 schema、NodeView 和节点渲染规则。
- 预期成果：`read` 与 `edit` 的标题、段落、列表、引用、代码块、链接等首批节点进入同一语义系统。
- 执行顺序：先完成只读渲染器，再切换阅读态。

### 阶段四：模式与工作区链路统一

- 主要工作：调整 `read/edit/split` 的渲染分工、标题导航来源、滚动锚点与文档容器边界。
- 预期成果：工作区仍保留现有模式语义，但渲染差异不再由双链路造成。
- 执行顺序：阅读态切换稳定后，再统一模式间协作。

### 阶段五：兼容与收尾

- 主要工作：处理 unsupported content、懒加载策略、性能回归、样式 token 收敛与测试补齐。
- 预期成果：单内核方案可持续演进，而不是用一次性 CSS 补丁维持表面一致。
- 执行顺序：主链路跑通后再做收尾。

## 3. 整体结构框架

目标结构应收敛为四层：

1. Markdown 真相层  
   继续以原生 Markdown 正文为唯一内容真相，不引入专有 block JSON。

2. Shared Document Core  
   由 `Milkdown` schema、parser、serializer、NodeView 组成共享文档内核，服务 `read`、`edit`、后续 `split` 预览。

3. Workspace Mode Layer  
   `read`、`edit`、`split` 只负责交互权限和工作区布局，不再各自绑定不同的正文渲染技术栈。

4. UI / Chrome Layer  
   标题导航、工具栏、自动保存状态、文件树、滚动容器等继续停留在工作区壳层，不进入文档真相层。

# 1. 当前问题的真实边界

当前仓库的正文渲染是双链路：

- 阅读态：`src/app/WorkspaceLayout.tsx` 中的 `react-markdown + remark-gfm`
- 编辑态：`src/editor/visual-markdown-editor-impl.tsx` 中的 `Milkdown + Crepe + ProseMirror NodeView`

这条分叉已经带来了结构性后果：

1. 同样的 Markdown，在 `read` 和 `edit` 下不是同一套 DOM 结构。  
   阅读态输出标准 `ul/li/p`；编辑态输出 `milkdown-list-item-block -> li.list-item -> children/content-dom` 一类 NodeView 包装层。

2. 所有“视觉同构”工作都在做双系统对齐。  
   列表、编号、中文换行、段落后接嵌套列表、代码块外框、引用块内边距，都会反复落到“当前 CSS 是否成功把两套结构抹平”。

3. 当前问题不再是单个 CSS bug。  
   最新 ADR 已确认，列表差异属于 `Crepe/Milkdown` 列表 NodeView 与当前项目 CSS 扁平化收敛策略之间的集成层问题，而不是纯 `margin` 漏调。

4. 当前编辑链本身已经暴露出“不是全部 Markdown 都可直接交给可视内核”的事实。  
   `src/editor/visual-markdown-editor.tsx` 现在会先剥离前导 HTML comments，再把可编辑部分交给 `Milkdown/Crepe`，最后再写回原文本。这说明现有内核接入已经存在内容边界，不能把“编辑态可跑”误认为“阅读态直接切过去就会全量稳定”。

因此，如果目标是“以后尽量不再重复出现这类 read/edit 漂移”，继续维持双渲染链不是稳态。

# 2. 是否应该让两个模式使用同一种渲染方式

结论：**应该，但需要定义成“同一渲染内核”，而不是字面意义上的“同一份 DOM”。**

更准确的目标不是：

- `read` 和 `edit` 完全渲染出相同 HTML

而是：

- `read` 和 `edit` 共享同一套文档模型、解析规则、节点语义、NodeView / schema 边界

原因是：

1. 编辑器模式必然需要额外的 selection、cursor、editable node、composition、NodeView 包装层。  
   所以可编辑态和只读态很难做到完全同一份 DOM。

2. 但它们可以共享同一个内核。  
   只要阅读态不再走 `react-markdown`，而改走只读版 `Milkdown` 文档渲染器，那么标题、列表、引用、代码块、任务列表等就会进入同一套语义系统。

3. 这样问题会从“双系统对齐”变成“同系统的只读态和编辑态差异收敛”。  
   这是成本和收益最合理的做法。

所以，这里的“同一种渲染方式”推荐解释为：

- `read`：只读 `Milkdown` renderer
- `edit`：可编辑 `Milkdown` editor
- `split`：源码编辑 + 右侧只读 `Milkdown` preview

# 3. 三条可选路线

## 路线 A：继续双链路，只做样式收敛

### 做法

- 保留阅读态 `react-markdown`
- 保留编辑态 `Milkdown/Crepe`
- 继续通过共享 token、编辑态专用覆盖、局部 NodeView CSS 修补来收敛视觉

### 优点

- 当前改动最小
- 阅读态性能最好
- 不触碰现有阅读链路和锚点注入主逻辑

### 缺点

- 本质问题不变
- 每个复杂 Markdown 元素都可能继续反复漂移
- 列表、任务列表、表格、HTML comment、future callout 会持续消耗维护成本

### 判断

不推荐作为长期路线，只适合短期止血。

## 路线 B：阅读态切换为只读 Milkdown，形成单内核

### 做法

- 保留 `Milkdown` 作为唯一文档内核
- 阅读态新增只读 `ReadonlyDocumentRenderer`
- 编辑态继续用 `VisualMarkdownEditor`
- `split` 的右侧预览也统一改为只读 `ReadonlyDocumentRenderer`

### 优点

- `read/edit/split-preview` 共享同一套 schema / NodeView / 列表语义
- 标题、列表、引用、代码块更容易保持一致
- 后续再扩表格、callout、折叠块时，不需要同时维护两套正文渲染规则

### 缺点

- 阅读态复杂度提高
- 首屏包体与只读性能压力会增加
- 当前基于 `react-markdown` 的标题锚点、样式选择器、滚动定位逻辑需要一起迁移
- 当前仓库里还没有现成可复用的 shared core，实施前必须先做技术探针确认只读态是否能稳定承接首批节点与目录链路

### 判断

这是**推荐路线**，但前提是先通过技术探针。  
它没有推翻现有编辑器路线，只是把阅读态也拉进同一内核，收益和风险最平衡。

## 路线 C：下沉到底层自定义 Milkdown / ProseMirror 渲染装配

### 做法

- 弱化或放弃 `Crepe` 默认壳层
- 自己接管文档节点、只读态与编辑态的渲染装配
- 为关键 block 单独建立统一 renderer / editor view

### 优点

- 一致性最高
- 可控性最强
- 长期最适合做深度 Notion-like 体验

### 缺点

- 成本最高
- 会显著扩大当前实现面
- 不适合在现阶段直接作为首选路线

### 判断

可作为路线 B 的后备方案，不建议现在直接进入。

---

# 4. 推荐方案

推荐采用 **路线 B：阅读态切换为只读 Milkdown，形成单内核渲染**。

## 4.1 推荐理由

1. 它直接解决当前最核心的问题。  
   现在的维护成本来自双渲染链，不是单个节点的视觉细节。路线 B 能从源头减少分叉。

2. 它不违反现有长期决策。  
   当前 truth 已经确认编辑器核心优先 `Milkdown`；切换阅读态到只读 `Milkdown`，是在强化这条路线，而不是推翻它。

3. 它保留渐进迁移空间。  
   先把 `read` 拉到 `Milkdown`，仍然可以继续保留 `Crepe` 作为第一版编辑壳层；如果后续发现 `Crepe` 可控性不足，再下沉到底层自定义装配。

4. 它最符合你的目标优先级。  
   你的核心诉求不是“理论上最轻”，而是“阅读和编辑尽量像一个系统，不要不断冒出小差异”。

## 4.2 推荐后的模式定义

- `read`
  - 使用只读 `Milkdown` renderer
  - 不显示编辑 chrome
  - 不允许光标编辑

- `edit`
  - 使用可编辑 `Milkdown/Crepe`
  - 保留当前 autosave、IME、mode switch 语义

- `split`
  - 左侧继续承担源码精确修订
  - 右侧预览从 `react-markdown` 改为只读 `Milkdown`
  - 因此 `split` 也进入同一渲染内核体系

# 5. 迁移后的目标架构

建议新增一层共享文档核心，而不是继续让 `WorkspaceLayout` 直接分别调用 `react-markdown` 和 `VisualMarkdownEditor`。

## 5.1 建议的模块分层

- `src/document-core/`
  - 共享 schema、parser、serializer、heading extraction、只读/编辑态公共配置

- `src/document-renderer/`
  - `ReadonlyDocumentRenderer`
  - 面向 `read` 与 `split` 右侧预览

- `src/editor/`
  - `VisualMarkdownEditor`
  - 面向 `edit`

- `src/app/WorkspaceLayout.tsx`
  - 只负责任务模式切换、工作区容器和右侧标题导航
  - 不再承担两套正文技术栈的直接拼装

## 5.2 共享内容

以下能力应收敛为 shared core：

- Markdown -> document model
- 标题提取与 heading id 生成
- 首批支持节点：正文、标题、列表、引用、代码块、链接
- unsupported content 降级策略
- 文档样式 token

这里要明确两条边界：

1. 第一轮 shared core 只保证首批支持节点，不把表格、callout、HTML block、frontmatter 一起混入切换范围。  
2. 当前已知的前导 HTML comments 保留策略必须继续存在，不能因为引入只读态就丢失 Markdown 原文保真。

# 6. 关键迁移步骤

## 第一步：先做只读 `Milkdown` 技术探针

这一步不是正式切换，而是做实现前探针，避免高估当前方案可行性。

探针必须验证 6 类节点：

- 标题
- 段落
- 列表
- 引用
- 代码块
- 链接

探针必须验证 3 个行为：

- 统一 heading id 注入
- 目录点击跳转
- 滚动高亮同步

探针通过标准建议写死为：

- 同一份测试文档在只读态下能稳定渲染以上 6 类节点
- 目录点击能跳到对应 heading，且 active heading 能随滚动稳定更新
- 不出现额外编辑 chrome、选择器工具条或可编辑光标
- 前导 HTML comments 不丢失，且不会误进入可视正文区

只有这一步通过，才进入 shared core 抽象。

## 第二步：抽出 Shared Document Core

先把当前分散在多个地方的正文语义收拢：

- `heading-outline.ts`
- 编辑态 NodeView 依赖的 schema / list semantics
- unsupported content 的边界定义
- 文档级排版 token

这里的重点不是抽一个抽象目录，而是把当前几处隐式耦合显式化：

- `read` 依赖 `remarkHeadingIds()` 注入 `data-heading-id`
- `edit` 当前通过 `querySelectorAll('h1...h6')` 再按 index 绑定 `documentHeadings`
- 前导 HTML comments 通过 `splitLeadingHtmlComments()` 在编辑入口单独处理

如果这三条链不先收敛，所谓 shared core 只是换文件夹，不是真正统一。

## 第三步：做只读 `ReadonlyDocumentRenderer`

新增只读渲染器：

- 输入仍然是 Markdown 文本
- 内部转成与编辑态一致的 Milkdown 文档模型
- 输出只读 `Milkdown` DOM
- 明确关闭所有编辑相关 feature、selection chrome 和 interactive UI

这一步完成后，仓库里会同时存在：

- 旧阅读态 `react-markdown`
- 新阅读态 `ReadonlyDocumentRenderer`

先并行，不立即替换。

## 第四步：统一标题导航和锚点策略

这是切阅读态之前的前置条件，不是切换后的补充优化。

当前 `read` 和 `edit` 的目录链路并不一致：

- `read` 依赖 `data-heading-id`
- `edit` 依赖 DOM 顺序和 Markdown heading 列表按 index 对齐

切换前必须统一为：

- 标题数据继续来自 Markdown 源文本
- 只读 renderer 和编辑态 renderer 都显式注入同一套 heading id
- `getHeadingTargets()` 不再分裂成“read 一种查法、edit 一种查法”

否则单内核切换后，目录点击和滚动激活会成为第一批回归点。

## 第五步：让阅读态切到新只读渲染器

在 `WorkspaceLayout` 中把：

- `read` 正文渲染入口  
  从 `react-markdown` 改成 `ReadonlyDocumentRenderer`

这一步是真正的单内核切换点。

切换后，阅读态不再依赖 `.markdown-document` 输出标准 HTML 结构，而是依赖 Milkdown 只读 DOM。

## 第六步：统一样式层，不再区分 `.markdown-document`

当前样式很多选择器长这样：

- `.markdown-document ...`
- `.visual-markdown-editor .editor ...`

切到单内核后，应该收敛为：

- shared document class
- readonly document class
- editable document class

重点是让样式语义从“阅读态 vs 编辑态”切到“只读态 vs 可编辑态的 chrome 差异”。

## 第七步：重定义 `split`

当前 truth 已明确 `split` 仍是正式模式，不能绕开。

因此建议：

- `split` 左侧继续保留源码精确修订
- `split` 右侧切到只读 `Milkdown` renderer

这样 `split` 仍然保留价值，但右侧预览不再成为第三套表现。

需要额外写死一个边界：

- `split` 左侧源码区继续承担原文保真与精确修订责任
- `split` 右侧只读预览追求与 `read` 同内核
- 不把 `split` 扩成“左右都要 Notion-like”

## 第八步：只在必要时下沉到更底层自定义装配

如果在路线 B 完成后，仍然发现：

- `Crepe` 只读态无法充分收敛
- NodeView chrome 很难彻底控制
- 阅读态仍然被迫承受太多编辑器遗留结构

再进入路线 C：

- 下沉到底层 `Milkdown` 自定义装配

这一步应作为后备方案，而不是首轮就进入。

建议把触发条件写死为任一命中即切换：

1. 只读 `Milkdown` 无法稳定注入统一 heading id。  
2. 只读态无法彻底关闭多余 chrome，导致阅读态残留明显编辑器 UI。  
3. 列表、引用、代码块三类核心节点在只读态和编辑态仍持续出现结构性漂移。  
4. 为了兼容只读态被迫在 `Crepe` 周围堆积大量样式补丁，导致维护成本重新回到“双系统抹平”状态。

# 7. 对现有关键能力的影响

## 7.1 对自动保存的影响

影响较小。

原因是：

- autosave 的核心在 `App.tsx`
- 它依赖的是 draft / saved content / revision / flush
- 不依赖阅读态使用 `react-markdown`

真正需要注意的是：

- `read` 切换后显示的是只读 `Milkdown` renderer
- 但其数据源仍应保持“已保存内容”语义，不要把 read 的展示源混成 draft

## 7.2 对标题导航的影响

影响较大，而且是前置约束。

原因是：

- 当前导航已经与阅读态 DOM 有耦合
- 当前编辑态导航又与标题顺序 index 对齐有耦合
- 单内核前必须先统一 heading id 注入和目标查找策略

这不是迁移后的补充优化，而是切阅读态前必须打通的链路。

## 7.3 对 unsupported content 的影响

影响中等。

如果阅读态改走只读 `Milkdown`，就必须重新明确：

- 哪些内容在只读态可以稳定渲染
- 哪些内容在编辑态只做源码片段降级
- 哪些内容在 split 右侧只读预览中也必须保持保真

推荐边界不变：

- unsupported content 仍然不在 `edit` 中强行结构化
- 只是在 `read` 中也交给同一文档内核处理显示
- 未被稳定结构化的内容继续保留为 Markdown 原文或源码片段语义，不允许为了“看起来统一”而自动重写原文

当前已知必须单列处理的内容至少包括：

- 前导 HTML comments
- 后续可能扩展到 frontmatter / HTML block / 复杂嵌入块

## 7.4 对性能和包体的影响

这是路线 B 最大的现实代价。

阅读态改成 `Milkdown` 后会带来：

- 阅读态包体变大
- 首次打开文档时初始化成本更高
- 当前“只在 edit 时按需加载编辑器”的收益会被削弱

因此必须同时设计：

1. 只读 renderer 是否独立成单独 chunk
2. 是否保留懒加载
3. `read` 首屏是否允许 skeleton / loading shell

推荐：

- 只读 renderer 与可编辑 renderer 分成两个 chunk
- `read` 只加载只读 chunk
- `edit` 再加载可编辑 chunk

这样仍然是“单内核”，但不是“单包”。

建议在实施前把性能验收口径写死：

- `read` 首次打开单篇中大型文档的感知等待时间目标不超过 500ms 到 800ms
- 只读渲染 chunk 与可编辑 chunk 必须拆分，避免把整套编辑器能力打进默认 `read`
- 目录树、文件树与顶部状态条不因只读 renderer 初始化而阻塞交互

---

# 8. 主要风险

## 风险 1：阅读态包体与首屏速度退化

这是路线 B 的最大风险。

缓解方式：

- 只读态和可编辑态分 chunk
- 只在需要时加载各自能力
- 对大文档加 loading shell 和延迟激活策略

## 风险 2：只读态仍带出编辑器遗留 chrome

如果只读 renderer 仍然沿用过多 `Crepe` 壳层，视觉上可能继续残留编辑器味道。

缓解方式：

- 优先评估 `Crepe` 的只读可控边界
- 如不够，再切到底层自定义 Milkdown 装配

## 风险 3：标题导航和滚动锚点回归

当前标题导航已经是一条稳定链路，切换阅读 renderer 后容易失配。

缓解方式：

- 标题数据继续以 Markdown 文本为真相
- 只把 DOM 锚点注入方式统一，不改变导航数据来源

## 风险 4：`split` 被边缘化

如果只讨论 `read/edit`，很容易把 `split` 留成遗留分支。

缓解方式：

- 在方案里明确 `split` 右侧也必须进入同一只读 renderer
- 不允许继续维持第三套预览表现

## 风险 5：技术探针未过却直接进入正式切换

这是最容易低估的执行风险。

缓解方式：

- 把探针阶段设为硬前置
- 探针不过不进入 shared core 抽象和阅读态替换
- 必要时直接切到路线 C，而不是在 `Crepe` 外围继续堆补丁

---

# 9. 非目标

本方案明确不追求以下目标：

1. 不追求 `read` 与 `edit` 完全相同 DOM
2. 不追求一次性解决所有 Notion-like 能力
3. 不在本轮引入新 Markdown 扩展语法
4. 不为了一致性而破坏 Markdown-first 保存边界
5. 不在方案阶段直接移除 `split`

---

# 10. 推荐执行顺序

推荐的真实执行顺序如下：

1. 先做只读 `Milkdown` 技术探针
2. 探针通过后抽 Shared Document Core
3. 新增只读 `ReadonlyDocumentRenderer`
4. 在不替换默认阅读态的前提下并行验证只读 renderer
5. 统一 heading id / 标题导航锚点
6. 阅读态切换到只读 renderer
7. `split` 右侧预览切换到只读 renderer
8. 统一样式语义与 chunk 策略
9. 仅当 `Crepe` 可控性不足时，再下沉到底层 `Milkdown` 自定义装配

这条顺序的核心逻辑是：

- 先验证路线成立
- 再统一内核
- 再统一工作区模式
- 最后再决定要不要进一步放弃 `Crepe` 壳层

---

# 11. 最终建议

如果你的目标是：

- 阅读和编辑尽量像同一个系统
- 避免列表、编号、中文换行、任务列表这类问题反复出现
- 接受阅读态不再是最轻的纯静态渲染

那么应采用：

**“阅读态切换为只读 Milkdown，形成单内核渲染；保留 edit 为可编辑 Milkdown，split 右侧也进入同一只读内核”**

这是当前阶段最稳的长期路线。

它不是最便宜的路线，但它最能从结构上减少你现在这种“每修一个细节，另一个模式又漂”的问题。

---

# 12. 第一轮实施范围与验收

第一轮只收敛以下节点与能力：

- 正文
- 标题
- 列表
- 引用
- 代码块
- 链接

第一轮不混入：

- 表格
- callout
- frontmatter
- HTML block
- 复杂嵌入内容

第一轮关键验收项：

1. `read` 不再 import `react-markdown`。  
2. `split` 右侧预览不再 import `react-markdown`。  
3. 标题导航在 `read / edit / split-preview` 三种模式下点击跳转和滚动激活一致。  
4. 中文列表与英文列表在 `read / edit / split-preview` 下的行距、段后间距和换行表现一致。  
5. 前导 HTML comments 不丢失，保存后仍按原文回写。  
6. 只读 chunk 与可编辑 chunk 独立拆分，`read` 不加载整套编辑能力。  

---

# 13. 后续实施前的确认点

在真正开始实施前，建议先明确以下 4 个确认点：

1. 是否接受阅读态首屏成本上升，以换取长期一致性
2. 是否保留 `split` 正式模式不变，只替换其右侧预览内核
3. 是否接受 first step 先做“只读 Milkdown renderer 技术探针”，而不是直接替换现有阅读态
4. 若 `Crepe` 的只读可控性不足，是否接受第二阶段下沉到底层 `Milkdown` 自定义装配

这 4 点确认后，就可以进入实施计划，而不是继续做局部样式修补。
