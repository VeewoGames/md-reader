# 方案概述

## 1. 总体目标和范围

本计划不是重新讨论 `md-reader` 是否应该走单内核路线，而是基于当前已经完成的真实基线，继续推进第二阶段执行。当前真实基线是：

- `read` 与 `split` 右侧预览已经统一进入 `ReadonlyMarkdownRenderer`
- `ReadonlyMarkdownRenderer` 当前内部仍然是只读 `Milkdown / Crepe` 装配
- `edit` 继续使用 `VisualMarkdownEditor` 的 `Milkdown / Crepe` 编辑装配
- 阅读切编辑卡死的最小修复已经完成，编辑态已关闭 `ListItem`，并去掉了对编辑器根节点的持续 `MutationObserver`

本计划覆盖：

- 第二阶段的真实起点与过时描述清理
- 只读内核是否继续从 readonly `Crepe` 外壳下沉
- 首批 6 类节点的同构验收边界
- 编辑态是否继续保留 `Crepe` 壳层的决策门
- 第二阶段的执行顺序、验证方式和失败切换条件

本计划不覆盖：

- 立即实施完整底层重构
- 扩大到表格、callout、frontmatter、HTML block 等复杂内容
- 重做自动保存状态机
- 重做 `split` 左侧源码编辑区

## 2. 各阶段任务概要

### 阶段一：基线纠偏与执行入口统一

- 主要工作：以当前代码和已接受 ADR 为准，纠正仍残留在 truth/旧方案中的历史描述。
- 预期成果：后续实施不再建立在“`read` 仍走 `react-markdown`”这类过时前提上。
- 执行顺序：必须最先完成。

### 阶段二：只读内核下沉决策

- 主要工作：判断当前 readonly `Crepe` 是否还能继续承担 `read / split-preview`，还是应直接进入更底层的只读 `Milkdown` 装配。
- 预期成果：明确第二阶段到底是在现有只读壳层上继续收敛，还是切换到更低层实现。
- 执行顺序：在开始大面积样式和节点同构前完成。

### 阶段三：六类节点同构收敛

- 主要工作：围绕标题、正文、列表、引用、代码块、链接，定义阅读态与编辑态的一致性目标与验收方式。
- 预期成果：后续实现只围绕明确节点集推进，不再泛化扩范围。
- 执行顺序：在只读内核路线明确后展开。

### 阶段四：编辑态装配决策门

- 主要工作：判断 `edit` 是否继续保留 `Crepe` 外壳，还是因为 NodeView、chrome、性能或结构漂移继续下沉。
- 预期成果：给出可执行的保留条件、失败条件与切换条件。
- 执行顺序：与六类节点同构实施并行评估，但决策在正式深入改造前锁定。

### 阶段五：文档、truth 与 ADR 收口

- 主要工作：把第二阶段形成的稳定事实沉淀回 canonical 文档，清理旧结论。
- 预期成果：团队后续不再混用旧 truth 与新基线。
- 执行顺序：在本轮执行完成后进行。

## 3. 整体结构框架

第二阶段的目标结构应收敛为三层：

1. 文档真相层
   - 原生 Markdown 仍是唯一内容真相
   - 标题数据仍然来自 Markdown 文本提取

2. 单内核渲染层
   - `read`
   - `edit`
   - `split-preview`
   共享同一文档模型、解析语义和首批节点渲染边界

3. 工作区交互层
   - `read / edit / split` 模式切换
   - 标题导航
   - 自动保存状态
   - 文件树与服务控制

这里的“单内核”定义为共享同一文档模型与语义规则，不要求 `read` 与 `edit` 输出完全相同的 DOM。

# 1. 当前真实基线

第二阶段必须以当前真实代码基线为准：

1. `src/app/WorkspaceLayout.tsx` 中，`renderDocumentPreview()` 已统一返回 `ReadonlyMarkdownRenderer`。
2. `read` 与 `split` 右侧预览已经不再使用 `react-markdown` 作为正式正文链路。
3. `src/document-renderer/readonly-markdown-renderer-impl.tsx` 目前仍是只读 `Crepe` 装配，而不是更底层的自定义只读 `Milkdown` 装配。
4. `src/editor/visual-markdown-editor-impl.tsx` 目前仍是编辑态 `Crepe` 装配，且已显式关闭 `CrepeFeature.ListItem`。
5. `src/app/WorkspaceLayout.tsx` 中，`edit` 模式不再持续观察编辑器根 DOM，而是改为有限次 `requestAnimationFrame` 补标。

这意味着：

- “是否统一到同框架”这个问题已经不是当前矛盾；
- 当前真正未完成的是“是否进一步统一到底层装配层”和“首批节点同构是否形成可执行验收边界”。

# 2. 当前残留问题的准确归类

第二阶段要处理的问题，不应继续混成一个大而化之的“阅读和编辑不一致”。

更准确的归类是：

## 2.1 已经解决的部分

- `read / split-preview / edit` 已经进入同一 `Milkdown` 体系
- 阅读切编辑卡死的主路径已完成最小修复
- 编辑态列表间距的第一轮集成层修复已落地

## 2.2 尚未解决的部分

- 阅读态当前仍然套在 readonly `Crepe` 外壳上，不等于已经统一到最低层装配
- `edit` 仍然存在第三方编辑器壳层带来的结构、chrome 和 NodeView 边界
- canonical truth 中仍残留若干旧描述，可能误导后续实现

## 2.3 本轮要避免的误判

- 不能再把问题描述成“当前 `read` 主链仍是 `react-markdown`”
- 不能把“同一渲染内核”误解成“完全相同 DOM”
- 不能把每个样式差异都直接提升成“必须立刻放弃 `Crepe`”

---

# 3. 第二阶段的核心目标

第二阶段只聚焦 3 件事：

1. 决定只读阅读链是否继续沿 readonly `Crepe` 最小壳层推进，还是直接下沉到更底层的只读 `Milkdown` 装配。
2. 把首批 6 类节点的同构目标写成明确的执行与验收边界。
3. 明确编辑态继续保留 `Crepe` 的条件，以及触发进一步下沉的条件。

首批 6 类节点固定为：

- 标题
- 正文
- 列表
- 引用
- 代码块
- 链接

本轮明确不扩展到：

- 表格
- callout
- frontmatter
- HTML block
- 原始 HTML
- 复杂嵌套富交互块

# 4. 阶段一：基线纠偏与执行入口统一

## 4.1 必须纠正的过时描述

当前已有文档中仍存在以下过时或半过时说法：

1. `read` 走 `react-markdown + remark-gfm`
2. 阅读态列表链路仍以 `react-markdown` 为正式主链描述
3. 标题注入统一仍完全依赖持续 `MutationObserver`

这些说法现在都不能再作为执行前提。

## 4.2 本阶段输出

- 把本执行计划作为第二阶段的唯一执行入口
- 在 closeout 阶段统一回写 truth / ADR
- 后续实现与验证一律以当前代码入口和已接受 ADR 为准

## 4.3 完成条件

- 团队后续不再引用“`read` 仍是 `react-markdown` 正式链路”作为现状判断
- 第二阶段的后续任务可以直接引用本执行计划开展实施

---

# 5. 阶段二：只读内核下沉决策

第二阶段首先要回答的问题不是“要不要统一框架”，而是：

**当前只读阅读链是否继续保留 readonly `Crepe` 外壳，还是直接下沉到更底层的只读 `Milkdown` 装配。**

## 5.1 保留 readonly `Crepe` 的条件

只有同时满足以下条件，才继续沿当前只读壳层推进：

1. 阅读态 chrome 可以继续保持足够干净，不暴露明显编辑器 UI。
2. 标题、正文、列表、引用、代码块、链接 6 类节点在 `read / edit / split-preview` 下能继续稳定收敛。
3. 为了维持只读态，不需要继续堆积大量只针对 `Crepe` 外壳的样式补丁。
4. 只读态不会重新引入明显性能或挂载时序问题。

## 5.2 触发只读内核下沉的条件

任一条件命中，就应停止在 readonly `Crepe` 外壳上继续缝补，进入更底层的只读 `Milkdown` 装配：

1. 阅读态持续残留明显编辑器 chrome。
2. 列表、引用、代码块等核心节点在只读态和编辑态仍持续出现结构性漂移。
3. 标题目标链、heading id 注入或滚动高亮需要继续依赖外层大量补丁才能成立。
4. 只读态性能或切换稳定性再次成为结构性问题。

## 5.3 本阶段建议

默认先不直接宣布下沉，而是按“保留 readonly `Crepe`，但带明确退出条件”推进。  
原因是当前第一阶段已经证明只读链路可用，第二阶段更适合先用清晰决策门约束，而不是立即扩大实现面。

---

# 6. 阶段三：六类节点同构收敛

第二阶段的同构目标必须固定在 6 类节点，不继续泛化。

## 6.1 同构定义

这里的“同构”定义为：

- 同一份 Markdown 在 `read / edit / split-preview` 下属于同一语义边界
- 页面宽度、主要块节奏、节点间距和文本层级不出现明显模式跳变
- 标题导航、滚动定位和可视结构行为保持同一路径

这里不要求：

- DOM 完全一致
- 可编辑态与只读态毫无结构差异

## 6.2 每类节点的最低验收

### 标题

- 层级、字号、粗细、上下节奏基本一致
- 目录点击和滚动高亮使用同一目标链

### 正文

- 正文宽度、字号、行高、段距保持一致
- `read -> edit` 切换不出现明显顶部留白跳变

### 列表

- 中英文列表项的条目间距一致
- 列表项换行、嵌套列表和长句换行节奏一致

### 引用

- 引用块边距、内边距、前景/背景层级一致

### 代码块

- 代码块外框、背景、圆角、内边距和上下节奏一致

### 链接

- 链接颜色、下划线策略、hover 反馈和行内位置不出现模式级割裂

## 6.3 非目标边界

第二阶段不承诺以下内容已经完全同构：

- 表格
- task list
- frontmatter
- 原始 HTML
- callout
- 复杂嵌套结构

但这些内容必须满足一个底线：

- 不能在读取、切模式、分栏预览和保存链路中被静默改坏。

---

# 7. 阶段四：编辑态装配决策门

第二阶段不应默认把 `edit` 直接重写为更低层实现，但也不能无上限地继续修补 `Crepe`。

## 7.1 继续保留编辑态 `Crepe` 的条件

1. 默认 chrome 可以收敛到“先看到文档，而不是先看到第三方编辑器”。
2. 六类节点的主要视觉节奏可以稳定对齐。
3. 不再频繁出现由 NodeView 结构直接触发的新一轮布局回归。
4. 编辑态挂载性能在大文档场景下仍可接受。

## 7.2 触发编辑态下沉的条件

命中以下任一情况，即进入“自定义 `Milkdown` 装配”路线：

1. `Crepe` chrome 仍需大量 hack 才能隐藏。
2. 列表、引用、代码块等关键节点必须反复围绕 NodeView 结构打补丁。
3. 新一轮性能问题再次集中出现在 `Crepe` 壳层初始化。
4. 六类节点同构始终无法稳定闭合。

## 7.3 决策原则

这不是审美选择，而是维护成本选择。

如果后续 80% 的修复都还在围绕 `Crepe` 壳层做结构和样式补丁，那就说明第二阶段已经触发下沉条件，不应继续拖延。

---

# 8. 实施顺序

第二阶段建议按以下顺序推进：

1. 先以本计划为执行入口，停止沿用过时基线描述。
2. 逐项确认 readonly `Crepe` 是否继续成立，先不贸然重写只读链。
3. 以 6 类节点为边界继续做 read / edit / split-preview 的同构收敛。
4. 在实施过程中同步评估 `Crepe` 壳层是否已触发下沉条件。
5. 阶段完成后统一把稳定事实回写到 truth / ADR，清理旧描述。

这个顺序的核心是：

- 先把“判断基线”纠正；
- 再做“继续沿当前装配还是下沉”的决策；
- 最后才决定是否进入更大规模的结构替换。

---

# 9. 验证标准

第二阶段至少应满足以下验证标准：

1. 后续文档与实施不再把 `react-markdown` 当作当前正式阅读链路。
2. 六类节点的同构目标、非目标和失败切换条件写清楚并可复核。
3. 对 readonly `Crepe` 和编辑态 `Crepe` 都存在明确的保留条件与退出条件。
4. 本轮输出能直接指导下一轮实施，而不是停留在概念性建议。

如果后续进入代码实施，还应补充以下实测口径：

1. 大文档 `read -> edit` 切换不再出现冻结。
2. 编辑态标题导航链路仍稳定可用。
3. 列表、引用、代码块不会再次出现模式级大偏差。

---

# 10. 本轮结论

第二阶段已经不再是“是否统一框架”的讨论阶段，而是“如何把当前已统一到 `Milkdown` 体系的三条链路，继续收敛到底层装配与首批节点验收边界”的执行阶段。

因此，本轮推荐结论是：

1. 继续以当前 `read / edit / split-preview` 已共享 `Milkdown` 体系为前提推进。
2. 先不直接宣布放弃 readonly `Crepe` 或编辑态 `Crepe`，但必须带着明确下沉条件继续实施。
3. 第二阶段只围绕标题、正文、列表、引用、代码块、链接 6 类节点推进，不再扩范围。
4. 在本轮 closeout 时统一清理 canonical truth 中仍残留的旧描述。
