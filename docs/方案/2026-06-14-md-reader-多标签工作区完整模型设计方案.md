# md-reader 多标签工作区完整模型设计方案

## 方案概述

### 1. 总体目标和范围

本方案用于确认 `md-reader` 是否从当前单文档工作区升级为“多标签 + 活动标签 + 每标签独立文档槽”的完整工作区模型，并给出可直接进入实施计划的状态边界、交互规则和验证口径。目标不是只在现有单文档状态机外层加一排标签 UI，而是把工作区真相源从“当前文档”提升为“当前项目下的一次会话”。

本轮范围包括：

- 多标签集合与活动标签
- 每标签独立的正文、draft、dirty/save/error、滚动与标题导航状态
- 顶部中央标签栏替换当前状态信息区
- 文件树与活动标签联动
- 关闭标签、切项目、重启服务、刷新恢复等状态流转
- 本地恢复模型与测试口径调整

本轮不包括：

- 标签拖拽排序
- 固定标签
- 标签分组
- 多窗口
- 同一路径多副本标签
- 多编辑器同时挂载渲染
- 独立的顶部项目名/当前文档/保存状态三行信息块

### 2. 各阶段任务概要

1. 工作区状态层重构  
   主要工作：把 `App.tsx` 从单文档状态源改成 `WorkspaceSession + WorkspaceTab` 会话状态源。  
   预期成果：`currentDocument*` 不再是唯一真相源，dirty/save/scroll/headings 都归到标签槽。

2. 工作区壳层与文件树联动改造  
   主要工作：让 `AppShell`、`WorkspaceLayout`、`TopBar` 全部消费 `tabs + activeTab`，并把顶部中央区域改造成标签栏、右侧收敛成固定图标区。  
   预期成果：文件树点击已打开文档时切到已有标签，活动标签变化驱动顶部栏与正文区更新，原三行状态块被移除。

3. 保存、关闭、恢复链路重写  
   主要工作：重写切标签、关标签、切项目、刷新恢复时的 dirty/save 规则。  
   预期成果：支持多个 dirty 标签共存，关闭 dirty 标签只拦截当前标签，刷新后恢复标签集合与活动标签。

4. 验证与回归  
   主要工作：补组件测试、状态测试和关键手工回归。  
   预期成果：多标签状态不串槽，文件树折叠逻辑不再被“当前文档强制展开”锁死。

### 3. 整体结构框架

- `src/App.tsx`
  - 工作区会话真相源
  - 打开/切换/关闭/保存/恢复标签动作入口
- `src/app/AppShell.tsx`
  - 将 `session` 与 `activeTab` 透传给顶栏和工作区
- `src/app/WorkspaceLayout.tsx`
  - 标签栏、文件树、正文区、标题导航全部围绕活动标签工作
- `src/app/TopBar.tsx`
  - 中央区域改为展示活动标签栏，右侧改为固定图标操作区，不再展示独立项目名与状态块
- `src/workspace/local-state.ts`
  - 从恢复当前文档升级为恢复标签集合与活动标签

## 现状与问题

当前实现仍是典型单文档状态机。`App.tsx` 只有一组 `currentDocumentPath/currentDocumentContent/editingDocumentContent/currentDocumentMtimeMs/saveState`，工作区所有链路都默认“当前只有一个文档正在工作”。这在以下几个点上已经成为结构性限制：

1. 无法同时打开多个文档并各自保留 draft/dirty 状态。
2. 文件树展开逻辑绑定到单一 `currentDocumentPath`，当前文档祖先目录被强制展开，用户无法折叠当前活动分支。
3. 切文档、保存、恢复和错误提示都默认只归属于一个全局文档槽，多标签下会直接串状态。
4. 本地状态只恢复 `activeDocumentPath`，无法恢复多标签工作上下文。

因此，本轮不应再停留在“局部补一个标签条”的思路，而应直接把工作区状态模型升级为多标签会话。

## 决策结论

本方案采用“完整工作区模型”路线，而不是轻量标签壳层或伪多标签。

选择理由：

1. 用户已确认允许多个标签同时 dirty，这要求 dirty/save 状态归属于标签而不是全局文档槽。
2. 用户已确认关闭 dirty 标签时只拦截当前标签，这要求关闭动作能够精确落到单个标签对象。
3. 用户已确认刷新后要恢复上次打开的标签集合并回到上次活动标签，这要求本地状态保存的是会话而不是单文档路径。
4. 用户已确认文件树点击已打开文档时只切到已有标签，不允许同一路径重复开副本，这要求会话层以 `documentPath` 作为唯一标签键。

## 用户已确认的行为约束

1. 允许多个标签同时处于未保存修改状态。
2. 关闭 dirty 标签时，只拦截被关闭的那个标签，不全局拦截所有 dirty 标签。
3. 在文件树中点击已打开文档时，只切换到已有标签，不重复打开同一路径。
4. 刷新页面或重新打开项目后，恢复上次打开的标签集合，并回到上次活动标签。
5. 文件树联动只跟活动标签走，不跟所有已打开标签集合一起做高亮/展开。
6. 顶部中央空白区域改为标签栏，不再单独显示项目名。
7. 顶部原有 `状态：... / 保存：...` 信息块删除。
8. 标签视觉参考 Chrome，右侧操作区收敛为固定图标带。
6. 顶部中央空白区域改为标签栏，不再单独显示项目名。
7. 顶部原有 `状态：... / 保存：...` 三行信息块删除。
8. 标签视觉方向参考 Chrome，而不是厚重 IDE 式矩形页签。

## 状态模型设计

### 会话模型

建议将当前工作区真相源收敛为：

```ts
type WorkspaceSession = {
  tabs: WorkspaceTab[]
  activeTabId: string | null
  mode: WorkspaceMode
  regularViewState: 'locked' | 'unlocking' | 'editable' | 'locking'
}

type WorkspaceTab = {
  id: string
  documentPath: string
  persistedContent: string | null
  draftContent: string | null
  mtimeMs: number | null
  saveState: 'clean' | 'dirty' | 'saving' | 'error'
  saveErrorMessage: string | null
  lastKnownScrollTop: number
  headings: MarkdownHeading[]
}
```

### 关键原则

1. `documentPath` 是标签唯一键，同一路径只保留一个标签实例。
2. `regularViewState` 继续属于 `WorkspaceSession`，不跟随单篇文档切换；这与当前已确认的“解锁是全局工作区状态”保持一致。
3. `draftContent/saveState/saveErrorMessage/mtimeMs/lastKnownScrollTop/headings` 都归标签，不再挂在全局 `App` 上。
4. 活动标签只是“当前正在看的槽位”，不是全局唯一文档真相源。
5. 编辑器实例当前仍可只渲染活动标签一个；但数据层必须先是完整标签会话模型。

## 交互与状态流转

### 1. 打开文档

- 文件树点击文档时执行 `openOrActivateTab(path)`。
- 若该路径已存在于 `tabs` 中，则只切换活动标签。
- 若不存在，则创建新标签、拉取正文、插入 `tabs`，并设为活动标签。
- 新标签默认继承当前会话的 `mode + regularViewState`；不为单个标签保存独立解锁态。

### 2. 切换标签

- 切换标签不因为其他 dirty 标签而全局阻断。
- 切走前，必须先把当前编辑器里的内容、滚动和标题同步回原活动标签槽。
- 切入后，正文区、标题导航、顶部栏、文件树高亮全部基于新活动标签重算。
- `mode` 与 `regularViewState` 都是会话级状态：切换标签时继续沿用当前工作区显示模式，而不是按目标标签恢复另一套模式。

### 3. 关闭标签

- 关闭 clean 标签：直接关闭。
- 关闭 dirty 标签：只对当前被关闭标签弹 `保存 / 放弃 / 取消`。
- `保存`：保存成功后关闭；失败则保留该标签。
- `放弃`：丢弃该标签 draft 后关闭。
- `取消`：保持当前状态不变。
- 若关闭的是活动标签，优先激活右侧标签；若没有右侧则激活左侧；若两侧都没有则进入空态。

### 4. 切项目 / 重启服务 / 关闭服务

- 这类动作属于“会话级离开动作”。
- 若当前项目下任一标签 dirty，则进入会话级确认流。
- 确认内容必须列出 dirty 标签集合，而不是继续沿用“当前文档未保存”的单文档提示。
- 本轮推荐统一采用 `保存全部 / 放弃全部 / 取消` 三选一：
  - `保存全部`：串行 flush 全部 dirty 标签，全部成功后继续原动作；任一失败则停留当前项目并暴露失败标签。
  - `放弃全部`：丢弃全部 dirty 标签 draft 后继续原动作。
  - `取消`：中断原动作，保持当前工作区不变。
- 不采用“逐标签串行询问”，避免在切项目、重启服务、关闭服务时把一次离开动作放大成多轮交互。

### 5. 刷新恢复

- 进入项目后恢复 `openDocumentPaths[] + activeDocumentPath`。
- 活动标签优先加载，其他标签可按顺序补齐正文。
- 未完成正文加载的后台标签在标签栏中显示“已恢复但未加载完成”的轻量占位态；若用户立即切换到该标签，则把该标签提升为前台加载并显示文档级 loading 状态。
- 若恢复路径已不存在，则跳过该标签并给出状态提示，但不得阻断其余标签恢复。

### 6. 模式与解锁态归属

- `mode` 是工作区会话级状态，不跟随单个标签保存独立值。
- `regularViewState` 也是工作区会话级状态，不允许某个标签记住“自己是 editable，另一个标签是 locked”。
- 因此：
  - 工作区当前若处于 `split`，切换标签后仍然是 `split`。
  - 工作区当前若处于 `regular + editable`，切换标签后仍然是 `regular + editable`。
  - 刷新恢复时恢复的是“上次工作区处于什么模式”，不是“每个标签各自处于什么模式”。
- 这样做的原因是当前仓库已经把“解锁是全局工作区状态”作为明确方向，多标签方案不能再把这一点拆回文档级。

## 动作与保存闸门矩阵

| 动作 | 作用域 | 是否需要保存闸门 | 推荐处理 |
| --- | --- | --- | --- |
| 切标签 | 标签级 | 否 | 只同步当前活动标签槽，再切活动标签 |
| 关闭 clean 标签 | 标签级 | 否 | 直接关闭 |
| 关闭 dirty 标签 | 标签级 | 是 | `保存 / 放弃 / 取消`，只作用于当前标签 |
| 文件树点击已打开文档 | 标签级 | 否 | 直接切到已有标签 |
| 文件树点击未打开文档 | 标签级 | 否 | 先把当前活动标签 draft 同步回标签槽，再新建标签并切入；不因别的 dirty 标签全局阻断 |
| 切项目 | 会话级 | 是 | `保存全部 / 放弃全部 / 取消` |
| 切 profile | 会话级 | 是 | `保存全部 / 放弃全部 / 取消` |
| 重启服务 | 会话级 | 是 | `保存全部 / 放弃全部 / 取消` |
| 关闭服务 | 会话级 | 是 | `保存全部 / 放弃全部 / 取消` |
| 切 mode | 会话级 | 否 | 直接切换会话 mode，并保留各标签 draft |
| 点击锁定 | 会话级 | 是 | 先 flush 当前 dirty 标签集合；全部成功后进入 `locked`，失败则保持 `editable` |

说明：

1. 多标签方案下，切标签和打开新标签都不应升级成全局保存拦截点，否则“允许多个 dirty 标签同时存在”会失效。
2. 会话级离开动作统一走一套三选一确认流，避免不同入口出现不同的脏数据语义。
3. `点击锁定` 仍然属于会话级动作；为了保持“解锁是全局工作区状态”，不允许只锁当前标签。

## 顶部标签栏与视觉方向

### 顶栏布局调整

顶栏继续保持三段式布局，但中间区域不再用于显示工作区状态文案，而是直接改为标签栏：

1. 左侧：项目切换与接入项目。
2. 中央：多标签栏。
3. 右侧：固定宽度图标操作区。

明确删除以下顶部信息：

- 独立项目名文本
- 独立当前文档名文本
- `状态：...`
- `保存：...`

原因是左侧项目下拉已经足够表达当前项目，中央区域应优先让位给标签；否则多标签主交互和旧状态块会争夺同一视觉焦点。

右侧操作区的进一步约束如下：

1. 锁定、模式切换、Profile、服务操作全部收敛成图标按钮，不再保留大段文字按钮。
2. 右侧操作区参考 Chrome 的窗口右端控制区思路，占据一段固定宽度区域，不随标签数量线性膨胀。
3. 中央标签栏只消费“左侧项目区”和“右侧固定图标区”之间的剩余空间。
4. 顶栏布局优先保证右侧图标区稳定可用，不允许标签栏把右侧控制区挤坏。

### 标签视觉方向

标签栏视觉参考 Chrome，而不是传统 IDE 风格的厚重标签：

1. 标签横向连续排列，占据顶部中央主航道。
2. 活动标签更亮、更抬起，非活动标签更扁、更弱。
3. 标签之间边界轻，不使用厚卡片或强描边矩形。
4. 标签内容优先展示文档名，并在右侧提供轻量关闭按钮。
5. dirty 状态优先通过标签内小圆点表达，而不是额外文本行。
6. saving / save failed 等状态只在活动标签内做轻量反馈，不再占据独立顶部状态区。
7. 参考 Chrome 的信息架构、宽度压缩方式和活动态层级，不直接复刻浏览器系统外轮廓或窗口控制按钮皮肤。

### 单行与窄宽度策略

标签栏必须保持单行，不允许在移动或窄宽度下掉到第二行。具体规则如下：

1. 顶部标签栏始终单行显示，不允许换行。
2. 当可用宽度变窄时，优先压缩单个标签宽度，而不是把标签栏拆成两行。
3. 标签宽度压缩方式参考 Chrome：
   - 常规桌面宽度下，标签保持较舒展的默认宽度。
   - 宽度不足时，标签按比例逐步变窄。
   - 允许压缩到很窄，但仍需保留最基本的可点击面积、活动态识别和关闭入口。
4. 当标签很多时，优先让标签文本截断，而不是挤压右侧控制区。
5. 标签栏可以使用单行横向滚动或内部裁切策略，但不能换行到第二层。
6. 窄宽度下仍要保持右侧图标区固定占位，不让其被标签吞没。

### 非活动标签状态表达

顶部取消全局 `状态 / 保存` 文案后，非活动标签仍需保留最低限度的状态表达：

1. dirty 的非活动标签至少显示小圆点或等价轻量标记。
2. 保存失败的非活动标签至少显示错误态徽记，不能只在活动标签上可见。
3. 活动标签可以额外展示 saving 等更细粒度反馈，但非活动标签不能完全失去脏状态和失败状态提示。

### 正文区元信息收缩

当前正文区上方的文档元信息块也应同步收缩，避免和顶部标签重复表达同一信息：

1. 顶部标签栏成为主文档入口后，正文上方不再需要重复显示大块文档名/路径。
2. 若仍需保留路径信息，应降级为 hover / tooltip 等次级信息，而不是继续占用显著垂直空间。
3. 本轮推荐直接删除或显著弱化 `workspace__document-meta`，让正文更快进入内容本身。

## 文件树联动规则

本轮不再保留“当前文档祖先目录不可折叠”的硬锁定策略。新的文件树规则是：

1. 文件树高亮只跟活动标签走。
2. 活动标签路径在激活时默认展开。
3. 用户仍可手动折叠当前活动分支。
4. 已打开但非活动的其他标签，不参与当前高亮与强制展开。

这样才能真正解决“当前打开文件夹无法折叠”的问题，而不是仅用多标签把原问题遮住。

## 数据持久化与恢复边界

`src/workspace/local-state.ts` 需要从当前单文档恢复模型升级为：

```ts
type WorkspaceLocalState = {
  openDocumentPaths: string[]
  activeDocumentPath: string | null
  activeMode: 'regular' | 'split'
  regularViewState: 'locked' | 'unlocking' | 'editable' | 'locking'
  tabStateByDocument: Record<string, {
    lastKnownScrollTop: number
  }>
  readingProgressByDocument: Record<string, number>
}
```

约束如下：

1. local state 只保存会话恢复所需的轻量信息，不保存整份 draft 正文。
2. 正文恢复仍以 bridge 重新读取为主，避免把本地会话状态变成第二份文档真相源。
3. `profile-store` 继续负责布局和外观偏好，不承担标签集合恢复职责。
4. 本轮应顺带清理旧的 `read/edit` 模式残留枚举，避免把过时模式继续固化进新的多标签恢复结构。

## 实现落点

### `src/App.tsx`

主状态源从单文档字段切换为 `WorkspaceSession`，并新增至少以下动作：

- `openOrActivateTab(path)`
- `activateTab(tabId)`
- `closeTab(tabId)`
- `updateActiveTabDraft(content)`
- `saveTab(tabId)`
- `restoreSession(projectId, profileId)`

### `src/app/WorkspaceLayout.tsx`

从接收 `currentDocumentPath/currentDocumentContent/editingDocumentContent` 转为接收：

- `tabs`
- `activeTabId`
- `activeTab`
- `onTabSelect`
- `onTabClose`
- `onDocumentOpen`

同时新增标签栏，并把文件树逻辑改为围绕活动标签路径工作。

正文上方现有文档元信息块需要同步弱化或删除，避免与顶部标签栏重复。

### `src/app/AppShell.tsx` 与 `src/app/TopBar.tsx`

- `AppShell` 作为透传层同步升级 props。
- `TopBar` 中央区域从 `topbar__status` 改为标签栏。
- 顶部不再单独显示项目名、当前文档名、`状态：...`、`保存：...`。
- 活动标签本身承担文档名、dirty、小型保存反馈等主要信息表达。
- 右侧操作区改成固定宽度图标带，不再使用大段文字按钮。
- 顶栏单行布局在窄宽度下也不换行；标签栏通过压缩宽度与截断来适配。

### `src/workspace/local-state.ts`

升级本地恢复模型，支持标签集合与活动标签恢复，并同步收敛旧的 `read/edit` 残留模式字段。

## 风险与控制点

### 1. 标签状态串槽

高风险字段包括：

- `draftContent`
- `saveState`
- `saveErrorMessage`
- `mtimeMs`
- `lastKnownScrollTop`

这些字段只要有任何一处仍按单文档全局变量回写，就会导致 A/B 标签状态互串。

### 2. 保存/读取异步回包污染活动槽

必须建立“请求只写回发起它的标签”的硬规则。否则会出现 A 标签保存或加载回包覆盖 B 标签状态。

这条风险必须直接落成验证规则：

- A 标签后台保存发起后切到 B 标签
- A 的保存回包回来时，只允许更新 A 标签的 `persistedContent/saveState/mtimeMs`
- 不允许覆盖当前活动标签 B 的正文、提示或 dirty 状态

### 3. 恢复链路分裂

如果 local state 已恢复标签集合，但 UI 某些分支仍按单一 `currentDocumentPath` 初始化，会导致：

- 标签条恢复了，但正文区不一致
- 文件树高亮错位
- 顶栏标题仍显示旧文档

### 4. 文件树旧规则残留

如果继续保留 `forcedExpandedDirectories` 的硬锁逻辑，只是把问题从“当前文档”挪到“活动标签”，并没有真正解决可折叠诉求。

这里应明确收敛方式：

- 不再保留“路径命中就不可折叠”的硬锁。
- 改成“激活标签时自动展开一次对应路径”。
- 用户手动折叠后，在下一次再次激活该标签前，不强行重新展开。

### 5. 顶栏空间竞争

顶部标签栏替换状态块后，新的主要风险变成“中央标签栏 vs 右侧控制区”的空间竞争。

收敛规则如下：

- 右侧图标区固定占位，优先稳定。
- 中央标签栏承担压缩与截断压力。
- 标签栏允许单行滚动或窄宽度压缩，但不允许把右侧控制区挤换行或挤出可点击范围。

## 完成标准

满足以下条件时，本轮多标签工作区设计即可进入实施计划：

1. 工作区真相源已从单文档模型升级为会话模型设计。
2. 多标签 dirty/save/error 的归属边界清晰，不再依赖全局 `currentDocument*` 假设。
3. 文件树折叠逻辑已从“当前文档强制展开不可折叠”调整为“活动标签默认展开但允许手动折叠”。
4. 恢复链路明确为“恢复标签集合与活动标签”，而不是只恢复一个当前文档。
5. 风险点、验证方案与非目标均已写清，可直接进入实施计划拆解。

## 最终结论

建议引入“多标签工作区完整模型”。

原因不是为了追求更复杂的架构，而是当前已确认的四条用户约束已经超出单文档状态机的可承受范围：允许多个 dirty 标签、关闭只拦截当前标签、点击已打开文档只切已有标签、刷新恢复整组标签集合。这四条约束叠加后，继续停留在单文档模型上只会得到一套表面像多标签、底层仍然频繁串槽的实现。

因此，本轮应以工作区会话为真相源进入后续实施计划，而不是继续在 `currentDocument*` 模型上做局部补丁。
