# 方案概述

## 1. 总体目标和范围

本方案的目标，是把 `md-reader` 当前的三模式工作区进一步收敛为“两层心智”：

- 主工作区只保留一个可视文档层
- 该层默认以“锁定态”打开
- 解锁后原地进入可编辑态
- `split` 继续承担源码精修

这意味着产品层面不再保留单独的“阅读模式”，而改成：

- `常规`
- `分栏`

其中：

- `常规` 是唯一主页面
- `常规` 默认锁定，仅可浏览和导航
- 用户解锁后，仍停留在同一页面骨架中编辑

本方案覆盖：

- 移除独立 `read` 模式的产品定义
- `edit` 重命名为 `常规`
- 右上角新增 `锁定 / 解锁` 交互
- 锁定态与解锁态的行为边界
- 自动保存与锁定切换关系
- `split` 保留边界
- 状态模型与实施验收标准

本方案不覆盖：

- 立即实施代码改动
- 重做 `split` 左侧源码编辑器
- 扩展表格、callout、frontmatter、HTML block 等复杂语法支持
- 重写 autosave 主状态机
- 在本轮内一次性删空所有只读技术实现；第一阶段允许存在过渡实现，但产品层与状态层不得继续暴露 `read` 模式

## 2. 各阶段任务概要

### 阶段一：模式收敛与状态定义

- 主要工作：移除 `read` 模式，重新定义 `常规 / 分栏` 两种模式，并引入 `isEditLocked` 状态。
- 预期成果：顶部模式切换和工作区主分支从三态收敛到两态。
- 执行顺序：必须最先完成。

### 阶段二：锁定态交互接入

- 主要工作：为 `常规` 模式加入 `锁定 / 解锁` 控制，并明确锁定态下的可用能力。
- 预期成果：主页面默认可浏览、可导航、不可编辑。
- 执行顺序：模式收敛后进行。

### 阶段三：保存与切换联动

- 主要工作：让“解锁 -> 锁定”动作先触发自动保存，保存成功后再真正锁定。
- 预期成果：锁定动作与当前 autosave 机制一致，不引入新的确认框心智。
- 执行顺序：锁定态骨架稳定后接入。

### 阶段四：文案、导航与回归

- 主要工作：统一顶部文案、标题导航、默认进入态、测试与浏览器回归。
- 预期成果：产品心智和代码边界一致。
- 执行顺序：最后收口。

## 3. 整体结构框架

收敛后的工作区结构如下：

1. 模式层
   - `常规`
   - `分栏`

2. 主文档层
   - `常规 + 锁定`
   - `常规 + 解锁`

3. 精修层
   - `分栏`
   - 左侧源码
   - 右侧预览

这条路线的核心不是“删除阅读能力”，而是“删除独立阅读模式和独立阅读心智”。

同时，这条路线需要拆成两个收敛面来看：

1. 产品收敛
   - 用户只看到 `常规 / 分栏`
   - 主页面默认锁定
   - 解锁后原地编辑

2. 技术收敛
   - 第一阶段可以接受锁定态短期复用现有只读技术实现
   - 但状态层和产品层不得继续保留 `read`
   - 长期目标仍是把锁定态与解锁态并回同一主编辑骨架

---

# 1. 现状问题

当前产品和代码结构里，`md-reader` 仍然保留三种模式：

- `read`
- `edit`
- `split`

这会持续带来 3 个问题：

1. 产品心智偏重  
   用户需要先理解“阅读”和“编辑”是两个模式，再理解“分栏”是第三种模式。

2. 视觉与状态收敛成本高  
   即使渲染内核持续统一，只要“阅读”和“编辑”是两个主模式，产品层就会天然保留切换感。

3. 主工作区目标不够明确  
   当前真正的长期目标其实已经不是“阅读页 + 编辑页”，而是“同一页面既能看也能改，只是默认锁住”。

所以，这一轮更合理的方向不是继续优化 `read <-> edit` 切换，而是直接取消独立 `read`。

# 2. 推荐方案

推荐采用以下方案：

## 2.1 模式收敛

顶部模式从：

- `阅读`
- `编辑`
- `分栏`

收敛为：

- `常规`
- `分栏`

其中：

- 原 `edit` 模式升级为唯一主工作区
- 原 `read` 模式被移除
- 原 `split` 模式保留

## 2.2 主页面锁定机制

`常规` 模式默认进入锁定态：

- 页面可滚动
- 标题导航可点击
- 标题高亮与折叠照常工作
- 不允许修改正文

用户点击右上角“解锁”后：

- 保持当前页面骨架不变
- 原地进入可编辑态
- 不发生模式切换

用户点击“锁定”时：

- 先触发保存
- 保存成功后再进入锁定态
- 保存失败则留在解锁态

这里补一条明确边界：

- 锁定按钮只作用于 `常规`
- 不作用于 `分栏`

## 2.3 `split` 保留职责

`split` 继续保留，不参与这次主模式简化：

- 左侧：源码精确修订
- 右侧：预览对照

它的职责不变：

- 不是默认主页面
- 不是一般阅读入口
- 不是普通写作入口
- 只用于需要精修 Markdown 原文的场景
- 左侧源码编辑继续允许直接修改
- 完全不受右上角锁定按钮影响

---

# 3. 为什么这是更好的方向

## 3.1 产品心智更简单

用户以后只需要理解：

- 平时在 `常规`
- 需要改时先解锁
- 需要精修原文时进 `分栏`

这比“阅读 / 编辑 / 分栏”更自然，也更接近你真正想要的 Notion 式感觉。

## 3.2 主工作区不再有“切页感”

如果继续保留 `read`，哪怕底层越来越统一，用户还是会感知到：

- 我在一个页面里看
- 切一下，进入另一个页面里改

而锁定态方案会把它改成：

- 我一直在同一个页面
- 只是现在锁着 / 现在解开了

## 3.3 更符合当前技术收敛方向

当前项目已经在持续推动：

- 阅读与编辑更接近同一渲染体系
- 视觉骨架尽量一致
- 自动保存替代手动确认

那最顺势的产品结构，就是把“阅读模式”这层产品壳去掉。

但这里要明确：去掉的是产品层的 `read`，不是承诺第一阶段就把所有只读技术实现一次性物理删空。第一阶段允许技术过渡，但不允许继续把它包装成独立模式和独立心智。

# 4. 详细交互设计

## 4.1 顶部模式切换

顶部模式按钮改为：

- `常规`
- `分栏`

不再显示：

- `阅读`
- `编辑`

同时，在右上角增加锁定按钮：

- 锁定态显示：`解锁`
- 解锁态显示：`锁定`
- 进入锁定中的中间态显示：`正在锁定…`
- 进入解锁中的中间态显示：`正在解锁…`

推荐放在模式按钮附近，而不是放进保存状态文案区。

## 4.2 默认进入行为

每次打开文档，默认进入：

- `常规`
- `锁定态`

不恢复上次这个文档的锁定状态。

但同一轮工作中的“解锁状态”是全局工作区状态，而不是文档私有状态：

- 进入项目后首次打开文档默认锁定
- 一旦用户在 `常规` 中解锁，继续切换到其他文档时仍保持解锁
- 切换文档时模式一律回到 `常规`

这样做的原因是：

- 心智稳定
- 安全边界更明确
- 避免一打开文档就处于可编辑态
- 同时满足“解锁是当前工作区状态，而不是跟随单篇文档”

## 4.3 锁定态行为

锁定态下允许：

- 页面滚动
- 标题导航点击跳转
- active heading 高亮
- 目录折叠展开
- 文本选择
- 链接点击

锁定态下禁止：

- 键入修改
- 粘贴改写
- 删除
- 回车插入块
- 任何会改变 Markdown 内容的输入

标题导航与标题锚点在锁定态下必须继续沿当前 `documentHeadings + data-heading-id` 机制作为唯一正式来源，不允许重新演化出“锁定态一套、解锁态一套”的目标链路。

## 4.4 解锁态行为

解锁态就是当前“可视编辑态”的正式工作态：

- 可输入
- 可自动保存
- 导航继续可用
- 页面骨架不变
- 可以出现最小编辑提示，例如光标、选区和必要的块级编辑 affordance
- 但页面排版和主骨架不能发生明显跳变

这里最关键的要求是：

- 解锁后不能像以前那样产生“进入另一套页面”的明显感觉

## 4.5 从解锁切回锁定

点击“锁定”时的推荐顺序：

1. 触发保存 / flush
2. 锁定按钮进入禁用态并显示“正在锁定…”
3. 若保存成功，切为锁定态
4. 若保存失败，保持解锁态
5. 顶部继续显示失败状态，按钮恢复可点击

这条规则必须和当前 autosave 机制保持一致，不要再引入新的确认弹框。

解锁按钮的规则也要写死：

1. 解锁本身不是保存动作
2. 解锁失败只可能来自编辑器未就绪或工作区阻塞状态
3. 若解锁失败，继续停留在锁定态
4. 顶部应显示可理解的失败状态，按钮恢复到可重试

# 5. 状态模型建议

当前模式状态建议从：

- `mode: 'read' | 'edit' | 'split'`

改为：

- `mode: 'regular' | 'split'`
- `regularViewState: 'locked' | 'unlocking' | 'editable' | 'locking'`

默认值：

- `mode = 'regular'`
- `regularViewState = 'locked'`

状态语义如下：

## `mode = 'regular' && regularViewState = 'locked'`

主可视页面，锁定态，只读可导航。

## `mode = 'regular' && regularViewState = 'unlocking'`

主可视页面，等待进入可编辑态，按钮禁用。

## `mode = 'regular' && regularViewState = 'editable'`

主可视页面，解锁态，可编辑。

## `mode = 'regular' && regularViewState = 'locking'`

主可视页面，正在 flush / 保存并准备切回锁定。

## `mode = 'split'`

源码精修模式。

这里有一个明确约束：

- `split` 不需要再叠加锁定态概念
- `split` 不读取也不写入 `regularViewState`

因为 `split` 本身已经是高意图模式，进入它就默认用户要精修内容。

# 6. 对现有实现的结构影响

## 6.1 `TopBar`

需要做的变化：

- 移除 `read`
- 将 `edit` 文案改为 `常规`
- 保留 `split`
- 增加 `锁定 / 解锁` 按钮
- 增加 `locking / unlocking` 中间态文案和按钮禁用规则

## 6.2 `WorkspaceLayout`

需要做的变化：

- 移除独立 `read` 分支
- 主页面改成 `regular` 分支
- `regular` 分支内部根据 `regularViewState` 控制可编辑性
- 标题同步策略必须继续保持单一正式来源，不允许锁定态和解锁态分裂

最重要的结构原则是：

- 不要再保留“一个 `read` 分支 + 一个 `edit` 分支，只是换名字”

否则产品看起来简化了，代码结构其实没变。

## 6.3 只读渲染层

长期推荐方向是：

- 不再把“只读渲染器”当成独立产品层
- 而是让主可视编辑骨架自己支持锁定与解锁

第一阶段可以接受为了风险控制复用当前只读能力，但必须同时满足：

- 顶部与状态层不再存在 `read`
- 文档切换逻辑只围绕 `regular / split`
- 所有代码注释、状态命名和实施计划都把它标记为过渡技术实现
- 第二阶段继续朝“同一主编辑骨架的 readonly / editable 双态”收敛

## 6.4 autosave

需要接入的只有一点：

- “锁定”动作必须先走保存成功，再真正锁定

autosave 的主状态机不应因为这次方案被重写。

---

# 7. 推荐实施策略

我推荐分两步实施，而不是一次性硬切。

## 第一步：产品层先收敛

- 模式改成 `常规 / 分栏`
- 默认进入 `常规 + 锁定`
- 加上锁定按钮
- 保持现有自动保存联动
- 明确解锁是全局工作区状态，不是文档私有状态
- 明确 `split` 不受锁定按钮控制

这一步先把用户看到的产品结构简化掉。

## 第二步：技术层再继续收敛

- 逐步减少独立只读实现分支
- 把锁定态尽量并回主编辑骨架
- 最终让“常规”的锁定和解锁只是一套内核的两种工作态
- 同时保持标题导航继续只走同一条 `documentHeadings + data-heading-id` 目标链

这样做的原因是：

- 产品收益可以先拿到
- 技术风险不需要一次性吃完

---

# 8. 验收标准

如果后续实施，这个方案至少应满足：

1. 顶部不再显示 `阅读` 和 `编辑`，而是 `常规` 和 `分栏`。
2. 打开文档默认进入 `常规` 锁定态。
3. 解锁状态是全局工作区状态；切换到其他文档时模式回到 `常规`，但仍保持当前是否解锁。
4. 锁定态下正文不可修改，但可滚动、可导航、可跳转。
5. 解锁后原地进入可编辑态，不出现明显页面骨架跳变；允许最小编辑提示出现。
6. 从解锁切回锁定时，必须先保存成功；保存失败则保持解锁。
7. `split` 功能不回退，左侧源码编辑完全不受锁定按钮影响。
8. 标题导航在锁定态和解锁态下继续使用同一条正式目标链。
9. 代码结构上不能只是把 `read` 改名成 `常规` 后继续保留完整独立分支心智。

实施前最小验证清单也应写死：

- 状态机测试：`regularViewState` 与 autosave flush 联动
- 组件测试：`TopBar` 模式文案与锁定按钮状态
- 集成测试：`WorkspaceLayout` 的 `regular / split` 分支与标题导航行为
- 浏览器回归：默认锁定、解锁编辑、切文档保持解锁、锁定失败留在解锁、`split` 不受锁影响

---

# 9. 结论

这次收敛的正确方向，不是继续维护“阅读页”和“编辑页”，而是：

- 只保留一个主可视工作区
- 默认锁住
- 解锁后原地编辑
- `split` 作为精修模式保留

并且顶部产品语言也应同步收敛：

- `常规`
- `分栏`

不再显示“编辑”这个模式名。

这是当前阶段最符合你目标的方案，因为它真正减少了模式数、状态分叉和产品心智负担，而不是只在现有三模式上继续做表面收敛。
