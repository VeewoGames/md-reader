# md-reader 第一阶段自动保存与分栏编辑执行计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在保留当前 `read / edit / split` 工作区的前提下，先把 `draft`、自动保存、切换前 `flush`、状态展示和本地恢复链路跑通，为后续 Notion-like 可视编辑器接入建立稳定底座。

**Architecture:** 当前阶段不直接上可视编辑器重写，而是先在 `App` 层补齐“已保存内容 / 草稿内容 / 保存状态 / flush 策略”四件套。`edit` 与 `split` 共用同一份 `draftDocumentContent`，`read` 只展示已成功保存的内容；任何会让用户离开当前上下文的动作，都必须先尝试 `flush`。第一阶段继续沿用现有 `textarea + preview`，只收敛模式语义和保存链路。

**Tech Stack:** React 19、TypeScript、Vite、Vitest、Testing Library、本地 bridge API

---

## 概述

### 1. 总体目标和范围

- 总体目标：让 `md-reader` 在本地服务模式下具备稳定的 Markdown 读取、编辑、分栏预览、自动保存、状态恢复与服务控制协同能力。
- 本阶段范围：
  - 固定 `read / edit / split` 三模式职责。
  - 引入 `draftDocumentContent` 与自动保存状态机。
  - 打通切文档、切项目、切到阅读、重启服务、关闭服务前的 `flush`。
  - 在顶部状态区展示保存状态。
- 不在本阶段范围：
  - Notion-like 块级可视编辑器。
  - unsupported fragment 的结构化降级实现。
  - 浏览器授权模式。

### 2. 各阶段任务概要

- 阶段 A：补齐实现计划与测试入口。
  - 主要工作：基于当前真实代码状态，明确状态模型、文件边界、测试命令。
  - 预期成果：执行顺序固定，避免继续在临时编辑逻辑上叠补丁。
- 阶段 B：先用测试钉住 `flush` 约束。
  - 主要工作：新增“编辑后切换到阅读会先保存”“切文档前先保存”的失败测试。
  - 预期成果：后续状态机实现有明确回归护栏。
- 阶段 C：实现 `draft + autosave + flush` 状态机。
  - 主要工作：在 `src/App.tsx` 收敛文档真相层，接入 bridge 保存 API。
  - 预期成果：`edit / split` 修改后自动保存，`read` 始终展示已保存内容。
- 阶段 D：补齐外围动作与状态展示。
  - 主要工作：服务重启、服务关闭、项目切换、文档切换与顶部保存状态联动。
  - 预期成果：用户不会因为上下文切换误丢改动，也能看到明确状态。
- 阶段 E：验证。
  - 主要工作：跑定向测试、全量测试和构建。
  - 预期成果：第一阶段可交付。

### 3. 整体结构框架

- `src/App.tsx`
  - 负责工作区主状态、自动保存调度、切换前 `flush`、bridge 调用编排。
- `src/app/TopBar.tsx`
  - 负责显示当前项目/文档/保存状态，并承接模式切换与服务控制入口。
- `src/app/AppShell.tsx`
  - 负责把保存状态透传给头部和工作区。
- `src/app/WorkspaceLayout.tsx`
  - 继续承载 `read / edit / split` 的渲染分支，本阶段不做可视编辑器重构。
- `src/workspace/local-bridge-access.ts`
  - 已有保存 API，本阶段只复用，不重复发明保存协议。
- `tests/app/*.test.tsx`
  - 承担工作区模式与自动保存行为回归。
