## 方案概述

### 1. 总体目标和范围

本方案用于收敛 `md-reader` 下一阶段的可视编辑优化方向，核心目标有两项：

- 让 `edit` 模式尽量与 `read` 模式保持同构，做到“切到编辑态时页面仍然像当前文档本身”，而不是明显切换到另一套编辑器皮肤。
- 把当前已接入的 `Milkdown / Crepe` 可视编辑器从首屏主包中拆出去，改为按需加载，降低阅读场景的初始加载成本。

本方案聚焦以下范围：

- 收敛阅读态与编辑态当前差异来源，并定义“同构编辑”的验收口径。
- 规划阅读渲染层与可视编辑层的样式 token 对齐方案。
- 规划 `Milkdown / Crepe` 的动态加载、降级占位与风险控制方案。
- 明确第一版 `edit` 的稳定支持范围与明确非目标，避免范围失控。
- 明确这轮只产出方案，不直接改运行时代码。

本方案不包含以下内容：

- 把 Markdown 真相层改造成专有 block JSON。
- 直接在本轮实现表格列宽、callout 类型、折叠块等复杂增强。
- 直接重写当前 `split` 模式职责。
- 要求第一版 `edit` 覆盖全部 Markdown 复杂结构并与 `read` 像素级一致。

### 2. 各阶段任务概要

第一阶段：差异收敛与同构目标定义

- 主要工作：明确当前 `read` 与 `edit` 的视觉差异到底来自哪些层。
- 预期成果：得到一份“必须统一”的样式与交互清单。
- 执行顺序：先找差异来源，再定义同构目标。

第二阶段：编辑态样式收敛方案

- 主要工作：把阅读态的排版 token、块间距、容器宽度、块级视觉语言迁移到编辑态。
- 预期成果：`edit` 模式在静态视觉上尽量贴近 `read`。
- 执行顺序：先确认编辑器可控边界，再统一内容区骨架，再统一块级样式，最后收敛编辑器 chrome。

第三阶段：编辑器按需加载方案

- 主要工作：把 `Milkdown / Crepe` 从主包首屏路径中拆分出来，只在进入 `edit` 时加载。
- 预期成果：阅读场景首屏更轻，首次编辑时再加载编辑器运行时代码。
- 执行顺序：先拆入口，再做加载占位，再补回归验证。

第四阶段：联调与验收

- 主要工作：验证阅读态、编辑态、分栏态三者职责是否仍然清晰，自动保存与标题导航是否未被破坏。
- 预期成果：形成可执行的后续实现计划和验收口径。
- 执行顺序：先视觉验收，再性能验收，最后行为回归。

### 3. 整体结构框架

本方案下，后续实现按四层收敛：

- Markdown 真相层：仍然只认原生 Markdown。
- 阅读渲染层：负责 `read` 模式的最终页面表达。
- 可视编辑层：负责 `edit` 模式的结构化编辑与最小 chrome。
- 工作区加载层：负责阅读/编辑模式切换时的运行时加载策略。

第一版 `edit` 的稳定支持范围先收敛为：

- 正文
- 标题
- 列表
- 引用
- 代码块
- 链接

以下内容不纳入第一版“必须与阅读态高度同构”的承诺范围：

- 表格
- 任务列表
- 复杂嵌套列表
- frontmatter
- HTML comment
- 原始 HTML
- 其他尚未被可视编辑器稳定结构化的内容

## 一、当前问题的真实含义

### 1.1 用户要的不是“更好看的编辑器”

你这次补充的重点是：

- 切换到 `edit` 后，页面不能明显“变成另一套 UI”。
- 编辑态应尽量像当前文档本身，而不是一个带默认皮肤的第三方富文本组件。
- 用户在阅读态看到的层级、留白、段落节奏，进入编辑态后应该大体不变。

这意味着目标已经从“接入可视编辑器”升级成：

- **阅读态与编辑态尽量同构**
- **只增加可编辑能力，不引入明显的视觉模式割裂**

### 1.2 当前为什么还不像

结合现有实现，差异主要来自四类来源：

#### A. 内容排版来自两套系统

当前阅读态依赖：

- `react-markdown`
- `remark-gfm`
- `src/index.css` 里的 `.markdown-document ...`

而编辑态依赖：

- `Milkdown / Crepe` 默认主题
- 默认块级 UI 和默认工具面板

所以两边虽然都在表达同一份 Markdown，但字体、层级、间距、块样式都不是一个体系。

#### B. 编辑器自带 chrome 暴露过多

当前浏览器实测已经能看到：

- 默认插入链接框
- 默认的块类型菜单
- 默认的文本 / 列表 / advanced 分组菜单

这些 UI 会让用户强烈感知“我正在使用一个第三方编辑器”，而不是“我正在编辑这篇文档本身”。

#### C. 容器骨架还没统一

阅读态现在更像“文档页”：

- 有稳定文档宽度
- 有统一段落节奏
- 有较轻的文档背景感

编辑态虽然已经进入 `workspace__editor-pane`，但编辑器内部根节点还没有继承同样的页面骨架，因此会出现内容区密度、宽度、边距与阅读态不一致。

#### D. 运行时代码首屏负担过重

当前 `visual-markdown-editor.tsx` 在模块顶层直接：

- 引入 `@milkdown/react`
- 引入 `@milkdown/crepe`
- 引入 `@milkdown/crepe/theme/classic.css`

这意味着即使用户只是打开项目、浏览文档、不进入编辑模式，也会把整套编辑器依赖打进主包。

#### E. 割裂不只是样式，还来自渲染链天然不同

当前阅读态与编辑态不是同一套渲染器：

- 阅读态走 `react-markdown + remark-gfm`
- 编辑态走 `Crepe` 内部的 ProseMirror / Milkdown 渲染链

这意味着很多差异不是单纯改 CSS 就能消掉，例如：

- 节点 DOM 结构不同
- 默认块级容器不同
- 默认交互与选区行为不同
- 默认 chrome 的插入点不同

因此，后续实现必须先验证：

- `Crepe` 默认壳层是否允许足够细粒度地收敛节点 class、theme 与 chrome
- 如果不允许，就要预留退到底层 `Milkdown` 自定义装配的后备路线

## 二、下一阶段目标定义

### 2.1 什么叫“阅读态与编辑态同构”

这里的“同构”不是像素级完全一致，而是满足以下体验目标：

1. 页面骨架一致

- 文档内容区宽度一致
- 左右留白一致
- 标题到正文的节奏一致

2. 块级表达一致

- `H1/H2/H3` 的字号、字重、上下间距接近一致
- 段落、列表、引用、代码块、表格的视觉语义一致
- 编辑态只比阅读态多出焦点提示与可编辑 affordance

3. 模式切换心智一致

- `read -> edit` 不是“跳到另一页”
- 更像“当前页面被激活为可编辑”

4. 结构来源一致

- 标题导航继续只基于 Markdown 标题语义
- 不是阅读态一套 DOM，编辑态再临时拼一套目录语义

### 2.2 不要求什么

这轮不要把目标定得过头：

- 不要求复刻 Notion 的全部 block toolbar 与动画细节
- 不要求第一版就支持所有复杂 Markdown 结构的完全可视编辑
- 不要求阅读态和编辑态做到零差异的像素级拟合

正确目标是：

- 先消灭最明显的“第三方编辑器味”
- 再把主视觉语言统一

### 2.3 第一版同构边界

第一版 `edit` 的目标不是“所有 Markdown 结构都与 `read` 同构”，而是先把最常用主链打通。

第一版必须重点覆盖：

- 正文
- 标题
- 列表
- 引用
- 代码块
- 链接

第一版明确接受暂不完全同构或继续降级处理的内容：

- 表格
- 任务列表
- 复杂嵌套列表
- frontmatter
- HTML comment
- 原始 HTML

这条边界必须先写死，否则执行时会自然滑向“所有内容都要一样”，范围会迅速失控。

## 三、编辑态样式收敛方案

### 3.0 先确认 `Crepe` 的可控边界

在开始大规模样式收敛前，必须先做一轮可控性确认，避免后面发现选错层。

需要先验证三件事：

1. `Crepe` 默认壳层里，哪些 UI 能通过 feature 配置直接关闭。
2. 哪些 UI 只能通过样式隐藏，但结构和交互仍然存在。
3. 哪些 UI 或节点行为如果想真正改干净，必须放弃 `Crepe` 默认壳层，改为更底层的 `Milkdown` 自定义装配。

这一步的目的不是马上重写，而是先确认路线：

- 如果 `Crepe` 可控性够高，就继续用它收敛第一版。
- 如果 `Crepe` 可控性不够，就尽早切换到底层 `Milkdown` 自定义装配，而不是在默认壳层上硬缝。

### 3.1 先统一“文档页骨架”

第一步不是改单个块，而是统一页面骨架。

建议抽一层共享文档样式 token，例如：

- `--doc-max-width`
- `--doc-body-font-size`
- `--doc-line-height`
- `--doc-heading-color`
- `--doc-block-gap`
- `--doc-inline-code-bg`
- `--doc-blockquote-bg`
- `--doc-code-block-bg`

然后让以下两套渲染共同消费：

- 阅读态 `.markdown-document`
- 编辑态 `.visual-markdown-editor` 内的 Milkdown 根节点

目标是：

- 两边先拥有同样的页面宽度、字体基线、行高与块间距
- 这样切换模式时不会先在骨架层面“歪掉”

### 3.2 再统一“块级排版”

需要重点对齐的块包括：

- `h1/h2/h3`
- `p`
- `ul/ol/li`
- `blockquote`
- `pre/code`
- `table`
- `hr`
- `a`

做法不是复制一份样式，而是把阅读态现有 `.markdown-document ...` 的视觉规则抽成共享 token / 共享选择器层，再映射到编辑态的 Milkdown DOM 结构上。

换句话说：

- 阅读态不应继续是“独占样式”
- 编辑态也不应完全依赖 Crepe 默认主题

### 3.3 最后收敛“编辑器 chrome”

这是最影响 Notion 感的地方。

下一阶段应做的不是一上来加更多工具，而是先减：

- 隐藏或重做当前暴露过强的默认分组菜单
- 收敛链接输入浮层的视觉存在感
- 弱化默认边框、容器盒感、工具面板感
- 只保留最小编辑 affordance：
  - 当前块焦点
  - 选区
  - 光标
  - 必要的浮层入口

原则是：

- 用户应该先看到“文档”
- 然后才感知“它可编辑”

而不是反过来。

这里还要把 chrome 分成两类：

#### A. 可直接通过 feature 关闭的部分

这类能力应优先走配置关闭，而不是样式遮挡。

#### B. 无法直接关闭、只能隐藏或需要改装配方式的部分

这类能力必须单独列清单，避免误判为“只是样式活”。

否则执行时很容易发现：

- 看起来能藏掉
- 实际交互还在
- 或 DOM 结构仍然持续影响布局与焦点行为

### 3.4 编辑态比阅读态允许保留的少量差异

以下差异是合理且应保留的：

- 当前块焦点高亮
- 选区样式
- 光标
- 占位提示
- 轻量块操作入口

这些属于“编辑能力提示”，不属于模式割裂。

## 四、按需加载方案

### 4.1 为什么必须做

当前可视编辑器已经能工作，但有个明显代价：

- 阅读场景也要承担编辑器代码体积
- 首屏解析与执行成本变大
- 后续如果再加插件、语法包、快捷菜单，主包还会继续膨胀

对于你的项目场景，这不是次要问题。

因为你明确说了：

- 项目里有上千个 Markdown 文档
- 加载和读取速度很重要

虽然“上千文档”更大的瓶颈通常在文件树、索引、文档读取，不在单个编辑器本身，但如果首屏把编辑器整包一起加载，用户会在“只是想看文档”时承担不必要的前端成本。

### 4.2 正确的拆分目标

建议把页面加载拆成两条路径：

#### 阅读路径

- 加载工作区壳层
- 加载文件树、标题导航、阅读渲染链
- 不加载 `Milkdown / Crepe`

#### 编辑路径

- 用户第一次进入 `edit`
- 再动态 import 可视编辑器模块
- 加载完成后挂载 `Milkdown / Crepe`

这意味着：

- 默认阅读更轻
- 首次编辑会有一次可接受的惰性加载

### 4.3 具体实现路线

推荐路线：

1. 把 `VisualMarkdownEditor` 改成懒加载入口

- 例如由 `WorkspaceLayout` 在 `mode === 'edit'` 时动态加载
- 或在组件层做 `React.lazy`

2. 把编辑器相关依赖锁进单独 chunk

- `@milkdown/react`
- `@milkdown/crepe`
- 相关 theme CSS

3. 为首次进入编辑态提供轻量占位

- “正在加载编辑器…”
- 保持与当前文档页骨架一致
- 不出现明显布局跳变

4. 可选预热策略

- 用户 hover 到 `编辑` 按钮时开始预取
- 或当前文档稳定打开后在后台低优先级预取

这样能兼顾：

- 首屏轻
- 首次点编辑也不至于过慢

### 4.4 风险与控制

按需加载会带来几个新问题：

#### A. 首次进入编辑会多一次等待

解决方式：

- 设计轻量 skeleton / loading state
- 可做 hover 预热

#### B. 编辑器 CSS 延迟注入可能造成闪动

解决方式：

- 把必要骨架样式保留在主样式中
- 只把 Milkdown 专属样式延后

这里需要把 CSS 装载策略提升为明确实现约束：

- 当前 `classic.css` 在组件顶层静态引入，后续不能只做 `React.lazy` 就算完成。
- 必须明确哪些是主包保留的骨架样式，哪些是编辑器 chunk 自带样式。
- 必须验证首次进入 `edit` 时是否出现 FOUC、布局跳变或主题闪烁。

#### C. 自动保存链路不能因为懒加载被打断

解决方式：

- 只延迟编辑器 UI 的装载
- 不改变现有 draft / autosave 状态机

#### D. 懒加载中的并发切换会引入新中间态

懒加载会引入一个当前没有的状态：

- 模式已经切到 `edit`
- 但编辑器还没挂载完成

这时如果用户立刻：

- 切回 `read`
- 切到别的文档
- 重启服务
- 恢复工作区状态

就可能出现：

- 空白工作区
- 旧加载结果覆盖新文档
- 错误 dirty 状态
- 标题导航来源抖动

因此这部分必须列为实现约束，而不是可选优化项。

### 4.5 标题导航与懒加载的状态约束

后续无论是否懒加载，标题导航都应继续坚持一个规则：

- 右侧导航仍以 Markdown draft 文本为唯一来源

不要切换成：

- 阅读态从 Markdown 提取
- 编辑态从 ProseMirror 文档树提取

否则很容易出现两套来源不一致的问题。

必须验证的链路包括：

- 编辑中标题变更 -> draft 更新 -> 右侧导航更新
- 点击右侧导航 -> 正文正确跳转
- 懒加载期间 -> 标题导航不抖动、不回退到旧文档

## 五、推荐实施顺序

### 第一步：样式系统抽象

先把阅读态排版规则从 `.markdown-document` 中抽出共享 token。

目标：

- 让阅读态和编辑态有同一个排版语言基础

在正式开始前，先补一个探针任务：

- 跑一次 `vite build`
- 记录当前主 chunk 与编辑器相关 chunk 基线
- 作为后续按需加载是否真正生效的对照数据

### 第二步：编辑态内容区同构

把 Milkdown 根节点套进与阅读态一致的文档骨架中。

目标：

- 切到编辑时页面宽度、留白、层级不突变

### 第三步：块级样式映射

逐个对齐标题、段落、列表、引用、代码块、表格。

目标：

- 消除最明显的视觉割裂

### 第四步：收敛默认 chrome

隐藏或改造当前默认工具面板与浮层入口。

目标：

- 减少“第三方编辑器味”

### 第五步：编辑器按需加载

把可视编辑器从首屏主包拆出去。

目标：

- 保持阅读场景轻量

### 第六步：回归验证

至少验证：

- `read -> edit -> read`
- `split` 不回归
- 自动保存不回归
- 标题导航不回归
- 首屏主包体积下降
- 懒加载中切文档 / 切模式 / 重启服务不回归

### 第七步：必要时切换到底层 `Milkdown` 自定义装配

如果前面验证后确认 `Crepe` 默认壳层无法把 chrome 和节点表达收敛到可接受水平，就应中止继续堆补丁，改为：

- 保留 `Milkdown` 作为核心
- 放弃 `Crepe` 默认成品壳
- 自定义编辑器装配与插件组合

这不是失败路线，而是为了确保：

- 编辑态真正尽量贴近阅读态
- 不长期背负不可控的默认 UI

## 六、验收标准

后续真正实现后，至少应满足以下验收点：

1. 从 `read` 切到 `edit` 时，页面骨架、主要层级与内容节奏保持高一致性。
2. 编辑态不再明显暴露 Crepe 默认分组菜单和强工具面板感。
3. 标题、段落、列表、引用、代码块、表格在阅读态与编辑态的视觉语言基本一致。
4. 编辑态允许保留焦点、选区、光标等最小编辑 affordance，但不应显著破坏阅读态页面观感。
5. 默认进入工作区且只阅读文档时，不加载完整 `Milkdown / Crepe` 编辑器运行时代码。
6. 第一次进入编辑态时，编辑器通过按需加载挂载成功，并有可接受的过渡占位。
7. 自动保存、标题导航、分栏模式不因样式收敛或按需加载而回归。
8. 懒加载期间切模式、切文档、重启服务时，不出现空白编辑区、旧内容覆盖或错误 dirty 状态。

建议补充量化性能口径：

- `index` 主 chunk 必须相较当前基线有可观下降
- 编辑器相关代码应拆入独立 chunk
- 首次进入 `edit` 的可接受等待时间需要在方案执行前先定阈值

---

## 七、结论

你这次追加的要求，本质上把下一阶段目标从“接入一个能编辑的可视编辑器”进一步收敛成了：

- **让编辑态尽量就是阅读态本身，只是在其上获得编辑能力**
- **让编辑器能力只在需要时加载，而不是默认拖慢整个工作区首屏**

因此，后续实现主线不应该再继续堆功能，而应先解决两类基础问题：

1. 视觉同构问题

- 统一阅读态和编辑态的页面骨架、排版 token、块级样式、chrome 暴露度

2. 加载边界问题

- 把 `Milkdown / Crepe` 从主包里拆出去，改成进入编辑态后再加载

同时还要先确认一个路线前提：

- `Crepe` 默认壳层到底够不够可控
- 如果不够，就尽早切到底层 `Milkdown` 自定义装配

另外，`split` 的长期职责在这轮不变：

- 左侧继续承担源码精确修订
- 右侧继续承担预览对照
- `split` 不参与“必须做成 Notion-like 同构编辑”的目标承诺

只有这两件事先收住，后面再继续做 callout、表格列宽、块级交互增强，整体体验才不会越做越重、越做越不像你要的那种“文档本身可编辑”的感觉。
