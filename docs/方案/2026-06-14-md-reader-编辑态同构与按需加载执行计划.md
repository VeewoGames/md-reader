# md-reader 编辑态同构与按需加载执行计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在保留当前 `read / edit / split` 工作区和 Markdown-first 真相层的前提下，把 `edit` 模式收敛为尽量接近 `read` 的可视编辑页面，并把 `Milkdown / Crepe` 从首屏主包中拆出去按需加载。

**Architecture:** 本轮不把问题笼统理解为“换皮”。执行主线拆成两段：先验证 `Crepe` 是否足够可控，并在当前 `react-markdown` 阅读链与 `Milkdown / Crepe` 编辑链之间建立共享文档骨架和块级视觉语言；再把编辑器入口改为懒加载，并围绕自动保存、标题导航和上下文切换补齐并发保护。若 `Crepe` 默认壳层无法把 chrome 和节点表达收敛到可接受水平，则中止对壳层继续缝补，切换到底层 `Milkdown` 自定义装配。

**Tech Stack:** React 19、TypeScript、Vite、Vitest、Testing Library、Milkdown、Crepe、本地 bridge API

---

## 概述

### 1. 总体目标和范围

- 总体目标：让 `edit` 模式在第一版先对齐正文、标题、列表、引用、代码块、链接六类主链内容的阅读/编辑一致性，并把编辑器代码从阅读首屏路径中拆出去。
- 本阶段范围：
  - 记录当前 chunk 基线，明确按需加载收益口径。
  - 验证 `Crepe` 默认壳层可控性，区分“可配置关闭 / 只能样式隐藏 / 必须下沉装配”三类 UI。
  - 抽共享文档骨架与排版 token。
  - 收敛 `edit` 模式的默认 chrome。
  - 实现编辑器按需加载、加载占位、并发切换保护。
  - 回归验证 autosave、标题导航、split、服务控制。
- 不在本阶段范围：
  - 表格列宽、callout 类型、折叠块等 sidecar 增强。
  - 表格、任务列表、复杂嵌套列表、frontmatter、HTML comment、原始 HTML 的完全同构编辑。
  - 重写 `split` 为 Notion-like 可视编辑模式。

### 2. 各阶段任务概要

- 阶段 A：探针与决策门。
  - 主要工作：记录当前包体基线，验证 `Crepe` 可控边界。
  - 预期成果：明确是否继续沿用 `Crepe`，还是应切到底层 `Milkdown` 自定义装配。
- 阶段 B：视觉同构最小闭环。
  - 主要工作：统一文档骨架、排版 token、块级样式和最小编辑 affordance。
  - 预期成果：`read -> edit` 不再明显切换成另一套产品。
- 阶段 C：按需加载与加载态。
  - 主要工作：把编辑器入口改成懒加载，并实现 skeleton / preload。
  - 预期成果：阅读首屏更轻，首次进入编辑态仍可接受。
- 阶段 D：并发与回归。
  - 主要工作：覆盖懒加载中的切模式、切文档、重启服务、恢复状态等中间态。
  - 预期成果：不出现空白编辑区、错误 dirty 状态或导航抖动。
- 阶段 E：收口。
  - 主要工作：跑全量测试、构建、浏览器实测并记录是否触发“下沉到 Milkdown”决策门。
  - 预期成果：形成可提交的代码与下一轮边界。

### 3. 整体结构框架

- 阅读真相层
  - `src/app/WorkspaceLayout.tsx`
  - `react-markdown + remark-gfm`
- 编辑器接入层
  - `src/editor/visual-markdown-editor.tsx`
  - `Milkdown / Crepe`
- 共享文档样式层
  - `src/index.css`
  - 后续建议拆出 `document-theme.css` 或等价 token 层
- 工作区与状态层
  - `src/App.tsx`
  - 自动保存、draft、flush、服务切换

第一版 `edit` 的稳定支持范围固定为：

- 正文
- 标题
- 列表
- 引用
- 代码块
- 链接

## 验收标准

- `edit` 第一版只承诺正文、标题、列表、引用、代码块、链接六类主链内容的高一致性。
- `read -> edit` 切换时，页面骨架、宽度、留白和主要块节奏不再明显跳变。
- 默认阅读路径不加载完整 `Milkdown / Crepe`。
- 编辑器首进加载有明确 skeleton，且无明显布局塌陷或主题闪烁。
- 标题导航继续只从 Markdown draft 提取，编辑中能稳定更新。
- 懒加载中切模式、切文档、切项目、重启服务不出现空白工作区、旧内容覆盖或错误 dirty 状态。
- 若 `Crepe` 无法收敛到可接受水平，执行计划能明确切换到底层 `Milkdown` 自定义装配，而不是继续无上限缝补。

量化口径先固定为：
- `index` 主 chunk 相较当前基线至少下降 `20%` 或至少下降 `150 KB raw`，满足其一即可视为达标
- 编辑器相关代码必须拆入独立 chunk，且体积可被单独记录
- 首次进入 `edit` 的额外等待时间目标为 `< 700ms`
- 若启用 hover 预热后仍稳定超过该阈值，则必须回到拆分策略或预热策略继续调整

## 决策门总结

### 决策门 A：是否继续沿用 `Crepe`

- 通过：进入视觉同构实施
- 不通过：切到底层 `Milkdown` 自定义装配

### 决策门 B：chrome 是否收敛成功

- 通过：进入按需加载阶段
- 不通过：中止继续 patch `Crepe` 默认壳层

### 决策门 C：按需加载是否真正带来收益

- 通过：主包达到量化下降目标，首次编辑等待时间落在阈值内
- 不通过：调整拆分策略或引入预热

## 复用边界

如果后续从 `Crepe` 切到底层 `Milkdown` 自定义装配，以下成果仍然默认复用：

- 共享文档骨架与排版 token
- 标题导航继续只从 Markdown draft 提取的规则
- `App` 层 autosave / draft / flush 状态机
- 编辑器按需加载骨架与 skeleton 思路

需要替换的仅是：

- 编辑器装配层
- 默认 chrome 控制方式
- 编辑态节点渲染映射方式

---

## 结论

这份计划把后续工作拆成了两条独立但衔接的主线：

1. 先做视觉同构最小闭环  
2. 再做编辑器按需加载与并发保护

这样可以避免一边改视觉、一边改加载路径导致回归面过大，也能在 `Crepe` 不够可控时及时止损，切到更底层、但更可控的 `Milkdown` 自定义装配。
