## 方案概述

### 1. 总体目标和范围

本方案用于解决 `md-reader` 当前在 `edit` 模式下的保存体验问题：**编辑过程经常被自动保存打断、`409` 冲突频发、保存失败后退出受阻**。本轮目标不是继续修补单个 `HTTP 409` 或单个切换拦截分支，而是重做编辑期保存模型，把现有“data-editor 风格自动保存”收敛为一套更适合 Markdown 可视编辑器的正式方案。

本方案覆盖以下范围：

- 重新定义 `edit` 模式下的保存状态机。
- 重新划分输入阶段、后台保存阶段、离开前 flush 阶段的职责边界。
- 明确 `mtime-only` 伪冲突、正文真实冲突、桥接短故障三类异常的处理路径。
- 明确 `read / edit / split` 三模式下的切换策略、退出策略和保存反馈。
- 明确前端状态、桥接接口、服务端冲突判定、UI 动作入口和实施顺序。

本方案不覆盖以下内容：

- 协同编辑、多人实时合并。
- sidecar 元数据泛化容器。
- 富文本 JSON 真相层替代 Markdown。
- 跨进程版本历史与 patch merge 引擎。

### 2. 各阶段任务概要

第一阶段：保存状态机重构

- 主要工作：把当前单一 `flushActiveDraft()` 驱动的保存逻辑拆成输入态、排队态、后台提交态、冲突态和失败态。
- 预期成果：编辑期间不再被保存动作频繁打断，保存状态不再只有 `idle / dirty / saving / error` 四档。
- 执行顺序：先定义状态机，再调整 UI 和交互。

第二阶段：冲突模型与恢复策略重构

- 主要工作：区分 `mtime-only` 伪冲突、正文真实冲突、桥接短故障，并为每类冲突设计独立恢复路径。
- 预期成果：不再把所有失败统一当成“保存失败后阻断切换”。
- 执行顺序：先扩展错误模型，再实现恢复动作。

第三阶段：桥接与服务端接口扩展

- 主要工作：让桥接保存接口返回结构化冲突信息，而不是只抛 `HTTP 409` 或普通字符串。
- 预期成果：前端可以根据冲突类型走不同交互。
- 执行顺序：先桥接层，再服务端判定细化。

第四阶段：模式切换与退出策略收敛

- 主要工作：为切 `read / edit / split`、切文档、切项目、重启服务定义统一离开流程。
- 预期成果：保存失败时不再只有“继续困住用户”这一条路。
- 执行顺序：在状态机和冲突模型稳定后接入。

第五阶段：浏览器回归与验收

- 主要工作：围绕连续输入、后台保存、冲突恢复、退出路径做端到端验证。
- 预期成果：确认“不打断输入、冲突可恢复、退出可预期”三条主目标成立。
- 执行顺序：所有代码路径收敛后统一回归。

### 3. 整体结构框架

本方案将编辑期保存链路拆成五层：

- 编辑器草稿层：维护当前正在输入的 `draft`，永远优先保证输入连续性。
- 前端保存调度层：决定什么时候排队保存、什么时候后台提交、什么时候离开前 flush。
- 桥接保存协议层：向本地服务提交保存请求，并返回结构化保存结果。
- 服务端冲突判定层：区分伪冲突和真实冲突，返回明确错误类型。
- 交互恢复层：给用户展示状态、失败原因和下一步动作。

本方案同时要求新增四个关键约束：

- 保存请求必须带有单调递增的 `saveRevision`，防止后台保存响应乱序覆盖新基线。
- 中文输入法 `composition` 期间禁止真正发起保存，只允许进入排队态。
- “不保存并退出”必须被明确定义为丢弃当前本地草稿，而不是模糊的临时离开。
- 伪冲突判定不能只依赖 `mtimeMs`，保存协议必须补 `expectedContentHash`。

## 一、当前问题的真实根因

### 1.1 当前模型来自 data-editor，不适合可视 Markdown 编辑

当前实现沿用的是 `data-editor` 风格自动保存基线：

- `debounce` 到时自动保存。
- 切模式、切文档、切项目、重启服务前先 `flush`。
- 保存失败时阻断切换。

这套模型对字段型或表格型编辑器成立，但对 Markdown 可视编辑器存在天然问题：

- 输入是连续流，不是离散字段提交。
- 编辑器内部包含选区、composition、块级结构变化和节点重排。
- 用户对“正在输入”和“正在保存”的感知应当完全不同。
- `mtime` 波动在文档场景里比表格型数据编辑更常见。

因此，当前问题不是“自动保存存在 bug”，而是**保存策略的抽象层级就不对**。

### 1.2 当前症状和根因的对应关系

症状一：编辑过程经常被保存打断

- 根因：输入期和提交期没有分层。
- 结果：用户刚停一小会儿就触发保存，请求返回又影响前端保存状态和后续切换。

症状二：`409` 冲突频发

- 根因：当前冲突判定只看 `mtimeMs`，没有区分“时间戳变了但正文没变”和“正文真的被改了”。
- 结果：大量本可自动恢复的伪冲突，被前端当成真实失败处理。

症状三：保存失败后不让退出编辑模式

- 根因：当前只有“保存失败则阻断切换”这一条原则，没有给失败态设计恢复出口。
- 结果：一旦进入失败态，用户只能继续困在编辑模式里。

## 二、正式目标

本轮正式目标收敛为三条：

1. **输入优先**：编辑期的首要原则是保证输入连续，不被保存动作打断。
2. **提交后台化**：正常自动保存应该成为后台行为，而不是前台交互事件。
3. **冲突可恢复**：不同类型的冲突必须有不同恢复路径，不能统一硬锁。

进一步展开为：

- 连续输入 20 到 30 秒期间，不应出现焦点丢失、内容跳动、重建编辑器或被强制阻断。
- `mtime-only` 伪冲突应该自动恢复，不需要用户参与。
- 正文真实冲突必须进入明确的决策态，而不是只抛报错。
- 短暂桥接失败应该允许继续编辑，并提供“重试 / 放弃本地草稿 / 稍后处理”的出口。

## 三、推荐的前端状态机

### 3.1 当前状态不足

当前保存状态只有：

- `idle`
- `dirty`
- `saving`
- `error`

这会把“正在输入”“后台保存中”“可重试失败”“冲突待决策”全压扁成几种粗粒度状态，导致 UI 和行为都只能用统一逻辑处理。

### 3.2 新状态机定义

建议改成以下状态：

- `clean`
  - 草稿与磁盘保存基线一致。
- `typing`
  - 用户正在连续输入，当前阶段不应主动抢保存。
- `save_queued`
  - 当前已有待保存草稿，但保存请求尚未发出。
- `saving_background`
  - 正在后台提交保存，不应打断继续输入。
- `save_failed_retryable`
  - 短故障失败，可自动或手动重试。
- `conflict_soft`
  - 轻冲突，优先自动恢复。
- `conflict_hard`
  - 正文真实冲突，必须用户决策。
- `leaving_with_pending_flush`
  - 用户尝试离开当前编辑上下文，系统正在做离开前保存判断。

### 3.2.1 请求级状态保护

除 UI 状态外，还必须显式维护请求级序号：

- `draftRevision`
  - 每次用户编辑后递增，用于表示当前草稿版本。
- `saveRevision`
  - 每次真正发起保存请求时递增，用于表示保存请求版本。
- `lastAckedSaveRevision`
  - 最近一次被确认写入成功的保存请求版本。

规则：

- 后台保存请求只允许推进 `lastAckedSaveRevision`，不能直接覆盖当前 `draftRevision`。
- 任意保存响应返回时，如果其 `saveRevision` 小于当前记录的最新已接受版本，则只记录日志，不更新保存基线。
- UI 上“已保存 / 待保存 / 保存中”必须基于 `draftRevision` 与 `lastAckedSaveRevision` 的关系判断，而不是只看某一个布尔态。

### 3.3 状态机约束

- `typing` 不等于 `dirty`；它表示“用户当前处于高优先级输入阶段”。
- `saving_background` 期间仍允许继续输入，不能把编辑器变成半锁定态。
- `save_failed_retryable` 不能直接等同于“禁止退出”。
- `conflict_hard` 必须拥有显式的动作弹层，而不是单纯状态文案。
- `composition` 期间不能真正发起保存请求；只允许刷新 `typing` 时间戳或进入 `save_queued`。
- 保存成功不能通过重置 editor value 的方式回写当前编辑器，否则会破坏选区、composition 和光标。

## 四、保存触发策略重构

### 4.1 轻量后台保存

适用于正常编辑流。

触发条件建议：

- 用户停止输入 `1200ms - 2000ms`。
- 编辑器失焦。
- 窗口失焦。
- 应用进入空闲时段。

行为要求：

- 进入 `save_queued`，随后转到 `saving_background`。
- 不抢焦点。
- 不切模式。
- 不阻断继续输入。
- 保存返回时只更新保存基线，不回写编辑器当前 draft。
- 当触发源是“编辑器失焦 / 窗口失焦”时，默认只提升为 `save_queued`，不立刻发起提交；真正提交仍由停顿阈值、空闲时段或离开前 flush 触发。

### 4.1.1 IME / composition 边界

必须补充以下规则：

- `compositionstart` 到 `compositionend` 期间，保存调度器不能发起后台保存。
- `composition` 期间如果计时器到期，只能记录“需要保存”，不能真正提交。
- `compositionend` 之后重新计算 debounce，确保保存拿到的是完整 Markdown 结果。
- 验收时必须覆盖中文输入法连续输入、候选词切换、输入中切模式等场景。

### 4.2 离开前 flush

适用于离开当前上下文的场景。

触发条件：

- `edit -> read`
- `edit -> split`
- 切文档
- 切项目
- 重启服务
- 关闭当前项目

行为要求：

- 如果当前 `draft` 已进入 `clean`，直接放行。
- 如果存在待保存内容，进入 `leaving_with_pending_flush` 并尝试提交。
- 如果提交成功，允许离开。
- 如果失败，则根据失败类型进入对应恢复流程。

### 4.3 强一致保护

适用于高风险离开场景：

- 服务关闭
- 页面卸载
- 真实正文冲突

这类场景不能静默吞掉失败，但也不能只有“继续锁死用户”这一条路。

### 4.4 `split` 模式的差异化策略

`split` 左侧承担的是源码精确修订，不应完全照搬 `edit` 的可视编辑节奏。

建议规则：

- `edit` 使用更积极的后台保存节奏，以保证可视编辑过程顺滑。
- `split` 共用同一套冲突模型和错误恢复模型，但保存触发阈值更保守，例如更长 debounce，且失焦默认只排队不提交。
- `split` 下发生真实冲突时，优先保留源码草稿，不自动尝试任何可视回写动作。

## 五、冲突与失败的分类处理

### 5.1 `mtime-only` 伪冲突

判定条件：

- 磁盘 `mtimeMs` 变化。
- 当前磁盘正文和上次加载基线正文一致。

处理策略：

- 自动重新读取磁盘版本。
- 更新本地 `lastSavedMtime`。
- 直接重试保存当前 draft。
- 整个过程不进入阻断态。

目标：

- 这类冲突对用户应当是无感的。

### 5.2 正文真实冲突

判定条件：

- 磁盘 `mtimeMs` 变化。
- 当前磁盘正文与本地保存基线正文不同。

处理策略：

- 进入 `conflict_hard`。
- 弹出明确冲突处理层。

必须提供的动作：

- `重新加载磁盘版本`
  - 放弃本地未保存草稿，以磁盘版本为准。
- `保留本地草稿`
  - 继续留在编辑态，稍后处理。
- `强制覆盖保存`
  - 用当前本地草稿覆盖磁盘版本。

目标：

- 真实正文冲突必须被显式暴露和显式决策。

### 5.3 桥接短故障失败

典型场景：

- bridge 忙
- 本地服务短暂不可达
- 写入瞬时失败

处理策略：

- 进入 `save_failed_retryable`。
- 保留当前本地 draft。
- 允许继续编辑。
- 提供：
  - `立即重试`
  - `继续编辑`
  - `不保存并退出`

目标：

- 短故障不应直接把用户锁死在当前模式。

### 5.4 “不保存并退出”的草稿归宿

本方案将该动作明确收敛为：

- “不保存并退出” = **显式丢弃当前本地草稿，并离开当前编辑上下文**。

不做的事情：

- 不在本轮引入独立的未保存草稿持久化仓库。
- 不在本轮把未保存草稿隐式恢复到下次打开。

原因：

- 当前目标是先解决编辑被保存打断和失败硬锁问题。
- 如果同时引入草稿仓库，会显著放大实现面和状态复杂度。
- 用户既然主动选择“不保存并退出”，就应被视为明确放弃当前草稿。

## 六、退出与模式切换策略

### 6.1 正常切换

在以下场景中：

- 切到 `read`
- 切到 `split`
- 切到其他文档
- 切到其他项目

应按以下流程处理：

1. 判断当前是否有未保存草稿。
2. 如果没有，直接切换。
3. 如果有，则执行离开前 flush。
4. 成功则切换，失败则进入对应失败/冲突态。

### 6.2 可重试失败时的退出策略

当保存失败属于 `retryable failure`，不应只有“继续卡住”。

应弹出三选项：

- `重试保存`
- `继续编辑`
- `不保存并退出`

这表示：

- 自动保存仍然是默认主路径。
- 但失败后的退出路径不能被完全剥夺。
- 其中“不保存并退出”是显式丢弃本地草稿，不做隐式恢复。

### 6.3 真实冲突时的退出策略

当保存失败属于 `content conflict`，应弹出三选项：

- `重新加载磁盘版本`
- `保留本地草稿`
- `强制覆盖保存`

这里的弹层语义和短故障弹层不同，不能复用成同一个“保存失败”弹层。

## 七、UI 与反馈方案

### 7.1 顶栏状态文案分层

建议状态区文案改成以下几类：

- `已保存`
- `输入中`
- `待保存`
- `后台保存中`
- `保存失败，可重试`
- `文档与磁盘版本冲突`
- `本地服务暂时不可用`

### 7.2 失败态动作入口

失败或冲突态不应只有一行文本，应提供明确动作按钮：

- `重试`
- `继续编辑`
- `不保存退出`
- `重新加载磁盘版本`
- `强制覆盖保存`

### 7.3 不应保留的旧反馈

以下反馈应淘汰：

- `保存失败：HTTP 409`
- 只显示错误，不给动作
- 所有失败都统一显示为“未保存”

### 7.4 错误类型到交互映射表

| 错误类型 | UI 状态 | 主文案 | 必备动作 |
| --- | --- | --- | --- |
| `mtime_conflict` | `conflict_soft` | 正在同步磁盘版本… | 无需用户操作，后台自动恢复 |
| `content_conflict` | `conflict_hard` | 文档与磁盘版本冲突 | `重新加载磁盘版本` / `保留本地草稿` / `强制覆盖保存` |
| `retryable` | `save_failed_retryable` | 保存失败，可重试 | `立即重试` / `继续编辑` / `不保存并退出` |
| `permission_denied` | `save_failed_retryable` | 没有写入权限 | `继续编辑` / `不保存并退出` |
| `bridge_offline` | `save_failed_retryable` | 本地服务暂时不可用 | `立即重试` / `继续编辑` / `不保存并退出` |

## 八、前端数据流重构建议

### 8.1 需要维护的核心前端状态

建议显式维护：

- `editorDraft`
  - 当前编辑器里真实显示并持续更新的草稿。
- `lastSavedContent`
  - 最近一次成功保存的正文。
- `lastLoadedDiskContent`
  - 最近一次从磁盘/bridge 读取到的正文。
- `lastSavedMtime`
  - 最近一次确认的保存基线时间戳。
- `saveIntent`
  - 当前保存的触发意图：`background` / `leave` / `force_overwrite`。
- `pendingLeaveIntent`
  - 当前是否有待完成的切换动作。
- `expectedContentHash`
  - 当前保存基线正文的内容哈希，用于服务端区分 `mtime-only` 和真实正文变化。
- `activeComposition`
  - 当前是否处于输入法组合输入阶段。

### 8.2 关键原则

- 编辑器中的 `draft` 是输入真相。
- 保存成功后只更新保存基线，不直接覆盖当前输入态。
- 后台保存返回不得导致编辑器 remount 或重置选区。

### 8.3 字段职责表

| 字段 | 含义 | 什么时候更新 | 参与什么比较 |
| --- | --- | --- | --- |
| `editorDraft` | 当前编辑器里实时内容 | 每次输入 | 与 `lastSavedContent` 比较是否待保存 |
| `lastSavedContent` | 最近一次成功写回磁盘的正文 | 保存成功后 | 与 `editorDraft` 比较是否 dirty |
| `lastLoadedDiskContent` | 最近一次从 bridge 读取到的磁盘正文 | 初次加载、冲突重读后 | 与 `lastSavedContent` 比较是否真实冲突 |
| `lastSavedMtime` | 最近一次确认的保存基线时间戳 | 保存成功、soft conflict 同步后 | 与服务端返回的 `mtimeMs` 对比 |
| `expectedContentHash` | `lastSavedContent` 的稳定哈希 | `lastSavedContent` 更新时同步计算 | 服务端 soft/hard conflict 判定 |
| `pendingLeaveIntent` | 当前想离开的动作 | 用户触发切模式/切文档/切项目时 | 决定保存失败后的后续流转 |

---

## 九、桥接接口与服务端建议

### 9.1 当前接口不足

当前桥接接口基本只返回：

- 成功时：文档正文和 `mtimeMs`
- 失败时：普通错误或 `409`

这不足以支持差异化恢复。

### 9.2 推荐保存结果模型

建议把桥接保存结果扩成结构化结果：

```ts
type SaveResult =
  | { ok: true; document: BridgeDocumentPayload }
  | { ok: false; kind: 'mtime_conflict'; latest: BridgeDocumentPayload }
  | { ok: false; kind: 'content_conflict'; latest: BridgeDocumentPayload }
  | { ok: false; kind: 'retryable'; message: string }
```

这样前端才能根据 `kind` 直接走不同分支，而不是继续解析字符串。

对应保存请求也要扩展：

```ts
type SaveRequest = {
  path: string
  content: string
  expectedMtimeMs: number | null
  expectedContentHash: string | null
  saveRevision: number
  intent: 'background' | 'leave' | 'force_overwrite'
}
```

其中：

- `expectedContentHash` 用于服务端判断伪冲突与真实冲突。
- `saveRevision` 用于客户端和 bridge 保持请求-响应对应关系，防止旧响应错误推进基线。
- `intent` 用于区分后台保存、离开前 flush 和强制覆盖路径。

### 9.3 服务端冲突判定建议

当前 `server/project-writer.mjs` 只基于 `mtimeMs` 判冲突。

建议扩成：

1. 读取当前磁盘正文和 `mtimeMs`。
2. 根据请求中的 `expectedContentHash` 比较当前磁盘正文是否等于客户端保存基线。
3. 如果 `mtime` 变了但正文哈希仍等于保存基线，返回 `mtime_conflict`。
4. 如果正文哈希也变了，返回 `content_conflict`。
5. 仅对真正无法完成写入的情况返回 `retryable` 或系统错误。

### 9.4 服务重启中的未完成保存

对于“后台保存还在路上，但用户点击重启服务”的场景，必须明确：

- 先把当前会话提升为 `leaving_with_pending_flush`。
- 如果存在进行中的 `saving_background`，不直接忽略其结果；但后续是否接受该结果，仍受 `saveRevision` 保护。
- 重启前必须再发起一次 `intent = 'leave'` 的保存判断，只有成功或用户明确放弃草稿后才允许继续。

## 十、实施顺序

### 阶段 1：状态机重构

主要改动：

- 扩展 `SaveState`
- 分离 `typing / save_queued / saving_background / conflict / retryable failure`
- 拆掉“所有场景共用一个 flush”
- 增加 `draftRevision / saveRevision / lastAckedSaveRevision`
- 增加 `activeComposition` 与 composition 保护

验收目标：

- 连续输入期间不再被保存动作打断。

### 阶段 2：桥接错误模型扩展

主要改动：

- `src/workspace/local-bridge-access.ts`
- `server/bridge-server.mjs`
- `server/project-writer.mjs`
- 保存请求补 `expectedContentHash / saveRevision / intent`

验收目标：

- 前端不再只收到 `HTTP 409`。

### 阶段 3：失败和冲突弹层

主要改动：

- 顶栏状态区
- 模式切换/文档切换离开流程
- 冲突动作弹层

验收目标：

- 保存失败或冲突时，用户始终有明确下一步动作。

### 阶段 4：浏览器回归

重点回归：

- 连续输入
- 顶部标题输入
- 中文输入法 composition 连续输入
- autosave 成功
- autosave 短失败
- `mtime-only` 伪冲突
- 正文真实冲突
- 切 `read / edit / split`
- 切文档 / 切项目 / 重启服务
- 保存请求乱序返回

### 阶段 5：证据化验收

验收不能只依赖主观“感觉顺滑”，还要补具体证据：

- 浏览器回归中确认编辑器 DOM 不因后台保存 remount。
- 顶部输入时确认选区和焦点稳定。
- 用模拟延迟接口确认旧的保存响应不会覆盖新的保存基线。
- 用真实冲突用例确认 `重新加载 / 保留 / 覆盖` 三动作都可走通。

---

## 十一、验收标准

本方案落地后，以下结论必须成立：

- 连续输入 20 到 30 秒期间，编辑器不失焦、不重建、不跳动。
- 中文输入法组合输入期间，不会在候选词未确认时触发真实保存请求。
- 普通停顿后的后台保存对用户基本无感。
- `mtime-only` 伪冲突自动恢复，无需用户参与。
- 正文真实冲突进入明确决策态，用户可以加载、保留或覆盖。
- 桥接短故障进入可重试失败态，用户可以重试、继续编辑或不保存退出。
- 切 `read / edit / split`、切文档、切项目时，不再被“单一失败态无出口”硬锁。
- 后台保存响应乱序返回时，不会错误覆盖新的保存基线。

---

## 十二、结论

`md-reader` 当前的保存问题，本质上不是单个 `409` 或单个 `flush` 分支错误，而是 **“字段型自动保存模型被直接搬进了可视 Markdown 编辑场景”**。正式修复方向不是继续补例外，而是把保存机制重构为：

- 输入优先
- 提交后台化
- 冲突分类处理
- 退出路径明确

只有按这个方向收敛，`edit` 模式才能同时满足：

- Notion-like 的顺滑编辑体验
- Markdown-first 的真相层约束
- 本地服务模式下的可靠写回
