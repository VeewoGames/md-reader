# md-reader 顶部标签页拖拽交互方案

## 方案概述

### 1. 总体目标和范围

本方案的目标是为 `md-reader` 顶部标签栏补齐“类似 Chrome 的左右拖动重排体验”，让用户可以直接拖动标签调整顺序，并在拖动过程中获得连续、平滑、可预期的让位动画与落位反馈。

本轮范围聚焦在“顶部标签栏内部重排”这一条主链路，不扩展到拖出标签栏生成新窗口、跨容器拖动、多行标签布局或移动端触摸专用交互。方案同时要求保持当前顶栏的单行约束、右侧固定操作区和紧凑工具栏密度，不为了拖拽交互破坏现有三段式布局。

此外，本轮必须把拖拽后的最终顺序写回现有会话恢复链路，而不是只做运行期视觉重排。当前恢复真相仍然依赖 `session.tabs` 与本地状态中的 `openDocumentPaths` 顺序，因此“拖完看起来换位了”不构成完成，只有“关闭/刷新后顺序仍然正确”才构成完成。

### 2. 各阶段任务概要

1. 现状收敛阶段  
   主要工作：确认当前 `TopBar`、`App`、`index.css`、现有测试和 `.claw` truth/ADR 中已经稳定的顶栏约束。  
   预期成果：得到“现有是静态 tab strip，而不是可重排 tab strip”的明确基线，并锁定不能破坏的布局规则。  
   执行顺序：先于所有交互设计与实现。

2. 交互设计阶段  
   主要工作：定义拖拽启动阈值、跟手表现、换位阈值、让位动画、松手落位、取消逻辑以及点击/关闭与拖拽的冲突处理。  
   预期成果：形成可实现的 Chrome-like 交互模型，而不是停留在“支持拖动”的泛描述。  
   执行顺序：在现状收敛后、实现前完成。

3. 技术拆分阶段  
   主要工作：明确 `App.tsx`、`TopBar.tsx`、`src/index.css`、本地状态和测试的职责边界，收敛状态模型与 DOM 测量策略。  
   预期成果：形成可直接进入编码的模块拆分与测试清单。  
   执行顺序：交互设计确认后执行。

4. 实现与验证阶段  
   主要工作：按模块落地拖拽状态、排序提交、动画样式与持久化，再补指针交互与顺序提交回归测试。  
   预期成果：顶部标签栏支持稳定的栏内拖拽重排，刷新后顺序可恢复。  
   执行顺序：方案确认后执行。

### 3. 整体结构框架

- 需求层：明确“Chrome 风格”在本仓库里的可落地定义。
- 交互层：定义拖拽生命周期、阈值、命中与动画。
- 状态层：定义会话标签顺序、临时拖拽顺序与最终提交顺序的关系。
- 视觉层：定义拖拽中悬浮标签、让位标签和收束动画的样式表达。
- 验证层：定义点击不误触、拖拽可重排、顺序可持久化的最小回归集合。

## 一、当前实现与证据链

### 1.1 当前顶栏只支持点击切换与关闭

当前标签栏入口位于 `src/app/TopBar.tsx`：

- `role="tablist"` 的容器在 `TopBar.tsx` 中通过 `tabs.map(...)` 顺序渲染。
- 单个标签只暴露了 `onTabSelect(tab.id)` 和 `onTabClose(tab.id)`。
- 当前不存在 `onTabReorder`、拖拽状态、指针监听、占位槽、临时排序或动画回流状态。

这说明当前模型仍是“数组顺序 = 渲染顺序 = 视觉顺序”，还没有进入可重排标签栏阶段。

### 1.2 当前样式有 hover/active 反馈，但没有位置动画

当前 `src/index.css` 中：

- `.topbar__tabstrip` 是单行 `flex` 容器；
- `.topbar__tab` 使用 `flex: 1 1 140px`、`max-width: 220px`；
- 仅存在 `background-color`、`border-color` 等过渡；
- 不存在拖拽中的 `transform`、`z-index`、`will-change`、占位元素或让位动画样式。

这意味着拖拽需求不能只靠 CSS 补几条过渡实现，必须补一层真实的拖拽状态和位置计算逻辑。

### 1.3 已有顶栏约束不能被拖拽需求破坏

当前仓库中已有两条稳定约束：

1. 顶栏必须保持单行，窄宽度下通过压缩标签宽度维持单行，不允许掉到第二行。
2. 顶栏是紧凑的三段式桌面工具栏，右侧固定操作区不能被拖拽标签挤坏。

因此，任何拖拽方案都必须在“单行 + 弹性宽度 + 固定右侧操作区”前提下成立。

## 二、需求拆解

### 2.1 明确需求

从“顶部标签页的左右拖动功能，类似 Chrome 的拖动效果和动画”可以提炼出以下明确需求：

1. 标签可以在顶部标签栏内左右拖动重排。
2. 拖拽中的标签需要跟手移动，而不是拖完后瞬间跳位。
3. 其他标签需要在拖拽过程中平滑让位，而不是最终一次性重排。
4. 松手后需要有自然的落位收束反馈，而不是立即硬切。
5. 点击切换、关闭按钮、活动标签样式和 dirty 状态点位不能因为拖拽而失效。

### 2.2 隐含需求

结合当前实现，实际还包含这些隐含需求：

1. 轻点标签仍然应该是“切换标签”，不能轻微抖动一下就进入拖拽。
2. 点击关闭按钮时不能误触拖拽。
3. 拖拽命中不能依赖固定宽度，因为当前 tab 是弹性压缩宽度。
4. 拖拽完成后的顺序应进入会话恢复链路，否则刷新后顺序丢失会破坏用户心智。
5. 不能破坏当前 `TopBar` 的无障碍角色语义和基础键盘切换能力。

### 2.3 非目标

本轮默认不纳入以下需求：

1. 把标签拖出标签栏生成新窗口。
2. 拖到标签栏外触发分离、关闭或跨容器移动。
3. 多行标签布局或 overflow dropdown。
4. 触屏手势专门优化。
5. 跨 profile、跨项目或跨浏览器窗口拖放。

## 三、关键决策与推荐路线

### 3.1 需要决策的点

当前真正需要拍板的不是“要不要能拖”，而是第一版的交互边界：

- 方案 A：只支持标签栏内部左右重排。
- 方案 B：在 A 的基础上，再预留拖出标签栏的语义。

### 3.2 推荐方案

推荐优先采用方案 A：只支持标签栏内部左右重排。

理由如下：

1. 它已经完整覆盖“左右拖动 + 类 Chrome 动画”的核心诉求。
2. 它与当前仓库结构最匹配，只需在 `App.tsx`、`TopBar.tsx`、`src/index.css` 和本地状态链路内完成闭环。
3. 一旦引入“拖出标签栏”，问题会立刻升级为窗口管理、外部释放区、取消/回弹策略和跨容器语义，复杂度与风险显著扩大。
4. 当前顶栏的单行和右侧固定操作区约束，决定了栏内重排已经足够形成一个独立、完整且可验证的第一版目标。

### 3.3 当前决策状态

用户已确认：第一版只支持标签栏内部左右重排，不做拖出标签栏。

因此，本方案的正式边界已经固定为“仅标签栏内部重排”；若后续需要拖出语义，应单独立项，不与本轮混合。

## 四、推荐交互设计

### 4.1 拖拽生命周期

建议将交互生命周期拆成四段：

1. `press_pending`  
   用户按下标签主体，但尚未超过拖拽阈值，仍保留点击切换的可能。

2. `dragging`  
   指针横向移动超过阈值后，进入拖拽态；当前标签悬浮并跟手移动，其他标签进入可让位状态。

3. `settling`  
   指针释放后，拖拽标签回到最终槽位，播放一段短收束动画。

4. `idle`  
   动画结束后回到静止态。

### 4.2 拖拽启动阈值

建议加入一个小的水平位移阈值，例如 `6px ~ 8px`。

目的不是增加迟滞感，而是解决两个冲突：

1. 用户点击标签切换时不应误进入拖拽。
2. 用户点击关闭按钮时不应被拖拽逻辑截获。

拖拽起点也需要进一步收紧：第一版只允许从标签主体 `.topbar__tab-button` 启动拖拽，不允许从关闭按钮区域启动。当前关闭按钮 `.topbar__tab-close` 是与标签主体并列的独立 sibling，而不是嵌套在 tab button 内，因此实现阶段必须显式排除关闭按钮的 `pointerdown`，必要时对其执行 `stopPropagation()`，避免“想关标签却把标签拖走”。

### 4.3 被拖标签的表现

被拖标签建议采用“原标签提升为浮层”的方案，而不是额外生成半透明 ghost：

1. 进入拖拽后抬高 `z-index`；
2. 使用 `transform: translateX(...)` 跟随指针；
3. 保留当前 tab 的真实宽度、内容、dirty 状态点和关闭按钮；
4. 增加轻微阴影和更高对比度，形成“被提起”的感觉。

这样更接近 Chrome 的视觉感受，也避免 ghost 与真实元素双份存在带来的同步复杂度。

这里还需要锁定一个实现约束：第一版不建议把被拖标签完全脱离文档流后再依赖真实 reflow 驱动其他标签位置变化。当前 tab 是弹性 `flex` 项，且 tabstrip 本身有 `overflow: hidden`；如果被拖标签直接脱离流而没有占位，其他标签会先发生真实回流，再叠加让位动画，容易导致缓存 rect 失真和抖动。

因此第一版建议采用以下二选一策略，并优先选择 A：

1. 方案 A：原标签仍留在流内，只通过 `transform: translateX(...)` 跟手移动，其他标签也只通过 `transform` 让位。
2. 方案 B：若必须让被拖标签脱离流，则必须同步插入一个与初始宽度一致的 placeholder，并锁定拖拽开始时的宽度。

默认推荐方案 A，因为它与当前弹性单行标签栏更一致，状态和测量复杂度也更低。

### 4.4 其他标签的让位逻辑

建议按“中线越过”策略决定换位：

1. 拖拽开始时测量所有标签的 `DOMRect`；
2. 根据当前拖拽标签中心点与其他标签中心点的相对位置判断是否穿越；
3. 一旦穿越相邻标签中线，就更新临时排序；
4. 其他标签根据临时排序结果做 `transform` 让位动画。

选择中线而不是边缘命中，是为了避免在弹性宽度标签上出现过早换位或频繁抖动。

### 4.5 松手与取消

建议：

1. 在标签栏内松手时，提交最终顺序并播放落位动画；
2. 若拖拽过程中指针短暂离开标签栏，只要未触发“拖出标签栏”语义，仍按栏内重排处理；
3. 若窗口失焦、`pointercancel` 或标签集合变化，终止拖拽并回到最近一次临时顺序或初始顺序。

另外，第一版实现必须使用 pointer capture。具体来说，进入拖拽态后需要对触发拖拽的元素执行 `setPointerCapture(pointerId)`，在 `pointerup` / `pointercancel` 或拖拽终止时执行 `releasePointerCapture(pointerId)`。否则在当前紧凑 tab strip 中快速横向拖动时，容易在光标短暂离开元素边界后丢失后续事件，导致拖拽中断或松手不落位。

## 五、技术设计

### 5.1 模块职责

建议职责拆分如下：

- `src/App.tsx`
  - 持有会话级 `tabs` 顺序。
  - 提供 `onTabReorder(nextOrderedTabIds)` 或等价的最小 reorder action。
  - 负责把最终顺序同时写回 `session.tabs` 与本地状态持久化链路。

- `src/app/TopBar.tsx`
  - 管理 pointer drag 生命周期。
  - 采集标签 DOMRect。
  - 维护临时拖拽顺序、拖拽偏移量和拖拽态样式开关。
  - 在 drop 时调用 `onTabReorder(...)`。

- `src/index.css`
  - 定义 dragging、shifting、settling 的样式与过渡。
  - 保证单行布局不被拖拽样式打散。

### 5.2 推荐状态模型

建议在 `TopBar` 内维护最小局部状态：

- `dragState: 'idle' | 'press_pending' | 'dragging' | 'settling'`
- `dragTabId: string | null`
- `pressPointerId: number | null`
- `pressStartX: number`
- `dragOffsetX: number`
- `previewOrder: string[] | null`
- `tabRectsById: Record<string, { left: number; width: number; center: number }>`

其中：

- `tabs` 真正来源仍然是 `App.tsx`；
- `previewOrder` 只服务于拖拽中的临时视觉顺序；
- drop 完成后再由 `App.tsx` 提交真实顺序。

### 5.3 顺序提交与持久化

当前 `App.tsx` 的会话状态已经以 `session.tabs` 作为标签集合真相层，因此拖拽完成后应直接重排 `session.tabs`。

如果只改 `TopBar` 渲染顺序而不改 `session.tabs`，会出现两个问题：

1. 关闭当前标签后的 fallback 顺序和视觉顺序不一致；
2. 页面刷新恢复时又回到旧顺序。

因此本轮不建议只做“视觉假排序”，而应让最终顺序进入会话状态和本地恢复链路。

这里要进一步明确：当前恢复链路并不是独立的 `tabOrder` 字段，而是本地状态中的 `openDocumentPaths` 顺序。因此 drop 完成后，除了重排 `session.tabs`，还必须同步更新：

1. `openDocumentPaths: session.tabs.map((tab) => tab.documentPath)` 的保存顺序；
2. `activeDocumentPath` 与 `activeTabId` 的一致性；
3. 后续 `closeTabInternal()` 使用的 fallback 顺序语义。

只有这样，关闭标签、刷新恢复和后续切换链路才会共享同一套顺序真相。

### 5.4 DOM 测量策略

由于当前 tab 宽度是弹性的，排序命中不能使用固定宽度估算。建议：

1. 在进入拖拽态时一次性读取所有标签 rect；
2. 拖拽中只根据 `dragOffsetX + 初始中心点` 推导当前拖拽中心；
3. 当临时顺序变化后再刷新一次基准 rect 或基于缓存宽度重算槽位；
4. 避免在每次 `pointermove` 都读取布局，减少 layout thrash。

如果第一版采用前文推荐的方案 A，即“原标签留在流内，只用 transform 跟手”，那么 rect 缓存会更稳定，也更容易把测量频率压到“进入拖拽时一次 + 临时顺序变化时必要刷新”。

### 5.5 动画策略

建议统一使用 `transform` 驱动动画，不直接动画 `left` 或 `width`：

1. 被拖标签：`transform: translateX(...)`
2. 被让位标签：根据临时索引差值做 `transform`
3. 松手收束：保留一次短 `transform` 过渡回到 `0`

这样能最大程度减少重排和闪烁，也更容易得到接近 Chrome 的平滑感。

## 六、验证与测试建议

### 6.1 最小测试集

建议至少补以下回归：

1. 轻点标签不会触发拖拽，只触发 `onTabSelect`。
2. 从标签主体开始拖动，超过阈值后进入拖拽态。
3. 拖拽越过相邻标签中线时，临时顺序发生变化。
4. 松手后提交最终顺序到上层。
5. 点击关闭按钮不会触发拖拽。
6. 拖拽结束后，被拖标签仍保有正确焦点，`aria-selected` 不丢失。
7. 重排后的顺序会写回本地状态，刷新恢复后仍保持一致。

### 6.2 更高价值的补充测试

若时间允许，建议继续补：

1. 窄宽度下压缩标签仍可重排。
2. dirty 状态点和 active 样式在拖拽中不丢失。
3. pointer capture 下快速横向拖动不会丢事件。

### 6.3 无障碍与焦点回归

当前 tab 主体仍然是 `role="tab"` 的 button，因此实现阶段不能只看鼠标效果，还要锁定拖拽结束后的焦点归属：

1. drop 后焦点应回到被拖动的标签主体；
2. 当前活动标签的 `aria-selected` 必须保持正确；
3. 原有键盘切换能力不能因为拖拽重排回退。

如果 drop 通过重排数组触发了重新渲染，上述三点必须纳入组件测试，而不是依赖人工观察。

## 七、实施拆分建议

### Task 1：补齐会话级标签重排能力

- 在 `App.tsx` 增加最小 reorder action。
- 同步更新 `openDocumentPaths` 恢复顺序，而不是只改内存态。

### Task 2：在 `TopBar.tsx` 增加拖拽状态机

- 新增 `onTabReorder` props。
- 接入 `pointerdown / pointermove / pointerup / pointercancel`。
- 实现阈值、pointer capture、测量、临时顺序与 drop。
- 限定只有标签主体可以启动拖拽，关闭按钮显式排除。

### Task 3：补齐拖拽样式与动画

- 新增 dragging / shifting / settling 样式。
- 保持单行与右侧操作区稳定。

### Task 4：完成测试与实机验证

- 补 `TopBar` 或 `AppShell` 层回归测试。
- 至少验证点击、拖拽重排、焦点/aria 保持、最终顺序提交和刷新恢复。

## 八、结论

这轮需求的本质不是“给标签加一个拖拽属性”，而是把当前静态标签栏升级成一个具备：

1. 栏内横向重排，
2. 跟手悬浮，
3. 中线换位，
4. 平滑让位，
5. 落位收束，
6. 顺序持久化

的完整拖拽交互系统。

推荐路线是先收敛到“仅标签栏内部重排”的第一版。这条路线已经足够覆盖用户所说的“类似 Chrome 的拖动效果和动画”，同时又不把问题过早扩展到窗口管理和跨容器拖动。
