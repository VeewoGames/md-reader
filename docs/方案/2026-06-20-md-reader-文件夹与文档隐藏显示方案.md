# md-reader 文件夹与文档隐藏显示方案

## 方案概述

### 1. 总体目标和范围

本方案用于为 `md-reader` 增加文件夹与文档的隐藏能力，并补齐对应的全局显示控制。目标是在不破坏现有文件树、搜索、标签页与 profile 状态模型的前提下，让用户可以：

- 在左侧文件树中按节点隐藏文件夹或单独文档
- 在右上角切换“隐藏项不可见 / 隐藏项可见”两种浏览视图
- 在“隐藏项可见”状态下继续对隐藏节点执行解除隐藏操作
- 让隐藏规则在当前项目 profile 中持久保存

本方案包含：

- 文件树节点 hover 操作位设计
- 目录与文档隐藏规则
- 顶栏全局显示隐藏项按钮设计
- profile 级隐藏状态持久化
- 搜索、当前文档、标签页与隐藏规则的联动约束

本方案不包含：

- 批量隐藏/批量恢复
- 多选节点操作
- 独立的“隐藏项管理面板”
- 按扩展名、标签或规则自动隐藏
- sidecar 或文档内容级隐藏

### 2. 各阶段任务概要

阶段一：状态模型与可见性规则收敛

- 主要工作：定义隐藏路径集合、可见性判断规则、全局显示隐藏项开关以及父子节点继承语义
- 预期成果：前端和持久化层都能统一判断一个节点是否显式隐藏、是否因祖先隐藏而不可见、以及当前视图是否应该展示它
- 执行顺序：先收敛数据语义，再进入组件改造

阶段二：文件树与顶栏交互接入

- 主要工作：在文件树节点 hover 状态下增加隐藏/显示按钮，在顶栏锁定编辑按钮右侧加入全局显示隐藏项按钮，并补足弱化样式
- 预期成果：用户可以直接在树中隐藏节点，并在“隐藏项可见”视图中解除隐藏
- 执行顺序：先改顶栏和文件树交互入口，再联通状态透传

阶段三：联动规则与验证补齐

- 主要工作：收敛搜索、标签页、当前文档、profile 恢复的联动行为，并补齐测试
- 预期成果：隐藏功能在刷新、切 profile、搜索和打开文档等路径上行为一致
- 执行顺序：功能接通后统一补测试和回归验证

### 3. 整体结构框架

本功能建议按四层组织：

- `workspace/profile`：持久化隐藏路径集合
- `workspace/local-state`：承载本次会话中的“显示隐藏项”临时开关
- `workspace/file-tree`：生成原始树结构，并基于原始树派生渲染树与可见性元数据
- `app/topbar`：承载全局“显示隐藏项”按钮
- `app/workspace-layout`：承载节点 hover 操作、节点弱化样式和可见性过滤结果

## 需求目标

用户最终应能完成以下主流程：

1. 鼠标移入左侧文件夹节点，在最右侧看到隐藏按钮
2. 点击后隐藏该文件夹及其所有子节点
3. 鼠标移入左侧文档节点，在最右侧看到隐藏按钮
4. 点击后只隐藏该文档
5. 在右上角点击“显示隐藏项”按钮，切换到隐藏项可见状态
6. 在隐藏项可见状态下看到弱化显示的隐藏节点
7. 鼠标移入已隐藏节点时，原先的隐藏按钮切换为“显示 / 解除隐藏”按钮
8. 点击后解除该节点的显式隐藏状态

## 关键决策

### 1. 隐藏路径采用 profile 级持久化，显示隐藏项采用会话级临时状态

`hiddenPaths` 应归属于项目 profile，`showHiddenItems` 应归属于当前会话 local state。

理由：

- 当前 `profile.navigation` 已承载文件树展开状态等长期导航偏好
- 隐藏文件树节点更接近“个人工作区整理偏好”，不是瞬时会话状态
- 当前 bridge 已支持整份 `WorkspaceProfile` 读写，新增隐藏字段可以沿用现有链路
- “显示隐藏项”更像临时浏览模式，刷新后应回到默认隐藏视图
- 用户已经明确要求 `showHiddenItems` 只做临时切换，刷新后恢复默认隐藏状态

建议结构：

```ts
navigation: {
  expandedFileNodes: string[]
  expandedFileNodesInitialized?: boolean
  expandedHeadingNodes: Record<string, string[]>
  hiddenPaths?: string[]
}
```

```ts
localState: {
  showHiddenItems?: boolean
}
```

### 2. 全局按钮语义是“显示隐藏项”，不是“执行隐藏”

右上角新增按钮的职责是切换当前文件树视图是否展示隐藏项，而不是直接修改某个节点的隐藏状态。

建议定义：

- 默认：`showHiddenItems = false`
- 切换后：`showHiddenItems = true`
- 刷新后：恢复为 `showHiddenItems = false`

这样可以保证：

- 顶栏按钮负责视图模式
- 节点 hover 按钮负责节点状态变更
- 两层职责不冲突
- 刷新后不会把“临时查看隐藏项”误当成长期偏好保留下来

### 3. 隐藏路径集合只记录显式隐藏，不记录派生隐藏

建议 `hiddenPaths` 只保存用户直接点击隐藏过的路径，不额外展开写入所有后代路径。

理由：

- 文件夹隐藏后，子节点的隐藏是祖先规则派生出来的，不应冗余写入
- 后续恢复父目录时，可以自然恢复那些没有被单独显式隐藏的后代节点
- 数据更小，逻辑更容易解释

### 4. 父隐藏优先级高于子可见

若某个文件夹路径在 `hiddenPaths` 中，则其所有后代节点都视为隐藏。

规则：

- 父目录显式隐藏时，子目录和子文档都不可见
- 若子文档本身也在 `hiddenPaths` 中，则它保留自己的显式隐藏标记
- 父目录解除隐藏后，那些没有单独显式隐藏的子项恢复可见
- 父目录解除隐藏后，那些曾单独显式隐藏过的子项仍保持隐藏

### 5. “隐藏项可见”状态下，hover 按钮切换为“显示 / 解除隐藏”

这是本轮已确认的交互规则，必须显式固化：

- 默认视图下，未隐藏节点 hover 时显示“隐藏”按钮
- `showHiddenItems = true` 时，已隐藏节点重新显示，但使用弱化样式
- 已隐藏节点 hover 时，右侧按钮从“隐藏”切换为“显示 / 解除隐藏”
- 未隐藏节点在该模式下 hover 时仍显示“隐藏”按钮

### 6. 原始树与渲染树必须分离

隐藏规则不能直接破坏传入工作区的原始文件树结构，否则会连带污染展开状态恢复。

规则：

- 原始树保留完整节点集合，用于目录存在性判断、展开状态恢复和祖先链计算
- 渲染树基于原始树和 `showHiddenItems` 派生
- 搜索输入边界基于渲染树，而不是直接覆盖原始树

这样可以避免：

- 隐藏目录后把对应路径从“可展开目录集合”里删掉
- 再次显示隐藏项时，原先保存的展开状态丢失

## 交互规则

### 1. 文件夹节点

- hover 时，在节点最右侧显示操作按钮
- 未隐藏时按钮文案/语义为“隐藏”
- 已隐藏且当前为隐藏项可见模式时，按钮文案/语义为“显示”或“解除隐藏”
- 点击按钮时不得触发展开/折叠
- 点击目录主体仍保持现有展开/折叠行为

### 2. 文档节点

- hover 时，在节点最右侧显示操作按钮
- 未隐藏时按钮文案/语义为“隐藏”
- 已隐藏且当前为隐藏项可见模式时，按钮文案/语义为“显示”或“解除隐藏”
- 点击按钮时不得触发打开文档
- 点击文档主体仍保持现有打开文档行为

### 3. 顶栏全局按钮

- 在 `regular` 模式下放置在右上角锁定编辑按钮右侧
- 在 `split` 模式下也必须常驻可见，位置保持在模式切换按钮左侧的固定区位，不因锁按钮缺席而消失
- 图标语义建议使用“眼睛 / 眼睛关闭”或等价可见性图标
- 按钮切换的是“是否显示隐藏项”
- 当 `showHiddenItems = false` 时，文件树中完全不渲染隐藏节点
- 当 `showHiddenItems = true` 时，文件树渲染隐藏节点，但增加弱化样式

### 4. 隐藏节点视觉状态

隐藏节点在“隐藏项可见”模式下建议采用以下视觉处理：

- 降低文字不透明度
- 图标颜色降低饱和度
- hover 时恢复足够对比度以保证可操作性
- 当前节点如果既是隐藏节点又是当前打开文档，需要保留当前页语义，但整体仍应看出它处于隐藏状态

## 联动规则

### 1. 搜索框

- `showHiddenItems = false` 时，搜索结果只包含可见节点
- `showHiddenItems = true` 时，搜索结果包含隐藏节点
- 搜到的隐藏节点在列表中同样应带弱化样式

### 2. 当前打开文档与标签页

- 当前已打开文档被隐藏时，不强制关闭标签页
- 当前文档在默认视图下可以从左侧文件树消失，但中间工作区继续保持打开
- 已打开标签不因为隐藏规则自动关闭
- 如果用户在“隐藏项可见”模式下查看当前隐藏文档，它在文件树中仍应可以被定位和解除隐藏

### 3. 文件树展开状态

- 隐藏/解除隐藏不应破坏当前目录展开状态
- 当父目录解除隐藏且其子目录此前处于已展开状态时，应尽量保留该展开状态

### 4. profile 切换与刷新恢复

- 隐藏路径集合随 profile 切换而变化
- `showHiddenItems` 属于当前会话临时状态，不写入 profile
- 刷新后应恢复到默认隐藏视图，即 `showHiddenItems = false`
- 切 profile 时建议同样回到默认隐藏视图，避免把一个 profile 下的临时浏览模式带入另一个 profile

## 数据与实现方案

### 1. 状态层

建议在 `WorkspaceProfile.navigation` 中新增：

- `hiddenPaths: string[]`

对应影响文件：

- `src/workspace/profile-store.ts`
- `src/workspace/local-bridge-access.ts`
- `src/App.tsx`

建议在会话级本地状态中新增：

- `showHiddenItems: boolean`

对应影响文件：

- `src/workspace/local-state.ts`
- `src/App.tsx`

### 2. 可见性判断

建议新增统一判断工具，而不是把逻辑散落在组件内。

核心判断至少需要：

- 节点是否显式隐藏：`hiddenPaths.includes(path)`
- 节点是否因祖先隐藏：任一祖先路径命中 `hiddenPaths`
- 节点是否在当前视图可见：
  - `showHiddenItems = true` 时始终可见
  - `showHiddenItems = false` 时，仅在未被隐藏时可见

建议工具同时输出可见性元数据，而不是只返回布尔值，例如：

- `isExplicitlyHidden`
- `isHiddenByAncestor`
- `isVisibleInCurrentMode`

建议放置位置：

- `src/workspace/file-tree.ts`
- 或新增 `src/workspace/file-tree-visibility.ts`

同时建议保留两份树：

- `sourceFileTree`：原始完整树
- `visibleFileTree`：根据 `showHiddenItems` 和搜索条件派生的渲染树

### 3. 组件透传链路

当前透传链路为：

- `App.tsx`
- `AppShell.tsx`
- `TopBar.tsx`
- `WorkspaceLayout.tsx`

建议新增透传内容：

- `hiddenPaths`
- `showHiddenItems`
- `onToggleShowHiddenItems`
- `onHidePath`
- `onUnhidePath`

并明确：

- `hiddenPaths` 来自 profile
- `showHiddenItems` 来自 local state

### 4. 文件树节点渲染

`WorkspaceLayout.tsx` 当前只渲染目录按钮和文档按钮，缺少节点右侧 action 区。

建议改造方向：

- 为目录节点和文档节点增加一个包裹层，使主体点击区和右侧操作区分离
- 右侧操作区只在 hover 或键盘 focus 状态下显示
- 操作按钮点击时使用 `event.stopPropagation()`，避免误触发目录展开或文档打开

### 5. 顶栏按钮区

`TopBar.tsx` 当前锁定编辑按钮位于 `topbar__mode-cluster` 内。

建议：

- 将“显示隐藏项”按钮放在锁定按钮右侧、模式切换按钮左侧
- 复用 `topbar__lock-toggle` 一类的圆角按钮样式，单独增加状态色或 pressed 样式

## 风险与注意事项

### 1. 目录节点点击热区冲突

当前目录按钮本身承担展开/折叠职责。若直接把隐藏按钮塞进同一个按钮内部，容易产生事件冒泡冲突。

应对策略：

- 必须把主体点击区和右侧操作区拆开
- 操作按钮显式阻止冒泡

### 2. 搜索与隐藏规则的顺序

如果先搜索再过滤隐藏，和先过滤隐藏再搜索，在“显示隐藏项”开关下的结果语义会不同。

应对策略：

- 明确以“当前视图可见性”作为搜索输入边界
- 只有 `showHiddenItems = true` 时才把隐藏节点纳入搜索结果

### 3. 原始树被覆盖导致展开状态丢失

当前工作区会根据传入树重新计算可用目录集合；若直接把隐藏过滤后的树当成唯一输入，会导致隐藏目录对应的展开状态被清理掉。

应对策略：

- 保留原始完整树作为展开状态与路径存在性判断依据
- 渲染阶段再从原始树派生当前可见树

### 4. 当前文档消失导致的认知断裂

用户可能在正文中阅读某文档，同时把它隐藏。

应对策略：

- 不强制关闭标签页
- 保持正文与标签页稳定
- 只改变文件树的可见性

## 建议执行顺序

1. 先扩展 profile hiddenPaths 与 local-state showHiddenItems
2. 再拆分原始树与渲染树，并抽出统一可见性判断
3. 然后改顶栏按钮与文件树 hover 交互
4. 最后补搜索、恢复与测试
