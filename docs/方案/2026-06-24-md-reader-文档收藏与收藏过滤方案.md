# md-reader 文档收藏与收藏过滤方案

## 方案概述

### 1. 总体目标和范围

本方案用于为 `md-reader` 增加“文档收藏”和“只看收藏文档”两项能力，目标是在不改动 Markdown 正文、不引入新的服务端索引链路、并保持现有文件树/搜索/profile 状态模型稳定的前提下，让用户可以：

- 在左侧文件树的文档行上对单个文档执行收藏/取消收藏
- 在左侧搜索框左侧通过收藏开关切换到“只看收藏文档”视图
- 在“显示隐藏项 + 只看收藏”组合模式下稳定查看“已收藏且已隐藏”的文档
- 让收藏状态随当前项目 profile 持久保存
- 让“只看收藏”作为当前会话下的临时浏览模式，与现有搜索框协同工作

本方案包含：

- 文档行收藏按钮设计
- 收藏状态的数据归属与持久化边界
- 搜索框左侧收藏过滤开关设计
- 收藏过滤与隐藏规则、搜索规则、展开状态的联动约束
- 实施阶段划分与验收标准

本方案不包含：

- 文件夹收藏
- 批量收藏/批量取消收藏
- 独立“收藏管理面板”
- 跨项目收藏聚合
- 按标签或内容规则自动收藏
- 修改 Markdown 正文或 sidecar 文档内容

### 2. 各阶段任务概要

阶段一：状态模型与边界收敛

- 主要工作：定义收藏对象、`favoritePaths` 持久化位置、`showFavoritesOnly` 过滤开关归属，以及与 `hiddenPaths`、搜索框的优先级关系
- 预期成果：前端和持久化层都能统一解释“某个文档是否已收藏”“当前是否只看收藏”
- 执行顺序：先定状态边界，再改 UI

阶段二：文件树与过滤链路接入

- 主要工作：在文件树文档行增加收藏按钮，在文件树过滤链路叠加收藏筛选
- 预期成果：用户可以收藏文档，并在左侧只显示收藏文档，同时保留目录上下文
- 执行顺序：先节点级交互，再接过滤

阶段三：顶层交互与联动验证

- 主要工作：在搜索框左侧加入收藏过滤开关，并补齐与隐藏、搜索、profile 恢复、当前文档等联动验证
- 预期成果：收藏功能在常规工作区路径上行为一致，不破坏现有隐藏和搜索能力
- 执行顺序：功能接通后统一回归

### 3. 整体结构框架

本功能建议按五层组织：

- `workspace/profile`
  - 持久化 `favoritePaths`
- `workspace/file-tree`
  - 基于原始树派生“当前可见树”，叠加隐藏和收藏过滤
- `app/App`
  - 恢复 profile、保存收藏、管理 `showFavoritesOnly` 会话态
- `app/WorkspaceLayout`
  - 承载搜索框左侧收藏开关、文档行收藏按钮和文件树过滤结果
- `app/TopBar`
  - 不承载本次收藏过滤入口，避免把文件树局部过滤塞到全局顶栏

## 需求目标

用户最终应能完成以下主流程：

1. 在左侧文件树中，将鼠标移到某个文档行
2. 在当前“隐藏”按钮左侧看到一个星形收藏按钮
3. 点击后将该文档收藏，再次点击可取消收藏
4. 在左侧搜索框左侧看到一个收藏过滤开关按钮
5. 点击后切换到“只看收藏文档”模式
6. 在该模式下，文件树仅显示收藏文档及其必要目录上下文
7. 若同时打开“显示隐藏项”，则可以查看“已收藏且已隐藏”的文档，但它们继续保持隐藏态弱化样式
8. 继续输入搜索词时，只在当前“可参与显示的收藏集合”内做过滤
9. 关闭收藏过滤开关后，恢复普通文件树视图

## 关键决策

### 1. 收藏对象只支持文档，不支持文件夹

本轮收藏能力应限定为“文档收藏”，不支持目录收藏。

理由：

- 用户给出的入口位置是“当前文档隐藏按钮左侧”，目标对象已经收窄到文档行
- “只显示收藏的文档”说明过滤目标是文档集合，而不是任意树节点
- 目录收藏会把语义从“个人快捷文档集合”扩大成“树节点书签系统”，复杂度明显上升
- 目录收藏与目录隐藏、目录展开状态会产生更多优先级冲突，不适合本轮一起做

### 2. 收藏状态应持久化到 `profile.navigation.favoritePaths`

收藏状态更接近个人长期导航偏好，应与 `hiddenPaths` 一样归属到当前项目 profile。

建议结构：

```ts
navigation: {
  expandedFileNodes: string[]
  expandedFileNodesInitialized?: boolean
  expandedHeadingNodes: Record<string, string[]>
  hiddenPaths: string[]
  favoritePaths: string[]
}
```

理由：

- 当前 `profile.navigation` 已承载展开状态与隐藏路径，收藏同样属于“个人文件浏览偏好”
- 收藏不属于 Markdown 文本内容真相，不应写进正文
- 收藏也不是瞬时 UI 反馈，切 profile 时应自然隔离
- 现有 profile 读写链路已经打通，新增字段可以复用现有机制

### 3. “只看收藏”开关应是会话态，不跨刷新持久化

`showFavoritesOnly` 建议只存在于 `App` 的 React 会话态，不写入 profile，也不写入 local-state store。

理由：

- 它和现有 `showHiddenItems` 一样，属于临时浏览模式，而不是长期偏好
- 刷新后回到普通视图更符合预期，避免用户误以为文件丢失
- 切 profile 时也应回到普通视图，避免把某个 profile 下的临时过滤状态带到另一个 profile

建议默认值：

- 初始：`showFavoritesOnly = false`
- 切 project/profile 后：重置为 `false`
- 刷新后：恢复为 `false`

### 4. 必须正式支持“显示隐藏项 + 只看收藏”组合模式

本轮应把这组组合模式作为正式能力定义清楚，而不是留作未来考虑。

组合规则建议固定为：

1. 原始树
2. 隐藏可见性模式
3. 收藏过滤
4. 搜索过滤

具体语义：

- `showHiddenItems = false` 时，隐藏节点先被排除；随后“只看收藏”只在剩余可见文档里生效
- `showHiddenItems = true` 时，隐藏节点重新进入可见候选集合；随后“只看收藏”再筛出收藏文档
- 因此，“已收藏且已隐藏”的文档只会在两个条件同时成立时显示：
  - `showHiddenItems = true`
  - `showFavoritesOnly = true` 或普通模式下未被其他过滤排除
- 组合模式下，隐藏文档继续保留现有弱化样式，收藏文档继续保留收藏星标

理由：

- 这与当前隐藏模式的真实存在相匹配，避免实施时临时补规则
- 用户已经明确要求本轮正式支持该组合模式
- 这样可以保持过滤顺序稳定，也更方便写测试

### 5. 隐藏规则优先级高于收藏规则

若某个文档已被隐藏，即使它在 `favoritePaths` 中，默认也不应因为“只看收藏”而重新出现。

建议优先级：

1. 原始树
2. 隐藏可见性模式
3. 收藏过滤
4. 搜索过滤

理由：

- 隐藏在当前产品中是更强的“从树里排除”语义
- 若收藏能绕过隐藏，用户会难以理解“隐藏”是否仍然生效
- 现有隐藏实现已经明确采用 profile 级排除逻辑，本轮不应推翻该语义

### 6. 收藏过滤必须保留目录上下文，不改成扁平列表

“只看收藏”开启后，左侧仍应显示树状结构，但只保留那些包含收藏文档的目录链路。

理由：

- 现有左侧搜索就是“保留目录上下文”的本地过滤，不应在收藏过滤上改成另一种浏览模型
- 用户仍需要知道收藏文档位于哪个目录
- 这样可以复用现有 `filterFileTree` 的设计思路，而不是重做文件树

## 交互规则

### 1. 文档行收藏按钮

- 收藏按钮放在文档行右侧操作区，位于隐藏按钮左侧
- 默认未收藏时显示空心星，已收藏时显示实心星或高亮星
- 点击按钮时不得触发文档打开
- 未收藏时可在 hover/focus 时显示收藏按钮
- 已收藏时即使未 hover 也要保留可见星标，保证用户能直接感知该文档已收藏
- hover/focus 时仍显示完整操作态，保证已收藏与未收藏都可点击切换
- 收藏按钮只出现在文档行，不出现在目录行

### 2. 搜索框左侧收藏过滤开关

- 收藏过滤开关放在左侧搜索框左边，属于文件树局部工具条的一部分
- 按钮语义是“只看收藏文档”
- `aria-pressed=false` 表示普通视图
- `aria-pressed=true` 表示只看收藏视图
- 开关切换后应立即刷新左侧文件树结果
- 当当前没有任何收藏文档时，开启后应显示专用空状态，而不是复用“没有匹配的文件”

### 3. 收藏与搜索叠加行为

- `showFavoritesOnly = false` 时，搜索在普通可见文件集合内过滤
- `showFavoritesOnly = true` 时，搜索只在收藏文档集合内过滤
- 无搜索词时，只看收藏模式下仍应显示所有收藏文档及目录上下文
- 搜索词清空后，应恢复到当前收藏过滤模式下的默认视图，而不是直接退出收藏过滤

### 4. 收藏与隐藏叠加行为

- 已隐藏但已收藏的文档，在 `showHiddenItems = false` 时不显示
- 已隐藏且已收藏的文档，在 `showHiddenItems = true` 且满足当前收藏过滤条件时显示
- 当 `showHiddenItems = true` 且 `showFavoritesOnly = true` 时，左侧树应只显示收藏文档，其中已隐藏项继续保留弱化样式
- 组合模式下的目录上下文仍应保留，只显示承载这些文档所需的目录链路

## 联动规则

### 1. 当前打开文档

- 当前已打开文档被取消收藏时，不强制关闭标签页
- 当前已打开文档在“只看收藏”模式下如果不再满足收藏条件，会从左侧树中消失，但中间正文保持不动
- 当前已打开文档被加入收藏后，应立即进入当前收藏集合
- 当前已打开文档在“只看收藏”模式下被取消收藏时，不自动退出收藏过滤

### 2. 标签页

- 收藏/取消收藏不影响标签页打开状态
- 收藏状态不应自动反映到顶部标签页结构上，本轮不做“收藏标签页”概念

### 3. 目录展开状态

- 收藏/取消收藏不应破坏已有目录展开状态
- 切换“只看收藏”时，已持久化的展开状态不应被覆盖
- 收藏过滤关闭后，应能自然恢复到原来的目录展开结构
- “显示隐藏项 + 只看收藏”组合切换时，同样不应清空已持久化的展开状态

### 4. profile 切换与刷新恢复

- `favoritePaths` 随 profile 切换而变化
- `showFavoritesOnly` 在切 profile 时重置为 `false`
- 刷新后恢复普通视图，但保留 `favoritePaths`

## 数据与实现方案

### 1. 状态层

建议在 `WorkspaceProfile.navigation` 中新增：

- `favoritePaths: string[]`

建议在 `App.tsx` 中新增会话态：

- `showFavoritesOnly: boolean`

对应影响文件：

- `src/workspace/profile-store.ts`
- `src/App.tsx`
- `src/workspace/local-bridge-access.ts`
- `tests/workspace/local-bridge-access.test.ts`
- `tests/server/project-profiles.test.ts`

### 2. 文件树过滤层

当前文件树已具备两类过滤：

- 基于 `hiddenPaths` 的可见性派生
- 基于搜索词的 `filterFileTree` 本地过滤

本轮建议在两者之间增加一层“收藏过滤”：

1. 先从原始树派生隐藏规则后的可见树
2. 若 `showFavoritesOnly = true`，再基于 `favoritePaths` 过滤文档
3. 最后再应用搜索词过滤

当 `showHiddenItems = true` 时，第 1 步的输出应包含隐藏节点及其隐藏元数据；后续收藏过滤和搜索过滤在这份“扩展可见树”上继续工作。

建议新增纯函数，避免把判断逻辑散落在组件内，例如：

```ts
filterFileTreeByFavorites(nodes, favoritePaths)
```

要求：

- 只保留收藏文档
- 自动保留其祖先目录
- 对空目录进行裁剪
- 不破坏原始树输入
- 输入类型应明确为保留隐藏元数据的渲染树节点，而不是重新退回普通 `FileTreeNode[]`

### 3. UI 透传链路

当前主要链路为：

- `App.tsx`
- `AppShell.tsx`
- `WorkspaceLayout.tsx`

建议新增透传内容：

- `favoritePaths`
- `showFavoritesOnly`
- `onToggleFavoritePath`
- `onToggleShowFavoritesOnly`

其中：

- `favoritePaths` 来自 profile
- `showFavoritesOnly` 来自会话态

### 4. 按钮布局

文件树局部工具条建议从单一搜索框改为：

- 左：收藏过滤开关按钮
- 右：搜索输入框

文档行右侧操作区建议从单按钮改为：

- 左：收藏按钮
- 右：隐藏按钮

这样能保持“快捷集合操作”在内侧，“排除操作”在外侧，视觉语义更清楚。

## 风险与注意事项

### 1. 目录收藏语义扩散

若本轮同时支持目录收藏，收藏系统会从“快捷文档集合”扩散成“树节点书签系统”。

应对策略：

- 本轮明确只支持文档收藏
- 若未来要做目录收藏，单独开方案处理

### 2. 收藏过滤与隐藏过滤冲突

若不明确优先级，用户会不清楚隐藏的收藏文档是否应该出现。

应对策略：

- 明确“隐藏可见性模式 -> 收藏过滤 -> 搜索过滤”的固定顺序
- 把“显示隐藏项 + 只看收藏”作为正式组合模式写进测试，而不是运行时临时兼容

### 3. 收藏过滤导致展开状态丢失

如果直接把收藏过滤后的树当作唯一目录来源，可能污染展开状态恢复链路。

应对策略：

- 继续保留原始树与可用目录集合
- 收藏过滤仅用于渲染输入，不直接覆盖展开状态真相

### 4. 零收藏时空状态误导

若直接复用当前“没有匹配的文件”或“当前项目还没有可用的 Markdown 文件”，用户会误判为搜索失败或项目无文件。

应对策略：

- 为“只看收藏且收藏数为 0”提供单独空状态
- 为“只看收藏 + 搜索词无命中”继续保留搜索无匹配文案

### 5. 过滤开关位置选错职责层

若把“只看收藏”做进顶栏，会把文件树局部过滤误提升成全局工作区模式。

应对策略：

- 将入口放在左侧搜索框左边
- 把它作为文件树面板局部工具，不放进 TopBar

## 建议执行顺序

1. 先扩展 `favoritePaths` 与 `showFavoritesOnly` 状态边界
2. 再实现文件树收藏过滤纯函数
3. 然后接入文档行收藏按钮
4. 最后接入搜索框左侧收藏过滤开关并补回归测试
